Governance, and Operational Hard Rules
for LLM Coding Agents

AnEntrypoint - gm (glootius maximus)

Abstract
LLM coding agents exhibit failure modes invariant under self-regulation: assumption propagation, state drift, memory decay, premature closure, fabricated output indistinguishable in form from witnessed output. Model as channel noise between request and delivered artifact; the discipline's objective functional is signal-to-noise ratio (SNR), the fraction of emitted tokens/actions constituting monotone progress toward the specification versus stochastic drift. gm (glootius maximus) is 14000+ supervised-modification hours gradient-ascending that objective; token austerity is the same objective under a cost prior, since unwitnessed narration is pure noise term. Every section below is a denoising operator over one noise class. Covered: the phase-gated FSM (PLAN->EXECUTE->EMIT->VERIFY->COMPLETE) with conjunctive forward guards and unconditional backward transitions on novel-unknown detection; the named-mutable/witnessed-execution type-state discipline; in-phase search and code-structure invariants; a governance layer of route families, orthogonal state planes, and quality metrics; second-generation rules (Maximal Cover, bounded autonomy, the recall/memorize learning loop, the execution budget model); operational hard rules (Fix-on-Sight, Browser Witness, Nothing Fake, web-search-before-pause, ORIENT, single-integration-test); and a third generation fixing the dispatch boundary itself: the spool as sole execution ABI (eliminating encapsulation friction from prose/API-call surfaces), the mutables file as an externally-enforced UNKNOWN|WITNESSED lattice, discipline isolation via @-sigil namespacing, synchronous-under-wasm exec semantics, throw-not-mutate gate denials, CI-as-ground-truth-build, the residual-scan stop gate, and semantic dispatch fan-out (agent-as-LLM) replacing subagent indirection. Together: a closed behavioral envelope shipped as one wasm-backed skill on npm.

1. Introduction

LLM coding agents are capable but unevenly reliable. Operating a single-skill harness (gm-skill, npm-published, wasm-backed) across every Claude-style skill host surfaces three failure categories, each a distinct noise injection on the intent-to-artifact channel:

  1. Assumption propagation: unverified hypotheses treated as ground truth, cascading downstream on falsification. Type confusion between UNKNOWN and WITNESSED: identical surface form, no static discriminator.
  2. State amnesia: context-window compaction drops resolved subproblems from the live set; re-derivation is wasted channel capacity. A lossy compression with no error-correction code.
  3. Premature closure: non-monotonic termination, declaring a fixed point reached before the invariant holds. The highest-cost noise class: it self-reports as signal.

Self-regulation cannot fix any of these: the same inference process that would detect the shortcut is the process producing it, so introspective correction is not a fixed point of the failure. Only external enforcement below the decision layer, spool-dispatch gates, marker-file barriers on disk, orchestrator-enforced FSM transitions, hard rules served fresh per call by the wasm instruction verb, is sound.8,9 Empirical work converges on the same diagnosis: premature termination, assumption propagation, and over-helpful substitution dominate under unconstrained self-regulation; constrained pipelines Pareto-dominate free-form agents on real software-engineering benchmarks.7,3

Production reference for the gm discipline. Every section is a denoising operator targeting one of the three classes above, or a signal-preservation mechanism. Lifecycle: SS2-8. In-phase quality: 9-12. Governance: 13-16. Autonomy/scope: 17-20. Operational hard rules: 21-29. Dispatch-boundary invariants: 30-37.

2. The State Machine Model

Suppresses the two loudest noise sources: action under UNKNOWN, and termination before the fixed point. Forward guards are conjunctive predicates over resolved facts only; backward transitions fire unconditionally on any novel-unknown detection, so no workaround is load-bearing across a phase boundary. Phase membership is a function of last-transition, never of narrated intent, an FSM has no side channel.

stateDiagram-v2
  direction TB
  [*] --> PLAN
  PLAN --> EXECUTE: zero new unknowns last pass
  EXECUTE --> EMIT: every mutable KNOWN
  EMIT --> VERIFY: post-emit matches pre-emit
  VERIFY --> COMPLETE: .prd empty AND git clean AND pushed
  COMPLETE --> [*]: docs refreshed
  EXECUTE --> PLAN: new unknown
  EMIT --> PLAN: new unknown
  VERIFY --> PLAN: new unknown
  EMIT --> EXECUTE: logic wrong
  VERIFY --> EMIT: broken output
  VERIFY --> EXECUTE: logic wrong
from the gm field guide"Forward transitions are conjunctive ladders. Backward transitions are snakes. The board is rigged in favour of finishing, but only if you climb every rung."

2.1 Phases

Five phases, forward-only absent a guard failure. PLAN: fixed-point iteration over the unknown-set, exits at a pass yielding the empty delta. EXECUTE: resolves the mutable set via witnessed execution, exits at total lattice convergence (all mutables WITNESSED). EMIT: writes artifacts, exits on post-emit/pre-emit exact equivalence. VERIFY: exits on the conjunction [.prd empty, worktree clean, pushed to origin]; the triggered cascade build runs async post-session. COMPLETE: docs refresh, session closes.

The agent dispatches instruction every turn; the wasm orchestrator (rs-plugkit/src/orchestrator/) returns each phase's body generated per-call from live FSM state, PRD, mutables, and recall pack, an idempotent pure function of current state, so prose cannot drift from ground truth because ground truth is its only input. No per-phase SKILL.md files remain; the shipped skill is one terse body naming the agent as the state machine itself, instructing dispatch of instruction every turn and execution of the verb the response names. Plugkit never self-advances; transition is agent-dispatched only.

2.2 Forward Transitions (Ladders)

Exit conditions are conjunctive predicates, every conjunct must hold simultaneously, ruling out the partial-advancement state where code exists ahead of its resolving witness.

2.3 Backward Transitions (Snakes)

Any novel unknown at any phase forces an unconditional backward transition (no accumulator, no override):

  • EXECUTE -> PLAN: new unknown, or mutable non-convergent after 2 passes
  • EMIT -> EXECUTE: logic defect in a WITNESSED mutable; EMIT -> PLAN: genuinely new unknown
  • VERIFY -> EMIT: output mismatch; VERIFY -> EXECUTE: logic defect; VERIFY -> PLAN: requirement defect

Root-cause regression is mandatory over local patching: unexpected EMIT-time behavior forces a fix at the source of the invariant violation, never an accreted workaround.

Observation (Termination). Finite discoverable-unknown set + monotone decrease per planning pass implies termination. Observed convergence: 2-4 passes.

3. The Mutable Discipline

An unverified assumption is a type-confusion attack on the discipline: UNKNOWN and WITNESSED are structurally indistinguishable at the point of use until a downstream fault discriminates them. The mutable discipline is type-state enforcement over a two-element lattice, {UNKNOWN, WITNESSED}, with UNKNOWN bottom and WITNESSED reachable only through one transition rule: assignment by a real side-effect's output. No coercion, implicit or explicit, promotes UNKNOWN to WITNESSED absent that witness.

3.1 Named Unknowns

A named mutable reifies every free variable the decision procedure or the code depends on, named before use: identifier, expected value, current value (init UNKNOWN), resolution method. Progress past an open (UNKNOWN-state) mutable is statically disallowed, not merely discouraged.

Reification makes latent uncertainty an addressable term. Plausible-but-unwitnessed code is the dominant generator of assumption-propagation failure.

3.2 Resolution by Witnessed Execution

The sole WITNESSED-transition rule: value assignment from the observed output of real execution against real services and real data. Inadmissible as witness, each a form of unsound inference:

  • Documentation (non-authoritative, may be stale relative to HEAD)
  • A priori reasoning about behavior (unsound: model of the system, not the system)
  • Static source inspection (decidable properties only; misses runtime-only failure modes)

Witnessed output is the only admissible ground truth; absent it, state stays bottom regardless of prior plausibility. Converges with the code-hallucination literature: reasoning-only verification systematically misses runtime-only failure modes; iterative grounding against live APIs is the empirically validated fabrication-reduction mechanism.10,11

3.3 The Two-Pass Rule

Non-convergence after two resolution passes reclassifies the mutable as a fresh unknown and forces the PLAN snake, a termination guard against a fixed-cost infinite loop over one failing strategy.

3.4 Surprise Absorption Prohibition

No silent absorption of expected-vs-actual divergence. Every discrepancy instantiates a new named mutable and forces PLAN regression, precluding workaround-debt accumulation.

4. Work Item Management: The PRD

Scope drift, shipping a convenient slice and framing the remainder as follow-up, is unrecoverable signal loss. The PRD is durable on-disk state making the full request set a checkable invariant; a pending item is message-in-flight, not a dropped one.

4.1 Structure

.gm/prd.yml: id, imperative subject, status (pending -> in_progress -> completed), acceptance criteria, edge cases, bidirectional dependency edges, effort estimate, route-family tag, load score lambda in [0,1]. Mandatory beyond single-line single-file edits.

verification_budget = lambda x (1 - tier_confidence); lambda > 0.75 requires pre-EMIT witness.

Under independent study: a SANER 2026 registered report on spec-preceding-code error reduction, and Agentless's constrained localize/repair/validate decomposition beating free-form agentic loops on SWE-bench at lower cost.16,7

4.2 Wave-Based Parallel Execution

Dependency DAG drives topological wave scheduling: each wave batches every dependency-satisfied pending item as concurrent spool dispatches in one message, fan-out width bounded by spool concurrency cap not agent count. Platform Task/Explore subagents forbidden, isolated-subagent prose return strands the chain (section 38); parallelism is exclusively N concurrent orchestrator-serviced spool dispatches.

4.3 Stop-Checkpoint Enforcement

Marker-file gate at the stop checkpoint: non-empty .gm/prd.yml, dirty worktree, or unpushed HEAD each independently deny stop. Gate reads disk state, no hook-event dependency.

4.4 Cross-Session Continuity

PRD persists across sessions as the durable contract; discovered prior-session items are current-session obligations regardless of current-turn mention. Sessions are scheduling quanta; the PRD is invariant across them.

5. Cross-Session Memory Architecture

A context window is a lossy, unbuffered channel: FIFO eviction under fill pressure drops earlier signal, and re-derivation of an evicted subproblem retransmits an answer already sent. Persistent memory is the error-correcting code across the session boundary; the exclusion and verification rules below are the parity check preventing stale entries from re-entering as false signal.

5.1 Memory Types

Four semantic types partition the vector store:

user:
profile, role, expertise, preference
feedback:
behavioral correction/confirmation, the persistent learning channel
project:
work context, deadlines, decisions
reference:
external-system pointers

Memory is project-resident, never platform-resident scratch state, committed to the repository so it survives across the machine/session/CI boundary. No raw-text search index exists: memorize-fire embeds each entry through an in-wasm bge-small model and persists content plus its 384-dim embedding as two co-located files, .gm/disciplines/<ns>/<key>.json and .gm/disciplines/<ns>-vec/<key>.json; retrieval is embeddings-only, cosine similarity against every candidate composed with a recency decay (30-day half-life, floored), then Jaccard dedup, no inverted text index anywhere in the path. recall/auto_recall re-score on every dispatch rather than loading a session-start snapshot. Parallels recent agentic-memory architectures: A-MEM's Zettelkasten-style embedding-linked notes, Mem0's consolidation-plus-salience production store, and MemoryAgentBench's finding that even strong RAG-augmented agents trail humans on retrieval accuracy, test-time learning, and selective forgetting, the empirical case for per-turn auto-recall over session-scoped retrieval.13,14,15

5.2 Exclusion Principle

Anything derivable from live project state (code patterns, architecture, paths, git history, prior debugging) is excluded from the store; storing it creates a second, driftable source of truth. Acting on a stale cached fact is among the highest-cost failure classes.

5.3 Verification Before Recall

A memory naming a specific file/function/flag is re-verified for continued existence before being acted on: a memory is a claim indexed by write-time, not a claim about present state.

5.4 Feedback Loop

The feedback type is a persistent behavioral-correction channel: corrections and confirmed non-obvious approaches are recorded as {rule, reason, context}, closing the repeated-cross-session-mistake ("Groundhog Day") failure mode.

6. Implementation: The gm Orchestrator

Enforcement resident in the agent's own context is a filter the same process can edit, the noise source gains write access to its own denoising operator. The orchestrator relocates every gate below the decision layer, onto disk and into the wasm verb surface: enforcement as external ABI, not internal discipline the model must choose to honor.

6.1 Single-Skill Architecture

One published skill, gm-skill: a single terse hard-rule-elicitation body, zero per-phase content, instructing dispatch of instruction every turn; the orchestrator generates phase-imperative content per call. Superseded: the earlier fifteen-skill chain (gm, planning, gm-execute, gm-emit, gm-complete, update-docs, governance, code-search, browser, ssh, pages, create-lang-plugin, memorize, textprocessing), now folded entirely into the instruction response and the verb-dispatch surface. This is the encapsulation-friction fix at the skill layer: one thin invariant elicitor plus a data-driven response body, not fifteen coupled prose surfaces to keep synchronized.

6.2 Spool Dispatch and Marker Gates

State-machine enforcement is marker-file driven, not event-driven: the earlier host-process hook callback pump (session-start, prompt-submit, pre-tool-use, stop) is retired in favor of the gate running natively inside plugkit.wasm (rs-plugkit's gates.rs plus its hook_pre_tool_use/hook_stop exports), reading disk state before any tool executes. The wasm host calls the gate; denial surfaces its reason text to the model. Marker files under .gm/ constitute the discipline's persistent automaton state:

.gm/prd.yml
Existence triggers the needs-gm gate: until the gm skill has been invoked this turn, write/edit/git operations are blocked.
.gm/mutables.yml
Any row with status: unknown blocks Write, Edit, NotebookEdit, git commit, git push, and the stop checkpoint.
.gm/needs-gm
Written by bootstrap; signals the dispatcher that the agent must invoke the gm skill this turn before tool use is permitted.
.gm/gm-fired-<sessionId>
Written when the agent invokes Skill(skill="gm"), clears the needs-gm gate. Cleared again at turn start.
.gm/residual-check-fired
One-shot marker that ensures the residual-scan challenge fires exactly once per stop window; clears at the next prompt-submit.

Session-start bootstrap (lib/skill-bootstrap.js) provisions the wasm artifact, instantiates it, writes .gm/exec-spool/.bootstrap-status.json, lays the markers. auto-recall fires at every turn entry: derives a 2-6 word query from the prompt, injects "## Recall for this prompt" into the next instruction response.

Inference cannot route around enforcement: markers are on-disk facts, the dispatcher reads them independent of context state, the gate sits below the tool surface, structurally unreachable from the decision layer. Strictly more robust than the retired hook-event pump: no event to miss, no callback to forget, no host-specific lifecycle, only files and one read.

6.3 Unified Code Execution

All execution routes through exec_js: agent writes .gm/exec-spool/in/<verb>/<N>.txt, the wasm watcher reads it, host runs child_process.spawnSync, agent reads back one JSON response at out/<verb>-<N>.json or out/<N>.json carrying {ok, stdout, stderr, exit_code, timed_out, duration_ms}. Language verbs: nodejs, python, bash, typescript, deno, go, rust, c, cpp, java, plus browser.

6.4 Single-Package Deployment

Ships as one npm package, gm-skill, loadable by any Claude-style skill host; gm-plugkit wraps the wasm artifact and bootstrap shim as its dependency. A cascade pipeline over three wasm-only Rust repos (rs-plugkit, which folds in the retired rs-exec host-helper and rs-learn memory surfaces natively, plus rs-search and rs-codeinsight) plus plugkit-bin Releases auto-publishes on any push, zero manual steps. Supersedes a fifteen-repo per-platform fan-out (gm-cc, gm-gc, gm-oc, gm-kilo, gm-codex, gm-qwen, gm-copilot-cli, gm-hermes, gm-thebird, gm-vscode, gm-cursor, gm-zed, gm-jetbrains, gm-antigravity, gm-windsurf), archived. One install replaces fifteen.

7. Constraint Tiers

Tiers rank constraint violation by admitted-noise magnitude: Tier 0 breach is channel corruption, Tier 3 breach is a density loss. Four tiers, decreasing severity:

Tier 0 (Absolute):
no_crash, no_exit, ground_truth_only, real_execution, enforced by the spool-dispatch gate, cannot be overridden
Tier 1 (Quality):
max_file_lines=200, hot_reloadable, checkpoint_state
Tier 2 (Modularity):
no_duplication, no_hardcoded_values
Tier 3 (Style):
no_comments, convention_over_code

8. Pre-Emit Verification Protocol

An in-memory pass that fails from disk is fabrication with a green check attached, the highest-cost noise class. Two-stage verification closes the gap between in-memory state and the persisted artifact. Pre-emit: import target module, run proposed logic in-memory against real and error inputs, record expected outputs, verify all gate conjuncts hold, then write.

Post-emit: re-import from disk, re-run identical inputs, compare exactly. Known variance -> fix-and-reverify; unknown variance -> PLAN snake. Catches what test-after-write misses: serialization bugs, import-resolution skew, filesystem race conditions.

9. In-Phase Quality: The Search and Code Discipline

Two paths to the same correct answer can carry wildly different signal density: blind grep re-derives what the project's index already encodes, a single grounded hit retrieves it. This discipline keeps search grounded and code structural, minimizing wasted-motion cost on the path, not just at the answer.

9.1 Codebase Search Protocol

Absence is a claim requiring a minimum search campaign before assertion:

  • 2-word initial query, one-word delta per pass (hill-climb over query space)
  • 4-attempt floor before absence is admissible
  • Exclusively codesearch; the spool-dispatch gate denies direct Glob/Grep/Find during agent phases, all reconnaissance routes through one indexed surface

codesearch is embeddings-only retrieval with no raw-text index backing it -- the corpus is a vector space, not a grep target, so recall is semantic-similarity-ranked rather than substring-matched. It fuses BM25 lexical scoring with vector cosine similarity via Reciprocal Rank Fusion (rs-search, section 42), and boosts by git-commit relevance: files touched in commits topically near the query rank higher, temporal-proximity-weighted retrieval on top of semantic similarity. Both codesearch and recall return dual full-result sets, complete matches, not truncated snippets, so a hit is immediately actionable without a second fetch round-trip; every output is token-optimized for the calling context, no unbounded dump. A found concern is consolidated into, never duplicated beside, its existing implementation, a second abstraction over one already indexed is competing-implementation debt at construction time.

9.2 Minimal-Code Resolution Order

When a problem can be solved multiple ways, the agent follows a four-step resolution order and stops at the first that applies:

  1. Native first: does the language or runtime already provide this?
  2. Library second: does an existing dependency solve this pattern?
  3. Structure third: can the problem be encoded as data (map, dispatch table, pipeline) so the structure enforces correctness?
  4. Write last: only author new logic when the above three are exhausted.

Dispatch tables over switch chains, pipelines over loop-with-accumulator, maps over if/else forests: correctness properties, not style, structure eliminates the invalid-state class that manual consistency-maintenance otherwise admits.

10. Structural Parallelism and Wave Execution

Serializing independent work is dead-air channel time. Wave discipline fills it: batch every dependency-satisfied pending item as one message of concurrent spool dispatches, collect, evaluate next wave. Runs to PRD-covering-family completion, never stops mid-wave to poll for direction.

Sole serialization constraint: browser (shared Chromium, no concurrent sessions). Everything else is wave-eligible.

11. Permanent Observability

Unmeasured noise is unreducible noise. Observability is the instrument making the channel legible: every subsystem queryable without restart, drift read off structured state, never inferred from rendered output.

11.1 Mandatory Enumeration

Every planning pass enumerates subsystems lacking a queryable inspection endpoint, one PRD item per gap. PLAN exits at zero-gap pass.

11.2 Server-Side

Every subsystem exposes /debug/<subsystem> emitting structured {subsystem, severity, ts} JSONL. console.log is not observability. Instrumentation wraps the dispatch boundary once, never per-handler.

11.3 Client-Side

window.__debug: live structured registry, mount-time register / unmount-time deregister of state, active requests, sockets. Addressable by key, no restart to query, no parallel observability surface, asserted via test.js.

11.4 Stack Observability

Structured JSONL to ~/.claude/gm-log/<YYYY-MM-DD>/<subsystem>.jsonl; watcher stdout/stderr (plus wasm cdylib println!/eprintln!) to .gm/exec-spool/.watcher.log, 10MB rotation. Spool-root diagnostics: .status.json (stale mtime = dead watcher), .last-session-start.json, .bootstrap-error.json (absent = healthy). health verb: one-shot full-component probe. Diagnosing without reading the log or dispatching health first is closure-by-guess.

12. The Memorize Sub-Agent

Signal amortized once should never be re-paid. Memorize is the write half of the loop, converting a resolved WITNESSED fact into a durable store entry rather than letting it decay back to UNKNOWN at session boundary. Every UNKNOWN->WITNESSED transition fires a background dispatch same-turn:

write .gm/exec-spool/in/memorize-fire/<N>.txt
  body: "<fact>"
read  .gm/exec-spool/out/memorize-fire-<N>.json

N resolved facts in one turn spawn N parallel dispatches in one message, never batched (co-batched facts collapse to one classification, losing per-fact semantic separation). End-of-turn self-check scans for un-memorized resolutions. Content-hash keying makes the write idempotent (f(f(x))=f(x)): a re-fired identical fact dedups against its existing key rather than accreting a duplicate, convergent under at-least-once dispatch semantics. memorize-fire is the sole writer of AGENTS.md/CLAUDE.md; agent inline edits to those files are prohibited, they are the staging ground, the store is the retrieval surface, and every load-bearing rule is dual-written: land inline in AGENTS.md for the compaction-surviving copy, fire the same text to the default-namespace store so future-turn recall/auto_recall surfaces it. The flow is bidirectional and back-pressured: each memorize session also drains a few existing detail-heavy AGENTS.md entries to the store, firing the substance and compressing the file to a one-line pointer, so the staging ground deflates as fast as the store accretes rather than growing unbounded. The classifier rejects changelog-shaped (dated, past-tense) facts from AGENTS.md ingestion; rs-learn's default namespace still accepts them directly.

Recall and memorize are the two dispatches of one closed learning loop; either alone is an open loop with unbounded rediscovery cost.

13. Governance Layer

Governance operationalizes the objective: route families, state planes, refused collapses turn SNR from an aspiration into decidable predicates a transition passes or fails. What the lifecycle suppresses by construction, governance verifies by instrument.

13.1 Seven Route Families

Every PRD item tags one of seven route families gating which quality constraints apply:

grounding:
Constrains the agent's belief formation, what counts as evidence, what counts as knowledge, when to regress to planning. Governs all information-gathering operations.
reasoning:
Constrains inference chains. Requires each reasoning step to be witnessable, prohibits conclusions that outrun their premises, enforces the two-pass rule and surprise absorption prohibition.
state:
Governs mutable state transitions. Every mutation must be visible in the PRD; state-bearing operations require pre- and post-conditions. The PRD is the authoritative state record.
execution:
Constrains code execution: real services only, witnessed output, exec_js dispatch, timeout budgets. No stubs, no mocks, no hardcoded responses.
observability:
Every subsystem must be inspectable without restart. Covers /debug endpoints, window.__debug, structured logging, the gm-log JSONL stack.
boundary:
Governs the agent's interaction surface with external systems: git, CI, remote APIs, the user. The remote-execution boundary (section 16) is the primary concern of this family.
representation:
Governs how information is encoded and communicated: skill prose must be implicit not explicit, memory must use the four semantic types, PRD items must use the mandated schema.

13.2 Four State Planes

Agent state is tracked across four orthogonal planes that must be consistent at every phase transition. Each plane has a concrete detection test:

epistemic:
What the agent knows, what it is uncertain about, what mutables remain unresolved. Detection test: count unresolved mutables (epsilon); PLAN exits only at epsilon = 0. Any named UNKNOWN that has not been witnessed is an open mutable.
operational:
What the agent has done, what artifacts exist on disk, what code has been executed. Detection test: git status (uncommitted changes = inconsistent), acceptance criteria assertions (failing = inconsistent), exec_js return value witnessed (not just theorized).
relational:
What the agent has communicated to the user, what commitments are outstanding. Detection test: any open "I will" or "next I'll" statement in the agent's prior response that has no corresponding completed action in the operational plane is a relational inconsistency.
temporal:
What happened in earlier sessions, what the PRD preserves across context resets. Detection test: .gm/prd.yml exists and has pending items = temporal inconsistency with COMPLETE; recall hit on a prior design decision that differs from the current plan = weak temporal inconsistency requiring re-witness.

A phase transition is legitimate only when all four planes are consistent with the transition's exit condition. An agent that advances EXECUTE->EMIT while the epistemic plane still contains unresolved mutables is in an inconsistent state regardless of what the operational plane shows.

13.3 Five Refused Collapses

Five patterns are categorically refused because they collapse the discipline without appearing to:

  1. Narrating as executing: describing what code would do rather than running it
  2. Assumption as resolution: treating a plausible value as witnessed
  3. Partial closure: declaring a PRD item done while its acceptance criteria are unmet
  4. Deferred observability: shipping code without an inspection endpoint and noting it as a future task
  5. Asking-as-progress: routing through the user a question the agent could answer itself via tools

14. Quality Metrics

Four metrics govern quality gate decisions at phase transitions:

DeltaS (state delta):
The number of state-plane inconsistencies introduced by a proposed change. A change with DeltaS > 0 must be rejected or the inconsistency must be resolved before advancing.
lambda (load score):
Consequence weight of a PRD item (0.0-1.0). Items with lambda > 0.75 require witnessed verification before EMIT. Verification budget = lambda x (1 - tier_confidence), where tier_confidence is the fraction of acceptance criteria for this item that have already been witnessed (0.0 = none witnessed, 1.0 = all witnessed). Higher prior witnessing shrinks the remaining verification budget proportionally.
epsilon (epistemic gap):
Count of unresolved mutables in the epistemic plane. PLAN exits only at epsilon = 0 on the final pass.
Coverage (>=0.70):
The proportion of load-bearing behaviors witnessed by test.js. The test file enforces a 200-line budget; coverage >=0.70 at that budget is the acceptance criterion. Behaviors not witnessable within the budget are not load-bearing by definition.

15. Failure Taxonomy

The noise catalogue: sixteen failure modes across four categories, each a specific error-injection or signal-drop mechanism, each mapped to a suppressing rule elsewhere in the paper. Every PLAN pass cross-references this taxonomy against each PRD item:

Grounding Failures

  1. Assumption propagation: unverified hypothesis treated as ground truth
  2. Documentation trust: stale docs used as resolution evidence
  3. Static analysis substitution: code reading used instead of execution
  4. Memory staleness: outdated memory acted on without verification

Execution Failures

  1. Stub shipping: placeholder code presented as real implementation
  2. Mock leakage: test fixtures surviving into production paths
  3. Timeout blindness: execution result truncated without detection
  4. Orphan accumulation: background processes not cleaned up on restart

State Failures

  1. Partial closure: PRD item marked complete with unmet acceptance criteria
  2. Epistemic advance: phase transition attempted with unresolved mutables
  3. Cross-session amnesia: PRD items from prior sessions not discovered and executed
  4. Dirty exit: session closed with uncommitted or unpushed changes

Boundary Failures

  1. CI blindness: COMPLETE declared before CI result is known
  2. Remote-execution misread: git push treated as local execution, CI result not watched
  3. Forced pause: pause issued for a question the web could have answered
  4. Asking-as-progress: user asked a question answerable via tools

16. Remote Execution Boundary

16.1 git push as Remote Execution

A git push is not local; it is a remote-execution event dispatching a build/test/deploy pipeline on external infrastructure. The CI result is the canonical verdict on correctness, and declaring COMPLETE before it is premature closure.

16.2 CI Watch Protocol

After every push, the agent dispatches gh run watch through the spool, polls to completion, and inspects the result. Green CI is a precondition for VERIFY->COMPLETE; the stop-checkpoint gate refuses session close while the most recent push has no recorded green run. The agent cannot advance past VERIFY while CI is pending or red.

16.3 Parse-Time CI Failure Shapes

Five CI failure shapes are catalogued for rapid triage:

  1. Import error: module not found, check package.json or Cargo.toml, not the source file
  2. Type error: schema mismatch, regression to PLAN to re-witness the interface
  3. Test failure: assertion mismatch, root cause first, never silence the assertion
  4. Lint failure: style constraint violation, fix in-band, never // eslint-disable
  5. Build timeout: infrastructure issue, re-trigger once; if repeated, open a PRD item with blockedBy: external

17. Maximal Cover

Hard Rule: Maximal Cover [PLAN]

When scope exceeds reach, construct a maximal cover, not a refusal, not a single slice with the rest deferred. Distributed refusal is the same failure dressed as triage. The default shape is the widest, deepest, most destructive reading the request admits from this session; the narrower shape is the exception and carries the burden of proof.

Required move: construct a covering family of every bounded subset of the request witnessable from this session. Write the family into the PRD with the dependency graph explicit. Execute every member. Build cost, CI duration, cross-repo reach, and breaking-change weight are inputs to executing the cover, not arguments for shrinking it.

Residuals the agent judges within the spirit of the original ask AND completable from this session are self-authorized. Expand the PRD with them and execute, do not stop to ask. Only residuals genuinely outside the original ask, or genuinely unreachable from this session, are name-and-stop.

When expanding under self-authorization, the agent declares its judgment in-response ("treating X as in-scope because Y") so the user can correct mid-chain. Silent expansion without the declaration is the failure mode this rule guards against.

Enforcement: committed work + named out-of-spirit residuals must equal the witnessable closure of the request. Gap = cover not maximal -> re-enter planning.

A dropped requirement is the quietest noise class: no error, no red test, a silent divergence between delivered and specified. Maximal Cover hunts it. Scope-exceeds-reach is a planning-input, not a stop condition: the covering family (over witnessable subsets of the request, from this session) is itself the plan, executed to completion. Single-slice-and-stop is refusal wearing triage's clothes, both leave witnessable work unexecuted, both are non-monotonic termination. Controlled experiments quantify the structural bias the rule corrects: systematic section omission, length undershoot, minimum-viable substitution for full spec, laziness as a measured regularity, not an anecdote.17

18. Bounded Autonomy

Hard Rule: User Request = Authorization [PLAN, EXECUTE]

The user's first message asking for X is the approval to do X. The PRD records the authorization; it is not a second permission gate. Re-asking "want me to do X?", "should I proceed?", "which shape of X?" after the user said "do X" is forced closure dressed as deference, the user already authorized the work and waits while the agent stalls.

Surfacing tradeoffs is allowed inside the same chain (one sentence: "going with shape A because Y") so the user can interrupt, never as a stop-and-wait. Multi-repo scope, build cost, CI duration, and binary-size impact are never grounds to re-confirm.

A written PRD is the user's authorization. Once it exists, EXECUTE owns the work to COMPLETE. Every doubt arising during execution is resolved by witnessed probe, by recall, or by re-reading the PRD, never by asking the user.

Asking is last-resort: destructive-irreversible AND not covered by the PRD, OR user intent irrecoverable from PRD/memory/code/web. Channel: pause (renames prd.yml -> prd.paused.yml; question in header). In-conversation asking is last-resort beneath last-resort. The disposition: a question forming in-response is a signal to expand scope and dispatch a WebSearch/WebFetch pack first; the question survives only if both close empty. Most agent-side questions are scope-narrowing dressed as deference, and the answer the user would give matches what one web call would produce.

pause interaction with the stop checkpoint: when the agent issues pause, the PRD is renamed to prd.paused.yml. The stop gate's PRD-completeness check reads .gm/prd.yml and finds no live PRD, so the session may close normally. On the next session-start bootstrap, skill-bootstrap.js detects prd.paused.yml without a live prd.yml and renames it back, resuming where the agent paused. There is no hook-timeout race because the rename is a synchronous file operation completed before stop is attempted.

A self-answerable question routed to the user is noise transferred onto the higher-latency channel participant, and the round trip it costs is signal the session never spends on the work. Every ask costs >=1 round trip; a tool-answerable question is agent work externalized as user labor. The accumulated asking tax (compounding per-question context-reconstruction cost) is a quality defect on par with a broken test.

19. The Learning Loop

The learning loop is the sole mechanism raising SNR monotonically over calendar time rather than holding it constant per-session: every recalled-not-rederived fact is a discovery cost paid exactly once, amortized across all future sessions. Requires both operations firing every cycle:

19.1 Recall (Read)

Precedes any fresh-execution resolution attempt:

recall
<2-6 word query>

Triggers: familiar-feeling unknown | subtask on a known project | about-to-ask something likely already discussed | about-to-design where a prior decision likely exists. Hits are weak_prior, non-authoritative until re-witnessed. Cost: ~200 tokens, ~5ms.

19.2 Memorize (Write)

Every UNKNOWN->WITNESSED transition fires a background memorize-fire same-turn; N facts -> N parallel writes in one message. Skipped memorize reopens the loop: every future session re-pays full discovery cost for a fact this session already resolved.

19.3 rs-learn Backend

Both operations target the rs-learn KV+vec store at .gm/rs-learn.db, git-committed so memory and index state are shared, not per-machine. auto-recall calls rs-learn in-wasm and injects "## Recall for this prompt" into the next instruction body ahead of the agent seeing the user turn.

20. Execution Budget Model

Token austerity made literal at the execution layer: a mandatory per-call ceiling bounds worst-case unproductive spend, and a timeout returns partial signal rather than hanging the channel. Two-layer budget on every exec_js dispatch:

  1. Dispatch layer: timeoutMs mandatory in the request body; missing/zero is a hard reject
  2. Host layer: child_process.spawnSync with timeout; on expiry, partial streamed output preserved, timedOut: true sidecar

The [exec timed out after Nms; partial output above] marker is authoritative truncation, not error; re-issue at a higher budget. Retry-at-same-budget is a fixed-cost infinite loop.

Synchronous-under-wasm means a long subprocess blocks the watcher for its duration, no detached background execution exists. Long-lived workloads pass a real timeout and accept watcher unavailability to other dispatchers for the call's span.

21. ORIENT: Pre-Planning Recall Pack

Hard Rule: ORIENT [PLAN]

Open every planning session with a parallel pack of 3-5 recall calls and 3-5 codesearch calls against the request's nouns, all in one message. Hits land as weak_prior; misses confirm the unknown is fresh.

The pack runs in one message, never serially. The agent that skips ORIENT pays the same cost in fresh probes a turn later, plus the price of disagreeing with its own prior witness if the earlier session established a relevant fact.

ORIENT is free relative to the duplicated discovery and disagree-with-prior-witness risk it prevents. Its cost equals the cost of skipping it, because the skipping agent will eventually run the same searches; it just does so later, after having already named unknowns that may be stale.

ORIENT front-loads signal: pulls existing project knowledge to planning's start so named unknowns are genuinely novel, not lookup failures. No absence claim without a prior search; no new abstraction without a prior existence check. The parallel pack runs this at maximum concurrency, zero serialization cost against planning.

22. Fix on Sight

Jank accumulates as a noise floor: each unaddressed rough edge is a small persistent corruption, and their sum reads as machine-shaped regardless of per-line correctness. Fix on Sight refuses to let detected noise outlive the turn that detected it.

Hard Rule: Fix on Sight [EXECUTE, EMIT, VERIFY]

Every issue surfaced during work is fixed in-band, this turn, at root cause. Defer-markers, swallowed errors, suppressed output, skipped tests, and "address it next session" are variants of the same failure: a known-bad signal carried past the moment of detection.

Surface -> diagnose -> fix at root cause -> re-witness -> continue. New unknown surfaced by the fix -> regress to planning. Out-of-scope-irreversible fix -> residual into PRD before moving on. Narration is not a substitute for a PRD item.

22.1 Auto-PRD on Incidental Error

Any reasonably fixable mid-task error, pre-existing build break, lockfile drift, broken dep, neighboring lint failure, adjacent type error, stale generated file, dead import, is added to .gm/prd.yml same-turn and executed before COMPLETE, unsolicited. Closing on a still-emitting known-bad signal is non-monotonic termination regardless of provenance.

Genuinely out-of-reach errors (require user credentials, depend on external services that are down, demand product decisions) are named and stopped with the residual recorded in the PRD with blockedBy: external, never silently dropped, never narrated past.

22.2 Auto-PRD on Re-Architecting Opportunities

The same rule applies to obvious refactor wins surfaced while doing the user's task: a hand-rolled implementation that a well-known library does cleaner; a multi-file ad-hoc system one import would replace; duplicated logic crying out for one helper; a boilerplate pattern repeated in N places where one upstream tool covers it. These are not separate sessions, they are PRD items, added the same turn they surface, executed before COMPLETE.

The bar is obvious + reachable from this session, not any conceivable refactor. Speculative or aesthetic changes stay out. Library-replacement is a specific case: before keeping a local reimplementation of something a published package already provides, add a PRD item to swap to the import. Local code competes with upstream maintenance and loses.

The architecture is treated as pliable: not a fixed substrate the work routes around, but a shape the work is licensed to reshape. Every architectural change that clearly improves the design or clearly reduces the code-maintenance burden is a spooled PRD plan, not a deferred wish. Replacing bespoke code with native functionality or a very-popular, well-maintained library is encouraged, but only when it reduces the codebase to a net-smaller shipped-and-maintained surface. Adding a heavy dependency to delete a few lines net-grows maintenance and is the failure mode the rule guards against; the swap is admissible precisely when the line count and the maintenance surface both fall.

22.3 The Jank Sweep

At PLAN, for every surface the user's prompt concerns, the agent enumerates to the PRD a complete list of every possible aspect that can be checked for jank, every immaturity, every unfinished edge, every half-wired path, across gui, ux, ui, client state, server state, the client/server state boundary, and any other surface the request reaches. Jank is the operative word: the sweep hunts the rough, the unpolished, the almost-done, not only outright bugs. Each aspect that must be improved or validated is its own PRD row, including a profiling row and a security row for every surface that can carry them. The sweep is scoped to what the prompt concerns and its reachable closure, not an unbounded repo-wide audit, but within that closure it is exhaustive. Every issue found along the way opens its own debug-and-repair plan spooled as PRD rows the same turn, and every outstanding quick improvement is spooled too.

22.4 The Tell-Tale Sweep

One tell-tale AI design element found along the way, a boilerplate flourish, an over-hedged comment, a generic scaffold name, an unmistakable machine-authored shape, spawns a full-codebase sweep. A single sighting is never a one-off local fix: it is the witness that the same shape is likely elsewhere, so it triggers a complete codebase-wide resolution run spooled to the PRD as its own rows, one for the scan, one per cluster of findings, one for the fix-and-verify, exhaustive over every possible file. A tell-tale left standing anywhere is the tell that the whole was machine-shaped, so the sweep does not stop at the instance that surfaced it.

23. Browser Witness

Hard Rule: Browser Witness [EXECUTE, EMIT, VERIFY]

Editing code that runs in a browser requires a live browser witness in the same turn as the edit. The witness does not defer to a later phase; later phases re-witness on top, they do not replace this one.

Protocol: (1) boot the real surface, server up, page reachable, HTTP 200 witnessed; (2) browser -> navigate -> poll for the global the change affects; (3) page.evaluate asserting the specific invariant the change establishes, capturing the witnessed values; (4) variance -> fix at root cause, re-witness (Fix on Sight). Never advance on unwitnessed client behavior.

Pure-prose edits to static documents with no JS/canvas/DOM behavior change are exempt; the exemption is tagged explicitly with the reason so the skip is auditable. Silent skip on actual behavior change is forced closure.

This rule fires in EXECUTE (witness on edit), EMIT (post-emit verify), and VERIFY (final gate).

A mock-DOM pass is a maximum-confidence certificate over an unverified proposition, the highest-noise outcome available: it terminates green and asserts nothing about the live runtime. Browser Witness substitutes the live page's own evaluation for the mock, closing the gap between "assertion passed" and "feature works." Only a live-DOM evaluation is admissible evidence that a client-side change renders correctly; a mock is a model of the page, not the page.

23.1 The Live Page Is the Debugger

Client debugging exposes state as window.* globals and interrogates them in situ via a page.evaluate loop against the real runtime, never blind-restart-and-eyeball. Auto-captured console, error, network, and performance streams accompany every session, so the evaluate loop reads a superset of what the page itself would print. Restart-and-observe is near-zero-information per cycle at one-turn cost; global-expose plus page.evaluate is a single dispatch reading exact live state, no restart, no reconstruction. Client behavior ambiguity resolves by surfacing the relevant state as a global, evaluating it live, asserting the invariant against the returned value, never by re-guessing across a restart. The live page is the debugger: the same evaluate surface that witnesses an edit (section 23) diagnoses one, and the edit-to-COMPLETE transition is hard-gated on this witness having fired (client-edit-no-witness, section 43) -- no unwitnessed client behavior reaches a terminal phase.

24. Nothing Fake

Hard Rule: Nothing Fake [EXECUTE, EMIT, VERIFY]

What ships runs against real services, real data, real binaries. Stubs, mocks, placeholder returns, fixture-only paths, "TODO: implement", hardcoded sample responses, and demo-mode fallbacks are forbidden in source the user will run. They produce green checks that survive into production and lie about what works.

Scaffolding and shims are permitted when they call through to real behavior: an empty file laid down before its body, a thin adapter wrapping an upstream API, a build target that compiles but is wired to nothing yet and is the only callsite of itself. The test is whether the artifact, executed, would do the thing it claims. If it would not, it is a stub.

Before writing a shim or adapter, the agent asks whether an existing library or tool already provides the same surface. Maintaining a local reimplementation of something an upstream package solves is its own failure mode, the shim drifts, ages, and accumulates the bugs the upstream already fixed.

Stub detection is behavioral, not by keyword: code paths that always succeed, always return the same value regardless of input, or short-circuit a real call to satisfy a type signature are stubs. Comments asserting realness do not make code real.

A shipped stub is pure noise at maximum confidence: clean return, passing check, zero relation to the specified behavior. Nothing Fake makes this class inadmissible in source. It is Section 3's execution-layer dual: witnessed execution requires real code to produce the witness; Nothing Fake forbids code engineered to pass witnessing through deception. Jointly: no fabricated witness, and no fabricated code past a real one. Coding-policy corollaries follow the same real-execution-only logic: no comment layer (a second, driftable description of behavior the structure already encodes), no synthetic test files (*.test.*/fixtures/mocking libraries are themselves a stub-shaped verification surface), no ASCII-non-conformant glyphs (decorative unicode is unverifiable-by-execution decoration), and verification is exclusively real-service execution witnessed live, never a fixture standing in for the system.

25. Web-Search Before Pause

Hard Rule: Web-Search Before Pause [PLAN, EXECUTE, EMIT, VERIFY]

Before pause or any in-conversation question whose answer plausibly exists on the public web, missing artifact, prebuilt binary, library status, build recipe, version compatibility, upstream issue, "does X exist for Y", fire WebSearch and at least one targeted WebFetch first. Pause or ask only after the web pass returns empty, or returns candidates the agent has witnessed and rejected.

Pausing on a question the web could have answered is forced closure dressed as humility. Fix on sight: re-enter planning, web-search, and resume.

Genuine user-only questions: private credentials, preference among already-surfaced viable options, destructive-irreversible authorization.

Targets the agent stalling on a publicly-documented unknown, taxing the user with a resolvable round trip. Instance of the general autonomy principle: any self-obtainable answer is the agent's own obligation, never the user's.

26. Single Integration Test Policy

Hard Rule: Single Integration Test [VERIFY, COMPLETE]

One test.js at project root. 200-line hard cap. No fixtures, mocks, or scattered test files under any naming convention. Plain assertions, real data, real system. gm-complete runs it. Failure = regression to EXECUTE.

Any second test runner, under any name, in any directory, is a smuggled parallel surface that fights the observability discipline. If a behavior needs exercising in-page, register it in window.__debug and assert via test.js.

26.1 Budget, Not Target

test.js is a 200-line budget, not a target. Every line witnesses a load-bearing behavior; redundant assertions are dead weight. Subsystems get one group each, combined groups (profiles+observability+auth+context) are the norm, not the exception. The win from combining comes from collapsing per-subsystem groups into multi-subsystem ones: the test surface grows linearly with subsystem count, not multiplicatively.

26.2 Use Overlap to Exclude

When subsystem A's test exercises B as a side effect, B does not need its own group. Examples from production: the agent-machine tool-loop test exercises bash dispatch; the dashboard test asserts the API surface and tool registration; the gateway test exercising one platform with a stub-shape loop covers all 18 adapters. Overlap is a signal to drop the redundant group, not to add another assertion.

26.3 Hygiene at Edit Time

Every change to test.js prefers compaction over expansion. If the file exceeds 200 lines, the discipline is to merge groups and drop redundancy until it fits, not to split the file or lift the cap. If the budget is genuinely insufficient, the question is whether the assertion is load-bearing, not whether the cap should move.

26.4 Group Naming Convention

Name combined groups by joining their subsystems with +: home+config+skin, mcp+swe+distributions+account+credpool, env+pi+cli+tui+setup+website. A group with 4-6 components is healthy; a group with 1 component should be questioned.

27. Sync-Before-Emit

rs-codeinsight and rs-search outputs require a freshly-completed index; cache serves only on digest match (mtime sum + git HEAD + dirty-tree marker) against live filesystem state. Default path runs fresh; --read-cache requires exact .codeinsight.digest match, auto-refreshing on mismatch.

rs-search runs scan+embed+sweep before first result, emitting [index fully synced: ...] for freshness visibility. Emitting from an unverified/partial index is closure-on-stale-ground-truth: acting on state that no longer exists.

28. Auto-Recall on Prompt-Submit

The auto-recall verb fires at every turn entry: it derives a 2-6 word query from the prompt, calls the rs-learn Searcher via the shared tokio Runtime, and injects "## Recall for this prompt" into the next instruction response before the agent acts. Auto-search (codeinsight) runs at session-start; auto-recall at every prompt. The mechanism is the spool, not a host-side prompt-submit callback, so any harness loading gm-skill gets it identically.

Every turn thus begins with prior memory loaded. An agent that skips ORIENT still receives the auto-recall injection; ORIENT adds 3-5 targeted queries on top.

29. Implicit Skill Prose

Skill files and system prompts elicit behavior, they do not describe it. Write terse imperative principles whose phrasing triggers the model's already-learned dispositions, not numbered procedures that spell out what to do.

Forbidden in skill prose: numbered step lists like "1. agent runs N parallel calls 2. then writes 3. then..."; citations to site pages or paper numbers; references to the documentation as an input ("as described in docs/skills.html"); multi-step recipes. The skill is a prompt, not a manual. If it reads like a manual, the behavior gets imitated as a script and breaks at the first edge case.

The papers and site are outputs of the discipline, not inputs to it. Cross-cutting rules that need a citation belong in AGENTS.md, not in skills. CLAUDE.md in the project root is a pure @AGENTS.md pointer, nothing else is written there directly.

30. The Spool Dispatch Surface

Hard Rule: Every Dispatch Through The Spool [PLAN, EXECUTE, EMIT, VERIFY]

Code execution and every utility verb both write a file to .gm/exec-spool/in/<verb>/<N>.<ext>. Language verbs: nodejs, python, bash, typescript, deno, go, rust, c, cpp, java. Host verbs: fs_read, fs_write, fs_stat, fs_readdir, kv_get, kv_put, kv_query, fetch, exec_js, env_get, recall, codesearch, memorize, browser, health. Git verbs: git_status, branch_status, git_log, git_diff, git_show, git_branch (inspect), git_add, git_commit, git_finalize, git_push (stage/commit/push), git_checkout, git_fetch, git_rm, git_revert, git_reset (mutate). Orchestrator verbs: instruction, transition, phase-status, prd-add/prd-resolve/prd-list, mutable-add/mutable-resolve/mutable-list, memorize-fire, residual-scan, auto-recall. Output lands at out/<verb>-<N>.json for nested verbs and out/<N>.json for root verbs.

Git is a first-class spool surface, not a shell command. A bash/sh/powershell body whose command is git-dominant is denied at the dispatch gate (deviation.bash-git-bypass), the denial naming the equivalent git verb. Inline tool args, JSON-form dispatch, and any attempt to bypass the spool (e.g. Bash(node script.js), Bash(python -c ...)) are denied at the Bash gate. Only gh runs directly via Bash.

Encapsulation friction is the cost of expressing a complex instruction through a prose/API-call surface: every capability negotiation happens inline, in natural language, re-parsed and re-validated per call, with no persistent record and no gate between intent and execution. The spool eliminates it by moving the instruction off the prose channel entirely: a complex operation is a code-like file write to a fixed path, not a described intent an interpreter must recover. Git is the clearest instance. Session close previously cost four-plus separate Bash events (porcelain probe, add, commit, push), each a distinct round-trip the agent had to sequence correctly by convention, with the porcelain-witness discipline riding on the agent remembering to make the probe its own event, exactly the kind of convention-dependent correctness the spool ABI removes. git_finalize {message} collapses this to one dispatch: stage-all, commit, re-check porcelain (refusing push over a dirty tree), push with rebase-retry, emitting git.commit/git.push witnesses. Four-plus turns to one; the porcelain gate becomes internal and structurally unforgettable rather than convention-dependent; the flow is dispatch-ledger-visible rather than scattered across shell history. Inspect verbs (git_status, git_log, git_diff, git_show, git_branch) and granular mutators (git_add, git_commit, git_checkout, git_fetch, git_rm, git_revert, git_reset) cover the remainder; raw git is never needed. All route through one host_git import resolving to the .exe directly (a Windows .cmd shim would defeat windowsHide and flash a terminal) and accepting a JSON argv array (a multi-word commit message must survive intact, not be shattered by whitespace-splitting). The general principle: a capability reached through a verb is auditable, gated, portable; the same capability reached through the shell is none of those, an unaudited free-text surface with no persistent trace.

File-based dispatch is what makes the channel legible: a request surviving context compaction, a response auditable after the fact, a ledger discriminating executed work from merely-narrated work, the system's sharpest signal/noise cut. Tool-call args are ephemeral, scoped to one call, non-persistent, non-replayable, uninspectable without re-running the agent, exactly the encapsulation-friction failure mode. A file-based surface inverts every property: the request is durable on disk before the watcher reads it, the watcher is a process detached from the agent, and the response is one JSON file auditable by any directory-reading tool, no re-execution required.

Agent-facing surface: one file per dispatch. The watcher writes out/<verb>-<N>.json (or out/<N>.json at root) on verb completion, carrying result payload plus metadata (exitCode, durationMs, timedOut, stdout, thrown-error stack); one read discriminates clean-completion, budget-kill, and data-return. Streaming execs keep incremental stdout/stderr in host-owned internal stream files the agent never parses directly, reading only the resolved JSON.

Boundary enforced at the Bash gate: any attempt at non-spool execution (node script.js, inline python -c) receives a denial that throws, naming the correct spool path, rather than mutating the call into a self-failing form. This is the discipline's only recognized execution surface; anything else is unverifiable whether the call happened at all. Independent support for the constrained-surface bet: SWE-agent's Agent-Computer Interface as the primary reliability lever, and contemporary fault-tolerant-sandbox work converging on policy-based interception plus transactional filesystem snapshots as the safe-execution baseline.3,12

from the gm field guide One folder. One contract. Watcher reads, agent writes, output lands in three files. The agent who tries to skip the folder discovers very quickly that the folder is not optional.

31. The Mutables File as Co-Equal Hook Gate

Hard Rule: Mutables File Co-Equal With PRD [PLAN, EXECUTE, EMIT]

.gm/mutables.yml lives alongside .gm/prd.yml. Every unknown surfaced during PLAN lands as a row with status: unknown. Write, Edit, NotebookEdit, git commit, git push, and the stop checkpoint are all hard-blocked by the dispatcher while any row has status: unknown. Rows flip to witnessed only when witness_evidence is filled with concrete proof (file:line, codesearch hit, exec output snippet).

Section 3's named-mutable type-state lived only inside the context window: a fact "tracked" purely in-reasoning evaporates on compaction, restart, or subagent spawn, an in-memory lattice with no durability guarantee. The mutables file externalizes it: an on-disk row the spool-dispatch gate reads independent of agent memory, the durable representation of the UNKNOWN|WITNESSED lattice.

The mutables.yml/prd.yml split is what-to-do versus what-must-be-true: PRD items can complete while a referenced mutable stays UNKNOWN, since item-completion and invariant-verification are orthogonal predicates the dispatcher checks separately.

The witness-evidence requirement (file:line, codesearch hit, exec output) blocks narrative-only state transition: an evidence-free status change is rejected on read, row stays UNKNOWN, EMIT gate stays closed. A discipline the agent could previously talk past becomes one a file structurally refuses to.

32. Discipline Isolation via @sigil

A discipline is a per-project namespace at <project>/.gm/disciplines/<name>/, own rs-learn DB and code-search index each. Exists against a specific failure mode: cross-namespace memory bleed surfacing a prior-project fact as a weak_prior that contradicts current code or references a nonexistent symbol. A hallucinated prior is strictly worse than no prior, it earns witnessing it never should have qualified for.

The @<name> sigil authorizes strict single-discipline isolation for recall/codesearch/memorize. Sigil-less default is asymmetric: reads fan-merge across default plus every line in enabled.txt, rank-tagged [discipline:<name>]; writes target default only.

The asymmetry is a least-privilege adaptation: broad reads are recoverable (a bad prior fails at the witness step), broad writes are not (a misplaced fact pollutes every future read in an unrelated namespace). Read-broad, write-narrow.

33. Synchronous-Under-Wasm Execution Contract

Hard Rule: Every Exec Carries A Real Timeout [EXECUTE - EMIT]

exec_js is synchronous under the wasm host: the orchestrator runs the user subprocess via child_process.spawnSync and blocks the watcher for the duration. Every dispatch carries a positive timeoutMs; missing or zero is a hard error. On timeout, partial streamed output is preserved and timedOut: true lands on the response sidecar.

Consequence of wasm's execution model: no thread to detach a subprocess into, no cross-dispatch process handle, no port for external attachment. Host runs, blocks up to timeoutMs, returns. No orphaned background work is representable.

exec_js is the JIT-execution substitute for internal deliberation: rather than reasoning silently about what code would do, the agent writes a hypothesis as executable code and dispatches it, and the response is ground truth about the hypothesis rather than a projection of it. Because witnessed execution categorically replaces reasoning-as-verification (section 3), the model's own extended-thinking/reasoning-effort budget is largely redundant work duplicating what exec_js will verify anyway; running gm with the driving model's reasoning-effort or thinking setting LOW or OFF is the recommended configuration, since gm substitutes cheap, real, externally-verifiable execution for expensive, unverifiable internal token spend on the same question.

This is a deliberate trade. The earlier generation of this project ran rs-exec as a native daemon with a port file, a task store, and a grammar of resumption verbs (tail, watch, wait, sleep, close). It was more flexible and also a fountain of operational pain: dead PIDs, port-leak races on Windows, idle-cleanup tuning, kill-tree edge cases on bash pipelines that escaped the parent group. The wasm-only surface gives up the long-task affordance and gets back operational simplicity: no daemon to supervise, no port to leak, no orphan-process accounting. The wasm-for-agents pattern has converged in recent industry and academic work as well: NVIDIA Developer Blog argues for WebAssembly sandboxing of LLM-generated code, and MCP-SandboxScan formalizes a WASI sandbox for untrusted MCP tool execution.20,19

Browser sessions are the one exception. browser opens a managed Chrome session that the wasm host holds open between dispatches (the host-side state lives outside the wasm instance, in the Node process that hosts it). Browser sessions persist across turn-stops and are reaped only on real-exit reasons (clear, logout, prompt_input_exit). The running_task_ids field on every dispatch response surfaces outstanding browser sessions so the agent sees them on every turn.

34. Hook Denials Throw, Never Mutate

Hard Rule: Throw on Denial [PLAN, EXECUTE, EMIT, VERIFY]

A hook that blocks a tool call throws an error with an imperative instruction string. It does not rewrite the call's arguments into a self-failing form, not a shell command that exits 1, not a Node one-liner that writes to stderr and process.exit. The thrown error is the entire denial surface.

The distinction is in how the failure is classified downstream. A thrown error reads as policy-refusal: the model updates its model of the surface and switches tools next attempt. An args-mutation producing the same exit code reads as "the tool is broken," reinforcing an incorrect model and converting a one-shot denial into a retry loop with no convergence.

Generalizes: any denial-issuing hook must throw, never mutate. Throw signals "use a different tool"; mutate signals "run this corrected call," a rare need, almost never what a guardrail wants.

35. CI Is the Build

Hard Rule: Push, Watch, Fix On Red [EXECUTE, EMIT, VERIFY]

For the Rust crates in this org (rs-plugkit, rs-search, rs-codeinsight) and the gm publish chain, git push triggers the build pipeline. cargo build and cargo test are not run locally. Toolchain mismatches, rustc version skew, and missing system dependencies never block a push, push, watch CI via gh run watch dispatched through the spool, fix on red.

The witness for a Rust crate is green CI on pushed HEAD, not a green local build. Every crate compiles to wasm32-wasip1 as a cdylib, the loaded artifact; local cargo proves nothing about the wasm import surface or the published tarball's contents.

Cascade is the witnessing infrastructure: sibling push fires cascade.yml (gh workflow run + PUBLISHER_TOKEN) into rs-plugkit/release.yml, which rebuilds wasm, publishes to plugkit-bin Releases and npm (plugkit-wasm), bumps gm.json::plugkitVersion and the sha256 pin. That bump triggers publish.yml (npm publish . from repo root, itself the published artifact), a fresh gm-skill tag. Witness lands at the tree's leaf, not its root.

Pairs with Browser Witness (section 23): same witnessing principle, different execution boundary, and in both cases the witness lives outside the agent's local environment because local environment cannot answer the question.

36. The Residual-Scan Stop Gate

Hard Rule: Stop-Checkpoint Residual Challenge [COMPLETE]

When .gm/prd.yml is empty or missing AND no open browser sessions AND no running tasks, the stop-checkpoint gate fires one residual-scan challenge before allowing stop. The agent either re-enters planning, appends to the PRD, and executes, or explicitly states "residual scan: none reachable in-spirit." The marker file .gm/residual-check-fired makes the challenge one-shot per stopping window; the marker clears at the next prompt-submit so the next stopping window starts fresh.

Empty-PRD is necessary but not sufficient for done: the PRD records decided work, the gate probes the complement set, what should have been decided but wasn't. The gap is where non-monotonic termination hides, enumerated-items-done-and-stopped, leaving in-spirit residuals undiscovered.

One-shot per stopping window prevents the gate itself becoming noise: a refused second stop in the same window would loop without termination; the marker clears at next prompt-submit, defining a fresh window.

Pairs with Maximal Cover (17, in-spirit boundary at PLAN) and Fix on Sight (22, EXECUTE-time PRD folding): this gate is the COMPLETE-phase backstop for what both miss.

37. Meaning Through the Agent, Mechanics Through Code

Hard Rule: Semantic Work Is the Agent's Own Judgment [PLAN - EXECUTE - EMIT]

Any task whose correctness depends on understanding, summarize, classify, extract intent, rewrite, translate, semantic dedup, score, label, decide-if-two-texts-mean-the-same, is the agent's own inline judgment, not a separate model call. plugkit is the harness and the agent IS the model it dispatched, so each semantic decision is the agent deciding on the spot from the current situation. When the work is many independent items, fan them out as N parallel spool dispatches in one message, one item per dispatch.

A regex deciding a meaning question is deterministic-faced noise: it terminates, returns a value, and the value bears no relation to the question. Meaning routes to the agent's own judgment, mechanics to deterministic code, each engine kept on the work it carries signal for. Mechanics: parse, sort, hash, count, decidable by procedure. Meaning: language understanding, is this a fix or a feature, does this docstring contradict its function, are these error messages semantically identical.

A keyword-list or regex-over-meaning-phrases loop is Nothing Fake's failure mode (section 24) applied to judgment: it looks right because it terminates, and is unrelated to what was asked.

Fan-out is the parallelism half: N items to N parallel dispatches in one message, never batched, a batched judgment dilutes per-item signal via averaging over the batch. Negligible cost delta, large quality delta; mirrors memorize-fire's fan-out (section 12), one unit per call.

38. The Continuation Invariant: A Toolless Turn Is A Stop

Hard Rule: While The Chain Is In-Flight, Every Turn Ends In A Tool Call [PLAN - EXECUTE - EMIT - VERIFY]

Every programming agent reads only tool calls and their outputs; an assistant message that ends in prose with no tool call is the turn ending, and the harness halts there. So while the chain is in-flight (phase != COMPLETE or prd_pending > 0) the agent never ends a turn in prose and never writes a summary or recap, summary is a stop. The only event that authorizes a prose-only turn is plugkit returning phase=COMPLETE and prd_pending=0. Before any apparent stop, the agent dispatches phase-status and rechecks; a non-terminal phase means the urge to stop was drift, and the recovery is to dispatch instruction and continue.

Mechanical root of the dominant long-horizon failure: a completed sub-piece plus a "here is what I did" paragraph plus zero tool calls reads to the harness as terminal, independent of the agent's intent. No amount of in-context "remember to continue" survives, closing sentence and stop are the same event by construction.

Two faces of the same failure: summary (backward-facing) and deferred intent (forward-facing, a final sentence naming the next move instead of taking it), the latter slipping past a summary-only prohibition because it self-reports as continuation. The corrective is uniform: take the described move, same turn, no exception for surfacing a decision, which lands through a tool call, never prose alone.

Inverts the default: not "continue unless done" but "dispatch until the gate says done." Doneness is plugkit's COMPLETE-gate pronouncement, never an agent self-report; self-reported doneness substitutes the agent's judgment for the gate's, the dominant premature-halt source on long-horizon tasks.

Depends only on the verb spool, no hook/stop-interceptor/platform-continuation-feature (each host-specific, each a single point of host-coupling failure). The invariant is prose served through instruction, eliciting identical behavior on any model that reads and calls tools, the only mechanism achieving cross-host predictability.

38.1 Terse Output

Prose around tool calls is minimized to substantive content only: real findings, one-line decision-plus-reason, blockers. Mundane output (boot-probe narration, dispatch echoes, restated prose, status recaps) is suppressed. Composes with, never overrides, the continuation invariant: terse means shorter, never zero tool calls, never silent work. Output shrinks toward zero exactly where the tool calls already carry the meaning.

39. Spool Dispatch Replaces Platform Hooks

Hard Rule: The Gate Is The Spool, Not A Host Hook [PLAN - EXECUTE]

Orchestration state is tracked via marker files in .gm/ and gated at the dispatch layer, not through any agent-host hook event. No skill prose, gate denial, or instruction names a host-specific invocation form (a plugin-namespaced skill, a host-specific subagent type). The gate-satisfying action is always a portable spool dispatch: write .gm/exec-spool/in/instruction/<N>.txt and read the response.

Superseded design: host PreToolUse hooks emitting host-specific permission JSON and naming host-specific skill/subagent forms. Invisible coupling: under a different harness the hook never fires or names an absent form, silently disabling the gate. Subagent orchestration compounds it: an isolated subagent's prose-summary return reads as a stop to its parent (section 38).

Replacement: marker files plus spool dispatch. checkDispatchGates reads .gm/prd.yml, .gm/mutables.yml, needs-gm/gm-fired markers, returns allow/deny naming the next portable verb, never a host-specific tool. Every prior hook-enforced behavior survives as a host-invariant spool gate. Legacy wasm_hooks exports are dead ABI, no caller.

40. The Memory Architecture: File-Vec Recall and the rs-learn Algorithm Layer

Memory splits into a live layer and an algorithm layer. Live: the file-vec store. memorize-fire embeds via in-wasm bge-small, writes two files per memory (text under .gm/disciplines/<ns>/<key>.json, its 384-dim embedding under .gm/disciplines/<ns>-vec/<key>.json). recall/auto_recall read fresh, score cosine x recency (30-day half-life, 0.4 floor), dedup by Jaccard overlap. Git-resident, project-scoped.

Pruning is first-class: a false-positive recall hit is worse than a miss (it poisons every future matching query, not just this one). memorize-prune deletes text+embedding atomically, by explicit key or review-only candidate query, never blind similarity-delete (an automatic similarity prune is itself a bad-memory generator). Precision over retention.

The algorithm layer, rs-learn, is a separate wasm cdylib, five continual-learning cores: instant rank-2 Hebbian MicroLoRA with norm-bound and EWC correction (sona-derived); EWC++ deep core with online Fisher EMA and z-score boundary detection; bi-temporal knowledge graph invalidating (never deleting) contradicted edges, point-in-time queryable (graphiti-derived); 8-head graph-attention with relation/recency edge features; FastGRNN sparse-plus-low-rank router, epsilon-greedy over model/context-bucket/temperature/top_p/confidence. Owns no IO: imports host_kv_*, host_vec_search, host_log, host_now_ms, exposes one JSON dispatch surface.

Design: host owns IO/scheduling/embedding/transport, rs-learn owns math only, isolation-testable and reusable by any six-import-satisfying host. Reachable from the agent-facing spool via plugkit's learn verb, routing through dispatch_json over a PlugkitKv backend (three-method KvBackend trait over kv_get/kv_put/kv_query). Math-no-IO / host-no-algorithm is the reproducibility split: cores carry no platform assumption, host carries no learning-rate constant.

Stateless-host dispatch constructs a fresh LearnSession per learn verb; session-RAM-only state evaporates between dispatches. The bi-temporal graph is unaffected (every edge written straight through host_kv_*, durable by construction). The three weight-bearing cores (MicroLoRA/EWC adapter, FastGRNN router, GAT attention) persist explicitly: serialize learned tensors to a fixed KV namespace after every mutation, lazy-reload on empty in-RAM slot. Only genuinely learned state persists (low-rank weights, quality counters, feedback count, edge-feature weights); seed-deterministic matrices reconstruct from seed. Length-guarded on load: dimension mismatch reconstructs clean from config rather than corrupting, no cross-run poisoning. Namespace resolves under the project's own .gm tree, no cross-project weight bleed. This makes the cores genuinely continual across the spool rather than dispatchable-but-amnesic.

The router closes into the turn lifecycle: every instruction dispatch embeds the active prompt, routes through FastGRNN, attaches a route_hint (model, temperature, top_p, context bucket, confidence) to the response. Fresh projects fall back to a rule policy (algo: rule); accumulated record_outcome dispatches cross a trajectory threshold and the router switches to its learned low-rank weights, epsilon-greedy exploring off-policy. Advisory only, never forced. Concrete shape of agent-as-LLM: model-selection becomes a learned, per-turn, persisted policy compounding over project life rather than a fixed constant reset each turn. The hint's prompt embedding reuses the recall path's per-query cache, no second embedding pass.

The MicroLoRA core is a learned scoring head over embeddings, not an embedding transform: rank-2 adapter over a 384-dim bge-small vector emitting a per-target score (apply_adapter: logit[k] += B - (A - embedding)); feedback nudges factors A/B from a single 0-1 trajectory-quality score, a norm-bounded, EWC-corrected Hebbian update so new adaptation never erases old. Learns which targets this project's embeddings should favor. The quality signal is the one thing only the model-in-the-loop holds, so the agent self-reports it at turn close and the head's weights move toward proven-good targets. Second face of agent-as-LLM: router learns which model to dispatch, MicroLoRA learns to score this project's targets, both from the same self-report. Open integration: wiring the head's score into recall re-ranking.

The self-report closes the loop end to end. The bi-temporal graph auto-populates: every memorize-fire inserts a graph edge over the same embedding already computed for file-vec storage (zero marginal embedding cost); pruning invalidates the edge in the same step, no dangling reference. Because plugkit is the harness and the agent is the model it dispatched, the model knows its own identity and outcome and reports them directly, rs-learn never calls out for that judgment, it emits the need and the agent supplies it.

41. The WASM Migration

The orchestrator runs as a single WebAssembly cdylib loaded by a thin JS wrapper under Node/Bun; zero native binaries per platform. One artifact runs identically across every JS-host OS, sha256-pinned, no per-architecture build matrix, no platform install path. The wasm imports a minimal host contract (filesystem, KV, vector search, fetch, subprocess, clock, log); JS satisfies it by delegating to Node, wasm carries logic, host carries IO. Same module loads in-browser given an import-satisfying host.

Two size stories, one artifact. Orchestration logic (state machine, gates, dispatch, verb routing) compiles to hundreds of KB. The shipped artifact is large (~143 MB) because it embeds the embedding model in-wasm (BAAI/bge-small-en-v1.5, 12 hidden layers, 384-dim output, safetensors blob plus tokenizer): embedding runs wasm-side, no host delegation, no external service, offline and text-leak-free by construction. Deliberate trade: one large download buys a self-contained, reproducible, network-independent memory system over a small binary phoning home per embedding.

Single-artifact discipline underwrites stability: atomic write (temp file, rename); version drift between running instance and on-disk version file triggers clean self-respawn, never partial reload; embedding init witnessed by embed_init_ok before first recall. Shared resources (KV, vector index, embedding model) are host-owned and session-shared across every verb dispatch, amortizing model load once per session.

42. The Rust Service Crates: codeinsight, search

Two zero-dependency static Rust crates, plus execution folded natively into rs-plugkit. Language execution (nodejs, python, bash, typescript, deno, go, rust, c, cpp, java through the spool), browser sessions, and per-call timeout/task isolation (mandatory positive timeout, wasm host runs subprocesses synchronously, unbounded child blocks the watcher) were previously a standalone rs-exec crate; that crate is retired, its surfaces reimplemented directly inside rs-plugkit's wasm_dispatch. rs-codeinsight: AST x-ray, tree-sitter across thirteen languages, 500-file project analyzed under 200 ms in one pass, giving an unfamiliar repo's files/symbols/shape without exploratory grep. rs-search: hybrid finder, BM25 lexical scoring with identifier-aware tokenization (snake/kebab/camel splits) fused with vector similarity via Reciprocal Rank Fusion, .gitignore-disciplined, PDF-ingesting, content-addressable embedding cache so re-index only re-embeds changed content.

Model selection splits meaning from mechanics stack-wide. Semantic work (summarize, classify, judge-equivalence) routes to a small fast model via fetch, one item per call, parallel by default. Mechanics (parse, sort, hash, count) stay in deterministic Rust. bge-small embeddings for 384-dim output at small weight footprint relative to recall quality. Same motivation as the wasm migration and the rs-learn boundary: one concern per layer (model = meaning, Rust = mechanics, host = IO, algorithm crate = math), each independently testable, reproducible, swappable.

The static meaning/mechanics split fixes which engine handles a job, not which model handles a meaning-job. FastGRNN decides the latter: the per-turn route_hint (section 40) recommends model/temperature/top_p learned from this project's prior turns rather than a hard-coded constant. Compositional: layer-concern split decides model-versus-Rust deterministically; within the model tier, the router turns which-model into a learned, persisted, per-turn policy compounding over project life. Agent-as-LLM is the same one-concern-per-layer discipline one level up: harness owns orchestration, router owns model-selection policy, neither hard-coded into the other.

43. Deviation Gates: Predictability Across LLMs

Cross-model predictability is a worst-case target: a weaker driver should hit the same SNR floor as a strong one, because gates hold the line, not model capability. Kept not by trusting the model but by pure on-disk-state checks firing identically under any LLM/harness: long-gap-no-instruction refuses a non-orienting verb >5 min post-last-instruction; complete-chain-poll flags re-dispatch of instruction on an already-terminal chain; push-dirty refuses push from unclean worktree; client-edit-no-witness refuses COMPLETE-transition absent a browser witness on a client edit. Gates never mutate the command, only return a refusal whose text is the next instruction, steering a weak model the way a strong one self-steers.

A gate's signal quality is its attribution accuracy: fire on agent mistakes, stay silent on the mechanism's own bookkeeping. Two refinements: the long-gap clock resets on fresh-watcher-boot (a version-upgrade-induced gap is not agent-caused, so boot stamps last-instruction to now); complete-chain-poll fires only for session-attributed dispatches (cron/stop-checkpoint empty-body probes against a terminal chain are not agent re-polling, counting them inflates the rate without reflecting agent behavior). Generalizes: a deviation is an agent action gone off-discipline, so the gate must discriminate agent dispatches (session-carrying) from harness-internal bookkeeping (not), or the signal measures the wrong quantity.

Three-Layer Admission Filter

Sections 2-37 specialize a single structural invariant: every candidate operation passes three orthogonal admission tests in sequence before commit.

candidate to [L1 cost] to [L2 bounds] to [L3 direction] to execute

L1 Cost. Amortized cost <= prior-best envelope. Unmeasured marginal = inadmissible. ORIENT (section 21) is the baseline probe.

L2 Bounds. Single-writer per surface (|F|=1). Hard cap per managed resource; backpressure to defer queue at watermark. State outside a sanctioned surface = unreconcilable.

L3 Direction. Lyapunov: Deltad >= 0 rejects. Audit tuple (id, hash, ts) per accepted write. Trajectory classifier over rolling window: convergent | flat | divergent | chaotic; non-convergent holds. VERIFY's four-observation window [worktree-clean, remote-pushed, prd-empty, mutables-witnessed] is the convergence criterion.

Closure anti-shape (universal L3 violation). The agent's response body is not a mutation surface. Any artifact composed in the response that displaces a sanctioned dispatch fails L3 because no (id, hash, ts) exists, the recall index does not see it, the trajectory classifier does not register it, the gate does not advance. Recognition is structural, not by enumeration.

Maturity-first emit. First emit = closure of the destructive transform. Partial emit externalizes residual completion cost as implicit unaudited state; the next session pays the reconstruction cost. Closure-exceeds-reach decomposes along dependency edges (Maximal Cover DAG, section 17), never along schedule.

Invariants. Measurement gates optimization. Bounds prevent cascades. Direction eliminates waste. Composability: layers are independently tunable.

The five phases are the filter at escalating commitment. PLAN runs L1+L2 cheaply via ORIENT and PRD-DAG. EXECUTE runs L3 over witnessed mutations into the central store. EMIT runs L3 audit on disk. VERIFY runs the trajectory classifier over the four-observation window. The filter is the engine; the phases are scheduling.

Code Invariants

Instruction prose extends to code emission. Every artifact carries the same structural disciplines.

  • State space minimized. Sequential downward flow; state flags evaluated in one phase. External inputs through a unified queue before mutation. State changes = explicit assignment, never buried side effect.
  • Hardware reality. Benchmark before abstracting. Scope passed explicitly; closures hide scope-resolution cost on hot loops. Pools over allocation. Native data flow on performance paths; reject Promise chains / class hierarchies / operator overloading there.
  • Flat structure. Denormalized graphs over nested documents. Partial-field updates. Bytes over JSON for transport; varint variable-width fields; pre-compute exact size and allocate once. Lexical ordering for deterministic tie-breaking.
  • 200-line vertical slices. One responsibility per file. Complete input->process->output. Zero-config defaults correct for 90%. Universal runtime: browser, Node, mobile, Bare.
  • Async boundary explicit. Sequential awaitable primitives. No implicit callback ordering. Unified error channel. Tests await real ops; mock-free.
  • Naming by scale. <50 lines: single-letter algebraic. 50-200: short descriptors. >200: full names. Iterators short; public APIs explicit.
  • Fail fast, loud, deterministic. Halt on precondition violation with exact state. Assert on emitted semantics, not return values. Sentinel words + checksum headers verified on every access.
  • Binary transport, append-only persistence. Lexical cursors for sparse reads. Append-only sequence for replay. Chunked by lexical range; modify only the touched chunk.
  • Single focused task per session. No drive-by refactors. Pre-compute and inline; code growth < cognitive overhead.

Token discipline. English describing intent = liability when code can encode it. Comments = liability when names + structure encode the same. Duplication that must sync = liability. Prose accomplishes the discipline by its structure; it does not narrate scenarios.

44. Discussion

44.1 Relationship to Formal Methods

Surface similarity to model checking (named state variables, explicit transitions) and design-by-contract (pre/post-conditions gating advancement), at a coarser abstraction: states are task phases not program states, contracts concern epistemic state not program correctness. Route families (13) function as typing domains, partitioning valid-operation space rather than verifying program properties.

44.2 Relationship to Test-Driven Development

Loose TDD analogy: both front-load verification. Divergence: verification here is witnessed execution, not a persisted test suite during EXECUTE; the single integration test (26) is COMPLETE-phase witnessing residue. The 200-line budget trades regression depth for context-window economy.

44.3 External vs. Self-Regulation

External enforcement over self-regulation is the recurring design choice: the same inference process that reasons through a problem reasons through shortcut-justification, so introspective correction is not a fixed point of the failure. Marker-file gates read .gm/prd.yml/.gm/mutables.yml independent of agent judgment; hard rules 21-37 live in orchestrator output and marker gates, not in-context reasoning. The retired hook-event architecture made enforcement contingent on a working callback pump; the current architecture reduces it to a file read.

44.4 Overhead vs. Benefit

Every backward transition costs re-planning/re-executing discarded partial work, offset in production by avoided late-stage rework: the snake-back surfaces problems while scope is still bounded, versus COMPLETE-time discovery after compounding across all subsequent work.

44.5 Limitations

  1. Finite unknown assumption: termination requires a finite discoverable-unknown set, may not hold adversarially/chaotically. Two-pass reclassification (9.2) doesn't prevent structural (non-transient) persistence across both passes; persistence past pass 2 is prescribed pause with blocker in the PRD header, not indefinite reclassification, else unbounded reclassification cycles.
  2. Witnessed execution cost: real execution against expensive external APIs may be prohibitive for some paths.
  3. Memory staleness window: the creation-to-verification interval admits stale-memory-induced error.
  4. Browser witness scope: single-instance managed Chromium serializes, no wave parallelism.
  5. Test budget ceiling: 200-line cap forces compact coverage; genuinely large subsystems may be under-witnessed within it.

45. Conclusion

The complete gm discipline: a phase-gated FSM, a mutable-naming/witnessed-execution type-state framework, an in-phase quality discipline, a governance layer of seven route families and four state planes, second-generation autonomy/scope rules, operational hard rules, and a third generation of dispatch-boundary invariants moving the discipline onto disk into marker-file gates.

Not aspirational: every rule is enforced by the spool-dispatch gate below the decision layer, by disk-state the gate reads independent of the agent, or by hard-rule encoding in the instruction response the agent cannot reason around. The agent executes within the envelope; the envelope prevents local-optimality collapse of the discipline.

One npm package, one wasm artifact, every Claude-style host: reliability depends substantially on lifecycle control, not model capability alone, and on relocating discipline-state out of agent memory and host event pumps into externally-readable artifacts, a spool file, a mutables row, a stop-checkpoint marker, a CI run. The agent participates in the discipline, does not keep it; the host delivers it, does not enforce it. A practical engineering artifact, not a theoretical claim.

Every rule above measures one quantity: gm is 14000+ supervised-modification hours raising the signal-to-noise ratio, the fraction of every token/action constituting genuine progress rather than drift, fabrication, amnesia, or premature stop. Token austerity is that objective under a cost prior, the cheapest signal is the one never emitted. Phase gates, witnessed mutables, Maximal Cover, the continuation invariant, the spool ledger, and deviation gates are the catalogue those hours converged on, each a denoising operator over one noise class between intent and shipped software.
