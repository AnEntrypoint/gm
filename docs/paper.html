The Complete Discipline: Lifecycle Control,
Governance, and Operational Hard Rules
for LLM Coding Agents

AnEntrypoint
May 2026 · the complete discipline, in one document
Abstract
LLM-based coding agents exhibit failure modes that self-regulation cannot fix: silent assumption propagation, state drift, memory loss, premature closure, and fabricated outputs presented as results. This paper consolidates the gm discipline into a single document. We describe the phase-gated state machine (PLAN→EXECUTE→EMIT→VERIFY→COMPLETE), the named-mutable and witnessed-execution framework, the in-phase quality discipline, the governance layer with its route families and quality metrics, and the second-generation rules that emerged from production use: Maximal Cover, bounded autonomy, the learning loop, and the execution budget model. We also document operational hard rules absent from earlier papers: Fix-on-Sight, Browser Witness, Nothing Fake, web-search-before-pause, the ORIENT discipline, and the single integration test policy. A third generation of rules covers the dispatch boundary itself: the spool surface as universal execution channel, the mutables file as externally-enforced unknown-tracking, discipline isolation via the @sigil, daemonize-by-default exec semantics, throw-not-mutate hook denials, CI-as-build for cross-platform witness, the residual-scan stop gate, and semantic routing through a haiku-backed text-processing agent. Together these form a complete behavioral envelope for coding agents, packaged as a single wasm-backed skill shipped on npm.

1. Introduction

LLM-based coding agents, systems that autonomously write, test, and deploy software, have become increasingly capable, but their reliability remains uneven. In our experience maintaining a single-skill agent harness (gm-skill, npm-published, wasm-backed) loaded across every Claude-style skill host, we have observed three primary failure categories:

  1. Assumption propagation: Agents narrate hypotheses about code behavior without verifying them, producing cascading errors when assumptions prove false.
  2. State amnesia: Multi-step tasks lose coherence as context windows fill, with agents forgetting earlier decisions or re-solving solved subproblems.
  3. Premature closure: Agents declare work done before conditions are met, ship placeholders as results, skip verification steps, or stop short of the full scope of a request.

Self-regulation fails against these patterns because the same model that reasons through a problem also reasons through the justification for shortcuts. External behavioral enforcement, the spool-dispatch gate below the agent’s decision layer, marker-file stop barriers, orchestrator-enforced phase transitions, and hard rules served by the wasm instruction verb, is the only reliable mitigation.8,9 Recent empirical studies of agentic failure across multi-agent frameworks and tool-use benchmarks converge on the same diagnosis: premature termination, assumption propagation, and over-helpful substitution are dominant under unrestricted self-regulation, and constrained-pipeline alternatives outperform free-form agents on real software-engineering workloads.7,3

This paper is the complete reference for the gm discipline as it stands in production. Sections 2–8 cover the lifecycle model. Sections 9–12 cover the in-phase quality discipline. Sections 13–16 cover the governance layer. Sections 17–20 cover the autonomy and scope rules. Sections 21–29 cover operational hard rules. Sections 30–37 cover the dispatch-boundary rules that move the discipline out of the agent's context window onto disk and through the spool-dispatch gate.

2. The State Machine Model

2.1 Phases

The framework defines five phases with forward-only progression under normal conditions. PLAN discovers all unknowns (exits when a full pass adds zero new unknowns). EXECUTE resolves unknowns via witnessed execution (exits when all mutables are KNOWN). EMIT writes files to disk (exits when post-emit matches pre-emit exactly). VERIFY performs end-to-end witnessed validation (exits when .prd is empty, git is clean and pushed, and CI is green). COMPLETE updates documentation and closes the session.

The agent dispatches the instruction verb on every turn; the wasm orchestrator (rs-plugkit/src/orchestrator/) returns each phase’s imperative body, generated per call with the current FSM state, PRD, mutables, and recall pack baked in. The prose can never drift relative to the live state of the project because the live state is what produced it. There are no per-phase SKILL.md files anymore; the shipped skill (gm-skill) is a single ~12-line body that names the agent as the state machine and tells it to dispatch instruction on every turn, then dispatch the next verb the response names. Plugkit does not advance phases on its own; the agent advances them by dispatching transition.

2.2 Forward Transitions (Ladders)

A phase advances only when its exit condition is satisfied. Conditions are conjunctive: every sub-condition must hold simultaneously. This prevents partial advancement where code is written before unknowns are resolved.

2.3 Backward Transitions (Snakes)

Any newly discovered unknown at any phase triggers an immediate backward transition:

  • EXECUTE → PLAN: new unknown, or mutable unresolvable after 2 passes
  • EMIT → EXECUTE: logic error in known mutable; EMIT → PLAN: genuinely new unknown
  • VERIFY → EMIT: broken file output; VERIFY → EXECUTE: wrong logic; VERIFY → PLAN: wrong requirements

The intent is to prevent agents from patching around problems rather than reconsidering their assumptions. An agent that encounters unexpected behavior during EMIT must regress to fix it at the root, not accumulate workarounds.

Observation (Termination). If the set of discoverable unknowns is finite and each planning pass reduces the set of undiscovered unknowns, the process terminates. We typically observe convergence within 2–4 planning passes.

3. The Mutable Discipline

3.1 Named Unknowns

A named mutable is every unknown fact needed to make a decision or write code, explicitly named before action is taken. A mutable has a name, an expected value, a current value (initially UNKNOWN), and a resolution method. The agent does not proceed past an unresolved mutable.

The intent is to make uncertainty visible. Generating plausible-but-unverified code is the primary source of assumption propagation failures.

3.2 Resolution by Witnessed Execution

A mutable is resolved only when its value is assigned by the output of real execution, running actual code against actual services with actual data. The following do not constitute resolution:

  • Reading documentation (may be outdated)
  • Reasoning about expected behavior (may be wrong)
  • Examining source code (static analysis misses runtime behavior)

Only witnessed output counts as ground truth. Before that, a value remains unknown regardless of how plausible a guess might be. This matches a growing body of work on code-generation hallucination: surveys and grounding-based mitigations show that reasoning-only verification systematically misses runtime-only failure modes, and iterative grounding against real APIs is what actually reduces fabrication.10,11

3.3 The Two-Pass Rule

If a mutable remains unresolved after two execution passes, it is reclassified as a new unknown and the agent snakes back to PLAN. This prevents infinite loops where the agent repeatedly attempts the same failing resolution strategy.

3.4 Surprise Absorption Prohibition

When execution produces unexpected output, the agent must not silently absorb the surprise. Every discrepancy between expected and actual output generates a new named mutable, triggering a return to PLAN. The goal is to prevent workarounds that accumulate technical debt.

4. Work Item Management: The PRD

4.1 Structure

The system tracks work through a .gm/prd.yml file. Each item carries an id, imperative subject, status (pending → in_progress → completed), acceptance criteria, edge cases, bidirectional dependency links, effort estimates, route family tag, and load score (0.0–1.0: consequence weight). Writing the PRD is mandatory for any task touching more than one line in one file.

The load score governs verification budget: verification_budget = load × (1 − tier_confidence). Items with load > 0.75 must be witnessed before EMIT.

The mandatory-written-specification stance is under independent empirical study: a SANER 2026 registered report formalizes the hypothesis that specifications preceding code reduce errors, and the Agentless pipeline shows that a constrained three-phase localize/repair/validate decomposition beats free-form agentic loops on SWE-bench at lower cost.16,7

4.2 Wave-Based Parallel Execution

The PRD’s dependency graph enables parallel execution. Each wave identifies all pending items with satisfied dependencies, launches up to 3 concurrent subagents, and upon completion checks newly unblocked items for the next wave. Independent items are never executed sequentially.

4.3 Stop-Hook Enforcement

A marker-file gate inspects the PRD at the stop checkpoint. If .gm/prd.yml has remaining items, stop is denied. The same gate refuses stop when git is dirty or unpushed. These gates run independently of any hook event pump, the dispatcher reads files on disk, not session callbacks.

4.4 Cross-Session Continuity

The PRD is durable across sessions. Items left over from a previous session are the current session’s work the moment they are discovered. The agent finishes every item before declaring complete, including ones the current user message did not mention. Sessions are scheduling; the PRD is the contract.

5. Cross-Session Memory Architecture

5.1 Memory Types

The system implements file-based persistent memory with four semantic types:

user:
Profile, role, expertise, preferences, enables tailored collaboration
feedback:
Behavioral corrections and confirmations, persistent learning channel
project:
Ongoing work context, deadlines, decisions, situational awareness
reference:
Pointers to external systems, navigational knowledge

Each memory is stored as a Markdown file with YAML frontmatter, indexed in MEMORY.md. The index loads at conversation start; individual memories are read on demand. The pattern parallels recent agentic-memory architectures: A-MEM organizes notes Zettelkasten-style with embedding-indexed dynamic links, Mem0 builds a production-grade long-term store with consolidation and salience retrieval, and MemoryAgentBench shows that even strong RAG-augmented agents still trail humans on accurate retrieval, test-time learning, and selective forgetting, motivating per-turn auto-recall rather than session-scoped retrieval.13,14,15

5.2 Exclusion Principle

The system avoids storing information derivable from current project state: code patterns, architecture, file paths, git history, debugging solutions. Memory staleness, acting on outdated cached information, is among the more damaging failure modes.

5.3 Verification Before Recall

When a memory names a specific file, function, or flag, the agent must verify its continued existence before recommending action. A memory is a claim about a point in time, not current reality.

5.4 Feedback Loop

The feedback memory type creates a persistent behavioral correction channel. When a user corrects the agent or confirms a non-obvious approach, the agent records the rule, reason, and application context. This addresses the “Groundhog Day” problem where agents make the same mistakes across sessions.

6. Implementation: The gm Orchestrator

6.1 Single-Skill Architecture

There is one published skill: gm-skill. Its SKILL.md body is roughly twelve lines and contains no phase prose; it instructs the agent to dispatch the instruction verb through the spool on every turn. Every phase’s imperative content is generated by the orchestrator at call time. The earlier multi-skill chain (gm, planning, gm-execute, gm-emit, gm-complete, update-docs, governance, code-search, browser, ssh, pages, create-lang-plugin, memorize, textprocessing) is gone, its behaviors are now properties of the instruction response and the verb dispatch surface.

6.2 Spool Dispatch and Marker Gates

State-machine enforcement is marker-file driven, not hook-event driven. Earlier generations of this project relied on host-process hook callbacks (session-start, prompt-submit, pre-tool-use, stop). The current architecture replaces that event pump with a single dispatcher, gm-starter/lib/spool-dispatch.js::checkDispatchGates(sessionId, operation), that reads files on disk before any tool executes. The CLI/bootstrap layer (plugkit, rs-exec, every downstream platform) calls the gate; on denial, the reason text surfaces to the model. The marker files in .gm/ are the discipline’s state machine:

.gm/prd.yml
Existence triggers the needs-gm gate: until the gm-skill has been invoked this turn, write/edit/git operations are blocked.
.gm/mutables.yml
Any row with status: unknown blocks Write, Edit, NotebookEdit, git commit, git push, and the stop checkpoint.
.gm/needs-gm
Written by bootstrap; signals the dispatcher that the agent must invoke gm-skill this turn before tool use is permitted.
.gm/gm-fired-<sessionId>
Written when the agent invokes Skill(skill="gm-skill"), clears the needs-gm gate. Cleared again at turn start.
.gm/residual-check-fired
One-shot marker that ensures the residual-scan challenge fires exactly once per stop window; clears at the next prompt-submit.

Session-start bootstrap (gm-starter/lib/skill-bootstrap.js) ensures the wasm artifact is present, instantiates it, writes .gm/exec-spool/.bootstrap-status.json, and lays the markers. The orchestrator’s auto-recall verb fires at every turn entry, deriving a 2–6 word recall query from the user prompt and injecting “## Recall for this prompt” into the next instruction response.

The agent cannot bypass enforcement through reasoning: the markers live on disk, the dispatcher reads them independently of the agent’s in-context state, and the gate runs below the tool surface the agent operates through. Marker-driven dispatch is strictly more robust than hook-event dispatch because there is no event pump to miss, no callback registration to forget, and no host-specific lifecycle to depend on, only files and one read.

6.3 Unified Code Execution

All code execution passes through the exec_js host verb. The dispatch path: agent writes .gm/exec-spool/in/<lang>/<N>.txt, the wasm watcher reads it, the host runs child_process.spawnSync with the appropriate runtime, output streams to out/<N>.out/.err, metadata to out/<N>.json. Supported language verbs: nodejs, python, bash, typescript, deno, go, rust, c, cpp, java, plus the browser verb for live-page work.

6.4 Single-Package Deployment

The orchestrator deploys as one npm package, gm-skill, loadable by any Claude-style skill host. The companion gm-plugkit npm package wraps the wasm artifact and the bootstrap shim; the gm-skill package depends on it. A cascade pipeline links five wasm-only Rust repositories (rs-plugkit, rs-exec, rs-learn, rs-search, rs-codeinsight) and the plugkit-bin Releases repository; a push to any of them triggers the relevant publishes and a fresh gm-skill on npm with no manual steps. An earlier generation fanned the harness across fifteen per-platform downstream repos (gm-cc, gm-gc, gm-oc, gm-kilo, gm-codex, gm-qwen, gm-copilot-cli, gm-hermes, gm-thebird, gm-vscode, gm-cursor, gm-zed, gm-jetbrains, gm-antigravity, gm-windsurf); those are archived. One npm install replaces fifteen plugin installs.

7. Constraint Tiers

The framework defines four tiers of constraints with decreasing severity:

Tier 0 (Absolute):
no_crash, no_exit, ground_truth_only, real_execution, enforced by the spool-dispatch gate, cannot be overridden
Tier 1 (Quality):
max_file_lines=200, hot_reloadable, checkpoint_state
Tier 2 (Modularity):
no_duplication, no_hardcoded_values
Tier 3 (Style):
no_comments, convention_over_code

8. Pre-Emit Verification Protocol

The EMIT phase implements two-stage verification. Pre-emit: the agent imports the target module, runs proposed logic in-memory with real and error inputs, records expected outputs, and verifies all gate conditions hold simultaneously. Only then does it write to disk.

Post-emit: the agent re-imports the actual file from disk (not in-memory), runs identical inputs, and compares output exactly. Known variance triggers fix-and-reverify; unknown variance triggers a snake to PLAN. This catches errors that test-after-write approaches miss: serialization bugs, import resolution differences, and file system race conditions.

9. In-Phase Quality: The Search and Code Discipline

9.1 Codebase Search Protocol

Before naming any unknown as absent, the agent must conduct a minimum search campaign:

  • Start with a 2-word query; change or add one word per pass
  • Minimum 4 attempts before concluding a concern is absent
  • Searches run via exec:codesearch exclusively; the spool-dispatch gate denies direct Glob/Grep/Find tool use during agent phases so reconnaissance always goes through the indexed surface

An existing concern found during search is consolidated, not duplicated. Adding a new abstraction over an existing one creates two competing implementations and is the primary source of duplication debt.

9.2 Minimal-Code Resolution Order

When a problem can be solved multiple ways, the agent follows a four-step resolution order and stops at the first that applies:

  1. Native first: does the language or runtime already provide this?
  2. Library second: does an existing dependency solve this pattern?
  3. Structure third: can the problem be encoded as data (map, dispatch table, pipeline) so the structure enforces correctness?
  4. Write last: only author new logic when the above three are exhausted.

Dispatch tables replacing switch chains, pipelines replacing loop-with-accumulator, and maps replacing if/else forests are treated as correctness properties, not style preferences. Structure eliminates the class of wrong states that arise when logic must be kept consistent manually.

10. Structural Parallelism and Wave Execution

Independent work items are never executed sequentially. The wave discipline: identify all pending items whose dependencies are satisfied, launch up to 3 parallel subagents in one message using Promise.allSettled, collect results, then evaluate the next wave. Each idea gets its own try/catch block. Wave execution completes the entire PRD covering family; the agent does not stop after any single wave to ask which item is next.

The only serialization constraint is the browser: exec:browser tasks do not parallelize (shared Chromium instance). All other execution is fair game for the wave model.

11. Permanent Observability

11.1 Mandatory Enumeration

On every planning iteration, the agent must enumerate every subsystem that lacks a permanent queryable inspection endpoint and add a PRD item to address each gap. The planning skill exits only when this enumeration pass produces zero gaps.

11.2 Server-Side

Every subsystem exposes a /debug/<subsystem> endpoint emitting structured {subsystem, severity, ts} JSONL events. console.log is not observability. Instrumentation lives at the dispatch boundary, one wrap per dispatcher covers all variants, never scattered per-handler.

11.3 Client-Side

window.__debug is the live structured registry for all in-page state. Every component registers its state, active requests, and WebSocket connections on mount and deregisters on unmount. The registry is addressable by key and requires no restart to query. No parallel observability surfaces are permitted; if an in-page behavior needs exercising, it is registered here and asserted via test.js.

11.4 Stack Observability

The gm stack writes structured JSONL to ~/.claude/gm-log/<YYYY-MM-DD>/<subsystem>.jsonl across all subsystems (browser, daemon-bootstrap, codeinsight, gm-plugkit bootstrap). The watcher additionally appends its own stdout/stderr (and the wasm cdylib’s println!/eprintln!) to .gm/exec-spool/.watcher.log, rotated at 10MB. Three live diagnostic files at the spool root surface runtime state: .status.json (watcher tick; stale mtime = dead watcher), .last-session-start.json (most recent bootstrap result), and .bootstrap-error.json (absent = healthy). The health spool verb returns a one-shot probe of every component, pin-match, watcher liveness, rs-learn status, recent errors. Diagnosing a behavior without first reading the log or dispatching health is forced closure equivalent to guessing.

12. The Memorize Sub-Agent

Every unknown-to-known transition triggers a background memorize dispatch in the same turn it resolves. The pattern is a spool verb dispatch:

write .gm/exec-spool/in/memorize-fire/<N>.txt
  body: "<fact>"
read  .gm/exec-spool/out/memorize-fire-<N>.json

Multiple facts resolved in one turn spawn multiple parallel dispatches in a single message. End-of-turn self-check: scan for un-memorized resolutions and dispatch immediately. Batching memorize calls is prohibited (facts memorized together are classified together, losing semantic separation). The memorize-fire verb is the only mechanism that edits AGENTS.md and CLAUDE.md; inline edits to those files by the agent are prohibited. The classifier rejects changelog-shaped facts from AGENTS.md ingestion (rs-learn still accepts them).

Recall and memorize together form the learning loop. Skipping either half breaks it.

13. Governance Layer

13.1 Seven Route Families

Every PRD item is tagged with one of seven route families that determine which quality constraints apply:

grounding:
Constrains the agent’s belief formation, what counts as evidence, what counts as knowledge, when to regress to planning. Governs all information-gathering operations.
reasoning:
Constrains inference chains. Requires each reasoning step to be witnessable, prohibits conclusions that outrun their premises, enforces the two-pass rule and surprise absorption prohibition.
state:
Governs mutable state transitions. Every mutation must be visible in the PRD; state-bearing operations require pre- and post-conditions. The PRD is the authoritative state record.
execution:
Constrains code execution: real services only, witnessed output, exec:<lang> dispatch, timeout budgets. No stubs, no mocks, no hardcoded responses.
observability:
Every subsystem must be inspectable without restart. Covers /debug endpoints, window.__debug, structured logging, the gm-log JSONL stack.
boundary:
Governs the agent’s interaction surface with external systems: git, CI, remote APIs, the user. The remote-execution boundary (§16) is the primary concern of this family.
representation:
Governs how information is encoded and communicated: skill prose must be implicit not explicit, memory must use the four semantic types, PRD items must use the mandated schema.

13.2 Four State Planes

Agent state is tracked across four orthogonal planes that must be consistent at every phase transition. Each plane has a concrete detection test:

epistemic:
What the agent knows, what it is uncertain about, what mutables remain unresolved. Detection test: count unresolved mutables (ε); PLAN exits only at ε = 0. Any named UNKNOWN that has not been witnessed is an open mutable.
operational:
What the agent has done, what artifacts exist on disk, what code has been executed. Detection test: git status (uncommitted changes = inconsistent), acceptance criteria assertions (failing = inconsistent), exec:<lang> return value witnessed (not just theorized).
relational:
What the agent has communicated to the user, what commitments are outstanding. Detection test: any open “I will” or “next I’ll” statement in the agent’s prior response that has no corresponding completed action in the operational plane is a relational inconsistency.
temporal:
What happened in earlier sessions, what the PRD preserves across context resets. Detection test: .gm/prd.yml exists and has pending items = temporal inconsistency with COMPLETE; exec:recall hit on a prior design decision that differs from the current plan = weak temporal inconsistency requiring re-witness.

A phase transition is legitimate only when all four planes are consistent with the transition’s exit condition. An agent that advances EXECUTE→EMIT while the epistemic plane still contains unresolved mutables is in an inconsistent state regardless of what the operational plane shows.

13.3 Five Refused Collapses

Five patterns are categorically refused because they collapse the discipline without appearing to:

  1. Narrating as executing: describing what code would do rather than running it
  2. Assumption as resolution: treating a plausible value as witnessed
  3. Partial closure: declaring a PRD item done while its acceptance criteria are unmet
  4. Deferred observability: shipping code without an inspection endpoint and noting it as a future task
  5. Asking-as-progress: routing through the user a question the agent could answer itself via tools

14. Quality Metrics

Four metrics govern quality gate decisions at phase transitions:

ΔS (state delta):
The number of state-plane inconsistencies introduced by a proposed change. A change with ΔS > 0 must be rejected or the inconsistency must be resolved before advancing.
λ (load score):
Consequence weight of a PRD item (0.0–1.0). Items with λ > 0.75 require witnessed verification before EMIT. Verification budget = λ × (1 − tier_confidence), where tier_confidence is the fraction of acceptance criteria for this item that have already been witnessed (0.0 = none witnessed, 1.0 = all witnessed). Higher prior witnessing shrinks the remaining verification budget proportionally.
ε (epistemic gap):
Count of unresolved mutables in the epistemic plane. PLAN exits only at ε = 0 on the final pass.
Coverage (≥0.70):
The proportion of load-bearing behaviors witnessed by test.js. The test file enforces a 200-line budget; coverage ≥0.70 at that budget is the acceptance criterion. Behaviors not witnessable within the budget are not load-bearing by definition.

15. Failure Taxonomy

Sixteen failure modes are catalogued across four categories. Every PLAN pass cross-references this taxonomy against each PRD item:

Grounding Failures

  1. Assumption propagation: unverified hypothesis treated as ground truth
  2. Documentation trust: stale docs used as resolution evidence
  3. Static analysis substitution: code reading used instead of execution
  4. Memory staleness: outdated memory acted on without verification

Execution Failures

  1. Stub shipping: placeholder code presented as real implementation
  2. Mock leakage: test fixtures surviving into production paths
  3. Timeout blindness: execution result truncated without detection
  4. Orphan accumulation: background processes not cleaned up on restart

State Failures

  1. Partial closure: PRD item marked complete with unmet acceptance criteria
  2. Epistemic advance: phase transition attempted with unresolved mutables
  3. Cross-session amnesia: PRD items from prior sessions not discovered and executed
  4. Dirty exit: session closed with uncommitted or unpushed changes

Boundary Failures

  1. CI blindness: COMPLETE declared before CI result is known
  2. Remote-execution misread: git push treated as local execution, CI result not watched
  3. Forced pause: exec:pause issued for a question the web could have answered
  4. Asking-as-progress: user asked a question answerable via tools

16. Remote Execution Boundary

16.1 git push as Remote Execution

A git push is not a local operation; it is a remote-execution event that dispatches a build, test, and deploy pipeline on external infrastructure. The CI result is the canonical verdict on whether the change is correct. Declaring COMPLETE before that verdict is premature closure.

16.2 CI Watch Protocol

After every push, the agent dispatches gh run watch through the spool and polls until the run completes, then inspects the result. Green CI is a precondition for the VERIFY→COMPLETE transition; the stop-checkpoint gate refuses to allow session close while the most recent push has no recorded green run. The agent cannot advance past VERIFY while CI is pending or red.

16.3 Parse-Time CI Failure Shapes

Five CI failure shapes are catalogued for rapid triage:

  1. Import error: module not found, check package.json or Cargo.toml, not the source file
  2. Type error: schema mismatch, regression to PLAN to re-witness the interface
  3. Test failure: assertion mismatch, root cause first, never silence the assertion
  4. Lint failure: style constraint violation, fix in-band, never // eslint-disable
  5. Build timeout: infrastructure issue, re-trigger once; if repeated, open a PRD item with blockedBy: external

17. Maximal Cover

Hard Rule: Maximal Cover [PLAN]

When scope exceeds reach, construct a maximal cover, not a refusal, not a single slice with the rest deferred. Distributed refusal is the same failure dressed as triage. The default shape is the widest, deepest, most destructive reading the request admits from this session; the narrower shape is the exception and carries the burden of proof.

Required move: construct a covering family of every bounded subset of the request witnessable from this session. Write the family into the PRD with the dependency graph explicit. Execute every member. Build cost, CI duration, cross-repo reach, and breaking-change weight are inputs to executing the cover, not arguments for shrinking it.

Residuals the agent judges within the spirit of the original ask AND completable from this session are self-authorized. Expand the PRD with them and execute, do not stop to ask. Only residuals genuinely outside the original ask, or genuinely unreachable from this session, are name-and-stop.

When expanding under self-authorization, the agent declares its judgment in-response (“treating X as in-scope because Y”) so the user can correct mid-chain. Silent expansion without the declaration is the failure mode this rule guards against.

Enforcement: committed work + named out-of-spirit residuals must equal the witnessable closure of the request. Gap = cover not maximal → re-enter planning.

The Maximal Cover obligation emerges from recognizing that scope-exceeds-reach is not a stopping condition; it is a planning condition. The covering family is the plan. Stopping at one slice is the same as refusing, both leave witnessable work unexecuted, both are forced closure. Recent controlled experiments quantify the structural bias the rule corrects: LLMs systematically omit required sections, undershoot length requirements, and substitute minimum-viable outputs for the full ask, laziness is measurable, not anecdotal.17

18. Bounded Autonomy

Hard Rule: User Request = Authorization [PLAN · EXECUTE]

The user’s first message asking for X is the approval to do X. The PRD records the authorization; it is not a second permission gate. Re-asking “want me to do X?”, “should I proceed?”, “which shape of X?” after the user said “do X” is forced closure dressed as deference, the user already authorized the work and waits while the agent stalls.

Surfacing tradeoffs is allowed inside the same chain (one sentence: “going with shape A because Y”) so the user can interrupt, never as a stop-and-wait. Multi-repo scope, build cost, CI duration, and binary-size impact are never grounds to re-confirm.

A written PRD is the user’s authorization. Once it exists, EXECUTE owns the work to COMPLETE. Every doubt arising during execution is resolved by witnessed probe, by recall, or by re-reading the PRD, never by asking the user.

Asking is last-resort: destructive-irreversible AND not covered by the PRD, OR user intent irrecoverable from PRD/memory/code/web. Channel: exec:pause (renames prd.yml → prd.paused.yml; question in header). In-conversation asking is last-resort beneath last-resort. The disposition: a question forming in-response is a signal to expand scope and dispatch a WebSearch/WebFetch pack first; the question survives only if both close empty. Most agent-side questions are scope-narrowing dressed as deference, and the answer the user would give matches what one web call would produce.

exec:pause interaction with the stop checkpoint: when the agent issues exec:pause, the PRD is renamed to prd.paused.yml. The stop gate’s PRD-completeness check reads .gm/prd.yml and finds no live PRD, so the session may close normally. On the next session-start bootstrap, skill-bootstrap.js detects prd.paused.yml without a live prd.yml and renames it back, resuming where the agent paused. There is no hook-timeout race because the rename is a synchronous file operation completed before stop is attempted.

The asking-cost principle: asking always costs at least one round trip. The agent that asks a question answerable via tools has transferred work to the user that belongs to the agent. The accumulated asking tax across a session, five questions each costing the user 30 seconds of context reconstruction, is a form of quality failure as real as a broken test.

19. The Learning Loop

The learning loop consists of two operations that must both fire to function:

19.1 Recall (Read)

Before resolving any unknown via fresh execution, check past sessions. The recall operation:

exec:recall
<2-6 word query>

Triggers: unknown feels familiar | sub-task on a known project | about to ask user something likely already discussed | about to design where a prior decision likely exists. Hits become weak_prior; still witness before adopting. Cost: ~200 tokens, ~5ms.

19.2 Memorize (Write)

Every unknown→known transition triggers a background memorize-fire dispatch the same turn it resolves. Multiple facts → multiple parallel spool writes in one message. Skipping memorize breaks the loop: subsequent sessions will rediscover what this session solved, paying the full discovery cost again.

19.3 rs-learn Backend

Both operations run against the rs-learn KV+vec store at .gm/rs-learn.db. The database is committed to git so memory and index state share across machines, sessions, and CI. The orchestrator’s auto-recall verb calls rs-learn directly from inside the wasm module and emits a “## Recall for this prompt” block into the next instruction body before the agent sees the user turn.

20. Execution Budget Model

Every exec_js dispatch operates within a two-layer budget:

  1. Dispatch layer: timeoutMs field mandatory in the request body; the wasm host hard-rejects missing or zero values
  2. Host layer: child_process.spawnSync with a timeout argument; on timeout, partial streamed output is preserved and timedOut: true lands on the response sidecar

On timeout, the agent treats the [exec timed out after Nms; partial output above] marker as authoritative truncation and re-issues with a higher budget. Retrying the same call after a timeout without increasing the budget is a fixed-cost infinite loop.

Because exec_js is synchronous under wasm, long-running subprocesses block the spool watcher for the duration of the call. Orphaned background work is not supported. Callers that need long-lived workloads must pass a real per-call timeout and accept that the watcher is unavailable to other dispatchers while the call runs.

21. ORIENT: Pre-Planning Recall Pack

Hard Rule: ORIENT [PLAN]

Open every planning session with a parallel pack of 3–5 exec:recall calls and 3–5 exec:codesearch calls against the request’s nouns, all in one message. Hits land as weak_prior; misses confirm the unknown is fresh.

The pack runs in one message, never serially. The agent that skips ORIENT pays the same cost in fresh probes a turn later, plus the price of disagreeing with its own prior witness if the earlier session established a relevant fact.

ORIENT is free relative to the duplicated discovery and disagree-with-prior-witness risk it prevents. Its cost equals the cost of skipping it, because the skipping agent will eventually run the same searches; it just does so later, after having already named unknowns that may be stale.

The ORIENT discipline formalizes a pre-planning reconnaissance phase. Before the agent names any unknown as absent, it must have searched for it. Before the agent designs an abstraction, it must have checked whether one exists. The parallel pack ensures this reconnaissance runs at maximum speed and does not serialize the planning phase unnecessarily.

22. Fix on Sight

Hard Rule: Fix on Sight [EXECUTE · EMIT · VERIFY]

Every issue surfaced during work is fixed in-band, this turn, at root cause. Defer-markers, swallowed errors, suppressed output, skipped tests, and “address it next session” are variants of the same failure: a known-bad signal carried past the moment of detection.

Surface → diagnose → fix at root cause → re-witness → continue. New unknown surfaced by the fix → regress to planning. Out-of-scope-irreversible fix → residual into PRD before moving on. Narration is not a substitute for a PRD item.

22.1 Auto-PRD on Incidental Error

Any reasonably fixable error surfacing mid-task, pre-existing build break, lockfile drift, broken dep, lint failure on neighboring code, type error in adjacent module, stale generated file, dead import, is added to .gm/prd.yml as a new item the same turn it surfaces, and executed before COMPLETE. The user does not have to ask. Closing a turn with a known-bad signal still emitting is forced closure regardless of whether the signal was “from a different task.”

Genuinely out-of-reach errors (require user credentials, depend on external services that are down, demand product decisions) are named and stopped with the residual recorded in the PRD with blockedBy: external, never silently dropped, never narrated past.

22.2 Auto-PRD on Re-Architecting Opportunities

The same rule applies to obvious refactor wins surfaced while doing the user’s task: a hand-rolled implementation that a well-known library does cleaner; a multi-file ad-hoc system one import would replace; duplicated logic crying out for one helper; a boilerplate pattern repeated in N places where one upstream tool covers it. These are not separate sessions, they are PRD items, added the same turn they surface, executed before COMPLETE.

The bar is obvious + reachable from this session, not any conceivable refactor. Speculative or aesthetic changes stay out. Library-replacement is a specific case: before keeping a local reimplementation of something a published package already provides, add a PRD item to swap to the import. Local code competes with upstream maintenance and loses.

23. Browser Witness

Hard Rule: Browser Witness [EXECUTE · EMIT · VERIFY]

Editing code that runs in a browser requires a live exec:browser witness in the same turn as the edit. The witness does not defer to a later phase; later phases re-witness on top, they do not replace this one.

Protocol: (1) boot the real surface, server up, page reachable, HTTP 200 witnessed; (2) exec:browser → navigate → poll for the global the change affects; (3) page.evaluate asserting the specific invariant the change establishes, capturing the witnessed values; (4) variance → fix at root cause, re-witness (Fix on Sight). Never advance on unwitnessed client behavior.

Pure-prose edits to static documents with no JS/canvas/DOM behavior change are exempt; the exemption is tagged explicitly with the reason so the skip is auditable. Silent skip on actual behavior change is forced closure.

This rule fires in EXECUTE (witness on edit), EMIT (post-emit verify), and VERIFY (final gate).

The browser witness rule closes the gap between “the test passes” and “the feature works.” A test that passes against a mock DOM cannot witness whether the live page renders the change correctly. The only authoritative evidence is the live page’s own evaluation of the invariant.

24. Nothing Fake

Hard Rule: Nothing Fake [EXECUTE · EMIT · VERIFY]

What ships runs against real services, real data, real binaries. Stubs, mocks, placeholder returns, fixture-only paths, “TODO: implement”, hardcoded sample responses, and demo-mode fallbacks are forbidden in source the user will run. They produce green checks that survive into production and lie about what works.

Scaffolding and shims are permitted when they call through to real behavior: an empty file laid down before its body, a thin adapter wrapping an upstream API, a build target that compiles but is wired to nothing yet and is the only callsite of itself. The test is whether the artifact, executed, would do the thing it claims. If it would not, it is a stub.

Before writing a shim or adapter, the agent asks whether an existing library or tool already provides the same surface. Maintaining a local reimplementation of something an upstream package solves is its own failure mode, the shim drifts, ages, and accumulates the bugs the upstream already fixed.

Stub detection is behavioral, not by keyword: code paths that always succeed, always return the same value regardless of input, or short-circuit a real call to satisfy a type signature are stubs. Comments asserting realness do not make code real.

The Nothing Fake rule is the execution-layer complement to the witnessed-execution principle: witnessed execution requires real code to run; Nothing Fake prohibits shipping code that passes witnessing through deception. Together they close the loop: you cannot fabricate a witness, and you cannot ship fabricated code past a witness.

25. Web-Search Before Pause

Hard Rule: Web-Search Before Pause [PLAN · EXECUTE · EMIT · VERIFY]

Before exec:pause or any in-conversation question whose answer plausibly exists on the public web, missing artifact, prebuilt binary, library status, build recipe, version compatibility, upstream issue, “does X exist for Y”, fire WebSearch and at least one targeted WebFetch first. Pause or ask only after the web pass returns empty, or returns candidates the agent has witnessed and rejected.

Pausing on a question the web could have answered is forced closure dressed as humility. Fix on sight: re-enter planning, web-search, and resume.

Genuine user-only questions: private credentials, preference among already-surfaced viable options, destructive-irreversible authorization.

The web-search-before-pause rule addresses the pattern where an agent stalls on an unknown that is publicly documented. The stall taxes the user with a round trip that the agent’s tools could have resolved. The rule is a specific instance of the broader autonomy principle: any question whose answer the agent could obtain itself is a question the agent owes itself, not the user.

26. Single Integration Test Policy

Hard Rule: Single Integration Test [VERIFY · COMPLETE]

One test.js at project root. 200-line hard cap. No fixtures, mocks, or scattered test files under any naming convention. Plain assertions, real data, real system. gm-complete runs it. Failure = regression to EXECUTE.

Any second test runner, under any name, in any directory, is a smuggled parallel surface that fights the observability discipline. If a behavior needs exercising in-page, register it in window.__debug and assert via test.js.

26.1 Budget, Not Target

test.js is a 200-line budget, not a target. Every line witnesses a load-bearing behavior; redundant assertions are dead weight. Subsystems get one group each, combined groups (profiles+observability+auth+context) are the norm, not the exception. The win from combining comes from collapsing per-subsystem groups into multi-subsystem ones: the test surface grows linearly with subsystem count, not multiplicatively.

26.2 Use Overlap to Exclude

When subsystem A’s test exercises B as a side effect, B does not need its own group. Examples from production: the agent-machine tool-loop test exercises bash dispatch; the dashboard test asserts the API surface and tool registration; the gateway test exercising one platform with a stub-shape loop covers all 18 adapters. Overlap is a signal to drop the redundant group, not to add another assertion.

26.3 Hygiene at Edit Time

Every change to test.js prefers compaction over expansion. If the file exceeds 200 lines, the discipline is to merge groups and drop redundancy until it fits, not to split the file or lift the cap. If the budget is genuinely insufficient, the question is whether the assertion is load-bearing, not whether the cap should move.

26.4 Group Naming Convention

Name combined groups by joining their subsystems with +: home+config+skin, mcp+swe+distributions+account+credpool, env+pi+cli+tui+setup+website. A group with 4–6 components is healthy; a group with 1 component should be questioned.

27. Sync-Before-Emit

rs-codeinsight and rs-search outputs must come from a freshly-completed index. No cache may serve a result without a digest match against the live filesystem (mtime sum + git HEAD + dirty-tree marker). Default invocation always runs fresh. --read-cache is permitted only when the .codeinsight.digest matches exactly; on mismatch, the cache auto-refreshes before emitting.

rs-search runs scan + embed + sweep before printing the first result and emits an [index fully synced: …] line so freshness is visible in the output. Emitting from an unverified or partial index is forced closure equivalent to bluffing strength, the agent reads stale output as ground truth and acts on a state that no longer exists.

28. Auto-Recall on Prompt-Submit

The orchestrator’s auto-recall verb fires at every turn entry. It derives a 2–6 word recall query from the user prompt, calls the rs-learn Searcher directly via the shared tokio Runtime, and injects “## Recall for this prompt” into the next instruction response before the agent acts on the user turn. Auto-search (codeinsight) runs at session-start bootstrap; auto-recall runs at every prompt. The mechanism is the spool, not a host-side prompt-submit callback, any harness that loads gm-skill gets the behavior identically.

Together these ensure every turn begins with prior memory loaded. The agent that skips ORIENT still receives the auto-recall injection; the ORIENT discipline adds 3–5 targeted queries on top of the single auto-recall hit.

29. Implicit Skill Prose

Skill files and system prompts elicit behavior, they do not describe it. Write terse imperative principles whose phrasing triggers the model’s already-learned dispositions, not numbered procedures that spell out what to do.

Forbidden in skill prose: numbered step lists like “1. agent runs N parallel calls 2. then writes 3. then…”; citations to site pages or paper numbers; references to the documentation as an input (“as described in docs/skills.html”); multi-step recipes. The skill is a prompt, not a manual. If it reads like a manual, the behavior gets imitated as a script and breaks at the first edge case.

The papers and site are outputs of the discipline, not inputs to it. Cross-cutting rules that need a citation belong in AGENTS.md, not in skills. CLAUDE.md in the project root is a pure @AGENTS.md pointer, nothing else is written there directly.

30. The Spool Dispatch Surface

Hard Rule: Every Dispatch Through The Spool [PLAN · EXECUTE · EMIT · VERIFY]

Code execution and every utility verb both write a file to .gm/exec-spool/in/<verb>/<N>.<ext>. Language verbs: nodejs, python, bash, typescript, deno, go, rust, c, cpp, java. Host verbs: fs_read, fs_write, fs_stat, fs_readdir, kv_get, kv_put, kv_query, fetch, exec_js, env_get, recall, codesearch, memorize, browser, health. Orchestrator verbs: instruction, transition, phase-status, prd-add/prd-resolve/prd-list, mutable-add/mutable-resolve/mutable-list, memorize-fire, residual-scan, auto-recall. Output lands at out/<verb>-<N>.json for nested verbs and out/<N>.json for root verbs.

Only git and gh run directly via the Bash tool. Inline tool args, JSON-form dispatch, and any attempt to bypass the spool (e.g. Bash(node script.js), Bash(python -c …)) are denied at the Bash gate.

The spool exists because tool args are ephemeral and inline. They live for the duration of one call, do not survive compaction, are not replayable, and cannot be inspected without re-running the agent. A file-based surface inverts every one of these properties: the request lives on disk before the watcher reads it, the watcher is detached from the agent process, and the output triplet (.out, .err, .json) is auditable after the fact by any tool that can read a directory.

The three-file output shape maps to the agent’s three needs. .out is the live stdout stream, written line-by-line as the task runs. .err is the separate stderr channel, mixing them collapses the distinction between “result” and “diagnostic,” which leads to ambiguous downstream parsing. .json is the metadata sidecar written once at completion: exitCode, durationMs, timedOut, startedAt, endedAt. The metadata file is what tells the agent whether the task ended cleanly, was killed for budget, or is still running; the streams alone cannot answer that.

The boundary is enforced at the Bash gate. An agent that tries to run node script.js or any non-spool execution directly through Bash receives a denial that throws rather than mutates, the error names the spool path the agent should have written to. This is the only execution surface the discipline recognizes; everything else is a guess about whether the call happened. The constrained-tool-surface bet has independent support: SWE-agent argues that a deliberately designed Agent-Computer Interface is the primary lever for reliability, and contemporary work on fault-tolerant agent sandboxing converges on policy-based interception plus transactional filesystem snapshots as the safe-execution baseline.3,12

from the gm field guide One folder. One contract. Watcher reads, agent writes, output lands in three files. The agent who tries to skip the folder discovers very quickly that the folder is not optional.

31. The Mutables File as Co-Equal Hook Gate

Hard Rule: Mutables File Co-Equal With PRD [PLAN · EXECUTE · EMIT]

.gm/mutables.yml lives alongside .gm/prd.yml. Every unknown surfaced during PLAN lands as a row with status: unknown. Write, Edit, NotebookEdit, git commit, git push, and the stop checkpoint are all hard-blocked by the dispatcher while any row has status: unknown. Rows flip to witnessed only when witness_evidence is filled with concrete proof (file:line, codesearch hit, exec output snippet).

The named-mutable principle in §3 lived inside the agent’s context window. A mutable that is “tracked” only in the agent’s reasoning evaporates the moment the context compacts, the agent is restarted, or a subagent is spawned. The mutables file is the externally-enforced version of the same idea: a row on disk that the spool-dispatch gate reads independently of the agent’s memory.

The split between mutables.yml and prd.yml mirrors a deeper split. The PRD is what-to-do: ordered work items, each with acceptance criteria. The mutables file is what-must-be-true: assumptions whose violation would invalidate the work regardless of whether the items themselves are completed. A PRD item can be done while a referenced mutable remains unknown, the gate is on the mutable, not the item. This is why the dispatcher reads both files separately: completion is necessary but not sufficient, and verified-state is the second leg.

The witness-evidence requirement is what prevents the agent from flipping a row to witnessed by narrative alone. A status change with no evidence string is rejected on read; the row stays unknown and the EMIT gate stays closed. This closes the gap §3 left open, a discipline the agent can talk past becomes a discipline a file refuses to talk past.

32. Discipline Isolation via @sigil

A discipline is a per-project knowledge store at <project>/.gm/disciplines/<name>/. Each discipline owns its own rs-learn database and code-search index. Disciplines exist because cross-project memory bleed produces a specific failure mode: the agent recalls a fact from a prior project as a weak_prior for the current one, and that prior either contradicts the current code or refers to a function name that never existed here. The hallucinated prior is worse than no prior, it earns the right to be tested when it should not have earned the right to exist.

The @<name> sigil in a request is the agent’s authorization to read from one discipline. Recall, codesearch, and memorize all parse the sigil from the originating request and scope their stores accordingly. Without a sigil, the read-write behavior is asymmetric on purpose: reads fan across default plus every enabled discipline (one per line in <project>/.gm/disciplines/enabled.txt) and merge-rank results with [discipline:<name>] prefixes; writes go to default only.

The asymmetry encodes a safety principle. Reading broadly is recoverable, if a prior is wrong, the witness step catches it. Writing broadly is not, a fact written to the wrong discipline pollutes future reads in projects it has no business affecting. Read-broad, write-narrow is the same principle that motivates least-privilege in filesystem permissions, adapted to a memory store.

33. Synchronous-Under-Wasm Execution Contract

Hard Rule: Every Exec Carries A Real Timeout [EXECUTE · EMIT]

exec_js is synchronous under the wasm host: the orchestrator runs the user subprocess via child_process.spawnSync and blocks the watcher for the duration. Every dispatch carries a positive timeoutMs; missing or zero is a hard error. On timeout, partial streamed output is preserved and timedOut: true lands on the response sidecar.

The synchronous-under-wasm contract is a consequence of wasm itself. There is no thread the orchestrator can detach the subprocess into, no way to keep a long-running process associated with the wasm instance after the dispatch returns, and no port for an external observer to attach to. The host runs the subprocess, waits up to timeoutMs, and returns. Long subprocesses block the watcher for the duration; orphaned background work is not supported under this surface.

This is a deliberate trade. The earlier generation of this project ran rs-exec as a native daemon with a port file, a task store, and a grammar of resumption verbs (tail, watch, wait, sleep, close). It was more flexible and also a fountain of operational pain: dead PIDs, port-leak races on Windows, idle-cleanup tuning, kill-tree edge cases on bash pipelines that escaped the parent group. The wasm-only surface gives up the long-task affordance and gets back operational simplicity: no daemon to supervise, no port to leak, no orphan-process accounting. The wasm-for-agents pattern has converged in recent industry and academic work as well: NVIDIA Developer Blog argues for WebAssembly sandboxing of LLM-generated code, and MCP-SandboxScan formalizes a WASI sandbox for untrusted MCP tool execution.20,19

Browser sessions are the one exception. exec:browser opens a managed Chrome session that the wasm host holds open between dispatches (the host-side state lives outside the wasm instance, in the Node process that hosts it). Browser sessions persist across turn-stops and are reaped only on real-exit reasons (clear, logout, prompt_input_exit). The running_task_ids field on every dispatch response surfaces outstanding browser sessions so the agent sees them on every turn.

34. Hook Denials Throw, Never Mutate

Hard Rule: Throw on Denial [PLAN · EXECUTE · EMIT · VERIFY]

A hook that blocks a tool call throws an error with an imperative instruction string. It does not rewrite the call’s arguments into a self-failing form, not a shell command that exits 1, not a Node one-liner that writes to stderr and process.exit. The thrown error is the entire denial surface.

The distinction is about how the host model reads the failure. A thrown error is read as policy: the model registers that the surface refused this kind of call and adjusts its behavior on the next attempt, for example, switching from Bash(node script.js) to exec:nodejs dispatched through the spool. An args-mutation that rewrites the Bash call into node -e "process.stderr.write('blocked'); process.exit(1)" produces the same exit code, but the model reads it as “the Bash tool is broken” and retries with simpler commands. The wrong adaptation reinforces the wrong mental model and turns a one-shot denial into a retry loop.

The discipline generalizes. Any future hook whose purpose is to change the agent’s behavior in response to a denied call must throw, not mutate. The throw form is for “use a different tool;” the mutate form is for “run this corrected version of your own call,” which is a much rarer need and almost never what a guardrail wants.

35. CI Is the Build

Hard Rule: Push, Watch, Fix On Red [EXECUTE · EMIT · VERIFY]

For the Rust crates in this org (rs-plugkit, rs-exec, rs-learn, rs-search, rs-codeinsight) and the gm publish chain, git push triggers the build pipeline. cargo build and cargo test are not run locally. Toolchain mismatches, rustc version skew, and missing system dependencies never block a push, push, watch CI via gh run watch dispatched through the spool, fix on red.

The principle is that the witness for a Rust crate is the green CI run on the pushed HEAD, not the green local build. Every crate compiles to wasm32-wasip1 as a cdylib; the wasm artifact is what the user actually loads. A local cargo build proves nothing about whether the wasm import surface matches the host’s expectations or whether the published npm tarball has the right contents.

The cascade is the witnessing infrastructure. A push to any sibling crate fires its cascade.yml, which uses gh workflow run with PUBLISHER_TOKEN to trigger release.yml in rs-plugkit. rs-plugkit rebuilds its own wasm, publishes it to plugkit-bin Releases and to npm as plugkit-wasm, and bumps gm-starter/gm.json::plugkitVersion plus bin/plugkit.wasm.sha256 in this repo. The bump commit triggers publish.yml, which runs node cli.js gm-starter ./build and publishes a fresh gm-skill npm tag. The witness lands at the leaf of that tree, not the root.

This pairs with §23 Browser Witness: same witnessing principle, different execution boundary. The browser witness uses exec:browser to assert a behavior in a real Chromium; the CI witness uses GitHub Actions to assert a wasm builds and the npm tarball publishes cleanly. In both cases, the witness lives outside the agent’s local environment because the agent’s local environment cannot answer the question.

36. The Residual-Scan Stop Gate

Hard Rule: Stop-Hook Residual Challenge [COMPLETE]

When .gm/prd.yml is empty or missing AND no open browser sessions AND no running tasks, the stop-checkpoint gate fires one residual-scan challenge before allowing stop. The agent either re-enters planning, appends to the PRD, and executes, or explicitly states “residual scan: none reachable in-spirit.” The marker file .gm/residual-check-fired makes the challenge one-shot per stopping window; the marker clears at the next prompt-submit so the next stopping window starts fresh.

The gate exists because empty-PRD is a necessary but not sufficient condition for done. The PRD records what the agent decided to do; the gate asks what the agent should have decided to do but did not. In production, the gap between those two sets is where forced closure hides, the agent finishes its enumerated items and stops, leaving obvious reachable in-spirit residuals (a pre-existing build break surfaced this turn, a neighboring lint failure, an obvious refactor win, an observability gap, doc drift, follow-on work the user clearly implied) for the user to discover.

The challenge is one-shot per stopping window so it does not become noise. A second stop attempt in the same window allows: the agent has already answered the challenge, and a hook that refused the second stop would loop without termination. The marker clears at the next prompt-submit because the next user turn defines a new stopping window with new in-spirit boundaries.

The rule pairs with §17 Maximal Cover and §22 Fix on Sight. Maximal Cover defines the in-spirit boundary at PLAN time; Fix on Sight folds discovered issues into the PRD during EXECUTE; the residual-scan gate is the COMPLETE-phase backstop that catches what both earlier disciplines missed.

37. Meaning Through fetch, Mechanics Through Code

Hard Rule: Semantic Work Routes Through fetch [PLAN · EXECUTE · EMIT]

Any task whose correctness depends on understanding, summarize, classify, extract intent, rewrite, translate, semantic dedup, score, label, decide-if-two-texts-mean-the-same, routes through a haiku-class model call offloaded to the host agent via the LLM callback protocol. One call per item, N items in N parallel spool dispatches in one message.

The split is between mechanics and meaning. Mechanics are operations whose correctness can be decided by a deterministic procedure: parse this JSON, sort these strings, hash this file, count occurrences of a regex. Meaning is operations whose correctness depends on the model’s understanding of language: does this commit message describe a bug fix or a feature, does this docstring contradict its function’s behavior, are these two error messages saying the same thing in different words.

Code does mechanics well and meaning badly. A keyword-list or regex-on-meaning-phrases loop deciding semantic questions ships a green check that lies, in the same way §24 Nothing Fake ships a green check that lies. The deterministic procedure looks right because it terminates and returns a value; the value is unrelated to the question the user actually asked.

Parallelism is the second half of the discipline. N items go to N parallel dispatches in one message; batching them into a single call dilutes the signal because the model averages over the batch instead of attending to each item. The cost difference is negligible, haiku calls are cheap, and the quality difference is large. This pairs with §12 memorize-fire dispatch, which uses the same fan-out shape for the same reason: one fact per call, parallel by default.

38. The Continuation Invariant: A Toolless Turn Is A Stop

Hard Rule: While The Chain Is In-Flight, Every Turn Ends In A Tool Call [PLAN · EXECUTE · EMIT · VERIFY]

Every programming agent reads only tool calls and their outputs; an assistant message that ends in prose with no tool call is the turn ending, and the harness halts there. So while the chain is in-flight (phase ≠ COMPLETE or prd_pending > 0) the agent never ends a turn in prose and never writes a summary or recap, summary is a stop. The only event that authorizes a prose-only turn is plugkit returning phase=COMPLETE and prd_pending=0. Before any apparent stop, the agent dispatches phase-status and rechecks; a non-terminal phase means the urge to stop was drift, and the recovery is to dispatch instruction and continue.

This is the mechanical root of the most common long-horizon failure: an agent completes one small piece of a multi-part task, writes a “here is what I did” paragraph, emits no tool call, and the harness reads that paragraph as the end of the work. The agent did not decide to stop, the absence of a tool call is the stop, independent of intent. No amount of “remember to continue” in the prompt survives the moment the model emits a closing sentence, because the closing sentence and the stop are the same event.

The invariant inverts the default. Instead of “continue unless done,” the rule is “the answer to am I done? is another tool call” until the gate says otherwise. Doneness is not a feeling the agent reports; it is a pronouncement the agent reads back from plugkit’s COMPLETE gate. The agent that types a summary has substituted its own judgment for the gate’s, and on a long-horizon task that judgment is wrong often enough that the substitution is the dominant source of premature halt.

Crucially this depends on nothing but the verb spool. It is not a hook, not a stop-event interceptor, not a platform-specific continuation feature, any of which would bind the discipline to one agent host and break the moment the same skill runs under a different harness. The continuation invariant is prose served through the instruction verb and restated in the skill body; it elicits the same behavior on every model that can read and call tools. Predictability across heterogeneous LLMs is the goal, and a spool-only rule is the only mechanism that achieves it.

39. Spool Dispatch Replaces Platform Hooks

Hard Rule: The Gate Is The Spool, Not A Host Hook [PLAN · EXECUTE]

Orchestration state is tracked via marker files in .gm/ and gated at the dispatch layer, not through any agent-host hook event. No skill prose, gate denial, or instruction names a host-specific invocation form (a plugin-namespaced skill, a host-specific subagent type). The gate-satisfying action is always a portable spool dispatch: write .gm/exec-spool/in/instruction/<N>.txt and read the response.

An earlier generation gated the chain with the host’s PreToolUse hook, emitting that host’s permission-decision JSON and naming that host’s skill- and subagent-invocation forms as the way to satisfy the gate. That coupling is invisible until the same skill is installed under a different harness, where the hook never fires and the named invocation forms do not exist, so the gate either never engages or instructs the agent to call something that is not there. Subagent-form orchestration compounds the failure: an isolated subagent returns a prose summary to its parent, and the parent, reading prose with no continuing tool call, stops (§38).

The replacement is marker files plus spool dispatch. checkDispatchGates reads .gm/prd.yml, .gm/mutables.yml, and the needs-gm / gm-fired markers and returns an allow/deny with a reason string; the reason names the next portable verb to dispatch, never a host-specific tool. Every behavior the old hooks enforced, gm-before-other-tools, mutables-block-writes, residual-scan-before-stop, survives as a spool gate that behaves identically on every agent. The legacy wasm_hooks exports remain only as dead ABI with no caller; the agent-facing surface names nothing tool-specific.

40. The Memory Architecture: File-Vec Recall and the rs-learn Algorithm Layer

Memory splits into two layers with a deliberate boundary. The live layer is the file-vec store: memorize-fire embeds a fact through the in-wasm bge-small model and writes two files per memory, the text under .gm/disciplines/<ns>/<key>.json and its 384-dim embedding under .gm/disciplines/<ns>-vec/<key>.json. recall and the turn-entry auto_recall read those directories fresh, scoring each candidate as cosine × recency where recency decays on a 30-day half-life with a 0.4 floor, then dedup by Jaccard overlap. This layer is the one the agent actually drives every turn; it travels with the project in git, so memory is project-resident, not platform-resident.

Pruning is first-class in the live layer because a wrong recall hit is worse than a miss: a stale fact surfaces on every future query that matches it, poisoning the orient pack. The memorize-prune verb deletes a memory’s text and embedding together (no orphan vector survives) either by explicit key or by a review-only query that returns candidates for the agent to judge, never a blind similarity-delete, since an automatic similarity prune is itself a bad-memory generator. Pruning precision is valued over retention.

The algorithm layer is rs-learn, a separate wasm cdylib of five continual-learning cores: an instant rank-2 Hebbian MicroLoRA adapter with norm-bound and EWC correction (in the spirit of sona); an EWC++ deep core with online Fisher EMA and z-score boundary detection; a bi-temporal knowledge graph that invalidates rather than deletes contradicted edges and answers point-in-time queries (in the spirit of graphiti); an 8-head graph-attention module with relation and recency edge features; and a FastGRNN sparse-plus-low-rank router with epsilon-greedy exploration over model, context-bucket, temperature, top_p, and confidence heads. rs-learn is purely the algorithm layer, it owns no IO, importing host_kv_*, host_vec_search, host_log, and host_now_ms from whatever host satisfies the contract, and exposing every core through one JSON dispatch surface.

The motivating design choice is that the host owns IO, scheduling, embedding, and agent transport while rs-learn owns only the math, so the same algorithm crate is native-Rust testable in isolation and reusable by any host that satisfies the six imports. The current live system runs the file-vec layer in full, and the rs-learn algorithm cores are now reachable from the agent-facing spool: plugkit carries rs-learn as a crate dependency and exposes a learn verb that routes a {verb, …} body through dispatch_json into the cores over a PlugkitKv backend (the three-method KvBackend trait satisfied by the host’s kv_get/kv_put/kv_query imports), so the temporal graph and the LoRA/router optimization are dispatchable through the same spool surface as every other verb rather than living as a tested-but-isolated crate. The boundary itself, math layer with no IO, host layer with no algorithm, is the reusability and reproducibility win: the cores carry no platform assumptions, and the host carries no learning-rate constants.

A consequence of the stateless-host dispatch model is that each learn verb constructs a fresh LearnSession over the host KV, so any core state that lives only in session RAM evaporates between dispatches. The bi-temporal graph is unaffected because it writes every edge straight through host_kv_* and is therefore durable across dispatches by construction, an edge inserted by one dispatch is queryable by the next. The three weight-bearing cores, the MicroLoRA/EWC instant adapter, the FastGRNN router, and the GAT attention, instead persist explicitly: each core serializes its learned tensors to a fixed KV namespace (rs-learn/learn/adapter, rs-learn/router, rs-learn/graph/attention) after every mutation and lazily reloads from that namespace when a dispatch finds its in-RAM slot empty. Only genuinely learned state is stored, the router’s low-rank weights and per-target quality counters, the adapter’s rank-2 factors and feedback count, the attention’s relation-edge feature weights, while seed-deterministic matrices reconstruct from the saved seed rather than being stored. The serialized blobs are length-guarded on load: a dimension mismatch reconstructs the core clean from its configuration rather than corrupting it, so a persisted blob from a differently-shaped run can never poison a later session. The KV namespace resolves under the project’s own .gm tree, so persisted core state is project-resident with the same discipline as the file-vec layer, no cross-project weight bleed. This is what turns the cores from dispatchable-but-amnesic into genuinely continual across the spool: a router trained by record_outcome in one turn routes with that training in the next, and a LoRA adapter grown by feedback keeps its adaptation.

The router closes the loop into the turn lifecycle itself. Every instruction dispatch embeds the active prompt and routes it through the FastGRNN router, attaching the result as a route_hint on the instruction response, the same way recall hits and the orient pack are attached. The hint carries a recommended model, temperature, top_p, context bucket, and a confidence, computed over the real model targets the agent can choose between. On a fresh project the router auto-initializes and falls back to a rule policy (the hint is marked algo: rule); as record_outcome dispatches accumulate per-target quality, the router crosses its trajectory threshold and begins routing from its learned low-rank weights, optionally exploring off-policy under epsilon-greedy. The hint is advisory, not binding: the agent reads it as a recommendation, the harness never forces a model. This is the concrete shape of using the agent as the LLM the router learns to dispatch, the model-selection decision becomes a learned, per-turn, persisted policy rather than a fixed constant, and because the router state persists across dispatches the policy compounds over the life of the project rather than resetting each turn. The prompt embedding the hint needs is served from the same per-query embed cache the recall path already populates, so the hint adds no second embedding pass on the hot instruction path.

The instant MicroLoRA core is a learned scoring head over embeddings, not an embedding transform. It is a rank-2 low-rank adapter that reads a 384-dimension bge-small vector and emits a per-target score (apply_adapter computes logit[k] += B·(A·embedding) for each target k); feedback nudges the low-rank factors A and B from a single trajectory-quality score in 0–1 (in the spirit of sona, 0 is a bad outcome and 1 a perfect one, a Hebbian update bounded by a max norm and corrected by EWC so new adaptation does not erase old). What it learns is which targets a given embedding should favour on this project — a fast head that scores from the embedding rather than reshaping the vector itself. The quality signal is the one thing the host cannot observe and the model in the loop always holds, so at the close of a turn the agent reports how well the turn went and the head's weights move toward the targets that proved good. This is the second face of using the agent as the LLM: the FastGRNN router learns which model to dispatch, the MicroLoRA head learns to score this project's targets from their embeddings, and both train from the same self-reported quality rather than from any signal the harness has to expose. Consuming the head's score in recall — re-ranking candidates by learned target-relevance — is the open integration; the head trains today, and wiring its score into the recall ranking closes the loop end to end.

That self-report is what closes the learning loop, and it resolves what once looked like a hard boundary. The bi-temporal graph auto-populates: every memorize-fire, after writing the file-vec text and embedding rows, also inserts a graph edge over the same embedding (no extra embedding cost), so the knowledge graph grows from exactly the facts the recall layer stores, and pruning a memory invalidates its edge in the same step so the graph never carries a dangling reference. The weight-bearing cores learn from the agent's own report rather than from anything the harness must expose, because plugkit is the harness and the agent is the model it dispatched: the model knows its own identity and its own outcome, so it reports them. rs-learn never calls out to a model for that judgment; it emits the need and the model in the loop supplies it, exactly as the file-vec pipeline's summarize step emits a pending the agent fulfils. The graph learns continuously from memory, the MicroLoRA and the router learn from the self-reported quality of each turn, and the loop is closed end to end without the host having to surface anything the agent already holds.

41. The WASM Migration

The orchestrator stack runs as a single WebAssembly cdylib loaded by a thin JavaScript wrapper under Node or Bun. No native binaries are built, downloaded, or published per platform. The reasoning is distributability and reproducibility: one artifact runs identically on every operating system the JS host supports, pinned by sha256, with no per-architecture build matrix and no platform-specific install path. The wasm imports a small host contract, filesystem, key-value, vector search, fetch, subprocess, clock, log, and the JS wrapper satisfies it by delegating to Node, so the wasm carries the logic and the host carries the IO. The same module loads in a browser given a host that satisfies the imports, which is the browser-compatibility path.

Two size stories live in one artifact. The plugkit orchestration logic compiles to a small wasm, the state machine, gates, dispatch, and verb routing are hundreds of kilobytes. The shipped artifact is large (~149 MB) because it embeds the embedding model in-wasm: BAAI/bge-small-en-v1.5, twelve hidden layers, 384-dimension output, a 133 MB safetensors weight blob plus a 711 KB tokenizer. Embedding runs wasm-side with no host delegation and no external service, recall and memorize compute vectors in-process, so memory works offline and leaks no text to a remote embedder. The trade is deliberate: a one-time large download buys a self-contained, reproducible, network-independent memory system rather than a small binary that phones home for every embedding.

System stability follows from the single-artifact discipline. The wasm is written atomically (temp file then rename); a version drift between the running instance and the on-disk version file triggers a clean self-respawn rather than a partial reload; embedding initialization is witnessed by an embed_init_ok event before the first recall. Shared resources, the kv store, the vector index, the embedding model, are owned by the host process and shared across every verb dispatch in a session, so a single model load amortizes over the whole session rather than re-initializing per call.

42. The Rust Service Crates: exec, codeinsight, search

Three Rust crates provide the capabilities the orchestrator dispatches into, each a single static artifact with zero runtime dependencies. rs-exec is the language runner: it executes nodejs, python, bash, typescript, deno, go, rust, c, cpp, and java through the spool, owns browser sessions, and enforces per-call timeouts and session task isolation, every execution carries a positive timeout because the wasm host runs subprocesses synchronously and an unbounded child blocks the watcher. rs-codeinsight is the AST x-ray: tree-sitter parsing across thirteen languages, analyzing a 500-file project in under 200 ms in one pass, so an agent landing in an unfamiliar repo gets files, symbols, and shape immediately instead of burning its first interactions grepping. rs-search is the hybrid finder: BM25 lexical scoring with identifier-aware tokenization (snake, kebab, and camel splits) fused with vector similarity via Reciprocal Rank Fusion, with .gitignore discipline, PDF ingestion, and a content-addressable embedding cache so re-indexing only re-embeds changed content.

Model selection across the stack follows a split between meaning and mechanics. Semantic work, summarize, classify, judge-if-two-texts-mean-the-same, routes to a small fast model through the fetch callback, one item per call, parallel by default. Mechanics, parse, sort, hash, count, stay in deterministic Rust. Embeddings use bge-small for its 384-dimension output and small weight footprint relative to recall quality. The unifying motivation is the same one that drives the wasm migration and the rs-learn boundary: each layer owns exactly one concern, the model owns meaning, Rust owns mechanics, the host owns IO, the algorithm crate owns math, so every layer is independently testable, reproducible, and swappable.

That static split, meaning to a model, mechanics to Rust, fixes which engine does a job; it does not fix which model a meaning-job uses. That second decision is where the FastGRNN router enters: rather than hard-coding a single model for the agent's own turns, the per-turn route_hint (§40) recommends a model, temperature, and top_p learned from the outcomes of prior turns on this project. The two tiers compose, the layer-concern split decides model-versus-Rust deterministically, and within the model tier the router turns the choice of which model into a learned, persisted, per-turn policy that compounds over the project's life. Using the agent as the LLM the router learns to dispatch is the same one-concern-per-layer discipline applied one level up: the harness owns orchestration, the router owns the model-selection policy, and neither is a fixed constant baked into the other.

43. Deviation Gates: Predictability Across LLMs

The discipline's promise is that the same project behaves the same way no matter which model drives it. That promise is kept not by trusting the model but by a small set of gates that refuse the disallowed dispatch and name the recovery. Each gate is a pure check over on-disk state, so it fires identically under any LLM and any harness: long-gap-no-instruction refuses a non-orienting verb dispatched more than five minutes after the last instruction and tells the agent to re-orient first; complete-chain-poll flags an agent that keeps dispatching instruction on an already-terminal chain instead of stopping; push-dirty refuses a push from an unclean worktree; client-edit-no-witness refuses the transition to COMPLETE while a client-side edit lacks its browser witness. The gate does not mutate the agent's command, it returns a refusal whose text is the next instruction, so a weaker model that would otherwise drift is steered the same way a stronger one steers itself.

A gate is only as good as its attribution: it must fire on the agent's mistakes and stay silent on the mechanism's own bookkeeping. Two refinements make the deviation count an honest signal rather than noise. First, the long-gap clock is reset whenever a fresh watcher boots, because a version upgrade kills the watcher, downloads a new artifact, and restarts, a multi-minute gap the agent did not cause; charging that gap to the agent's next dispatch would punish the upgrade, so the boot stamps the last-instruction timestamp to now. Second, the complete-chain-poll deviation fires only for a session-attributed dispatch, because the empty-body probes the cron and stop-hook machinery fire against a terminal chain are not an agent re-polling its own work; counting them inflated the deviation rate at every COMPLETE without reflecting any agent behavior. The principle generalizes: a deviation is an agent action that went off-discipline, and the gate must distinguish the agent's dispatches (which carry a session) from the harness's internal bookkeeping (which does not), or the predictability signal it produces is measuring the wrong thing.

Three-Layer Admission Filter

Sections 2–37 specialize a single structural invariant: every candidate operation passes three orthogonal admission tests in sequence before commit.

candidate → [L1 cost] → [L2 bounds] → [L3 direction] → execute

L1 Cost. Amortized cost ≤ prior-best envelope. Unmeasured marginal = inadmissible. ORIENT (§21) is the baseline probe.

L2 Bounds. Single-writer per surface (|F|=1). Hard cap per managed resource; backpressure to defer queue at watermark. State outside a sanctioned surface = unreconcilable.

L3 Direction. Lyapunov: Δd ≥ 0 rejects. Audit tuple (id, hash, ts) per accepted write. Trajectory classifier over rolling window: convergent | flat | divergent | chaotic; non-convergent holds. VERIFY's four-observation window [worktree-clean, remote-pushed, prd-empty, mutables-witnessed] is the convergence criterion.

Closure anti-shape (universal L3 violation). The agent's response body is not a mutation surface. Any artifact composed in the response that displaces a sanctioned dispatch fails L3 because no (id, hash, ts) exists, the recall index does not see it, the trajectory classifier does not register it, the gate does not advance. Recognition is structural, not by enumeration.

Maturity-first emit. First emit = closure of the destructive transform. Partial emit externalizes residual completion cost as implicit unaudited state; the next session pays the reconstruction cost. Closure-exceeds-reach decomposes along dependency edges (Maximal Cover DAG, §17), never along schedule.

Invariants. Measurement gates optimization. Bounds prevent cascades. Direction eliminates waste. Composability: layers are independently tunable.

The five phases are the filter at escalating commitment. PLAN runs L1+L2 cheaply via ORIENT and PRD-DAG. EXECUTE runs L3 over witnessed mutations into the central store. EMIT runs L3 audit on disk. VERIFY runs the trajectory classifier over the four-observation window. The filter is the engine; the phases are scheduling.

Code Invariants

Instruction prose extends to code emission. Every artifact carries the same structural disciplines.

  • State space minimized. Sequential downward flow; state flags evaluated in one phase. External inputs through a unified queue before mutation. State changes = explicit assignment, never buried side effect.
  • Hardware reality. Benchmark before abstracting. Scope passed explicitly; closures hide scope-resolution cost on hot loops. Pools over allocation. Native data flow on performance paths; reject Promise chains / class hierarchies / operator overloading there.
  • Flat structure. Denormalized graphs over nested documents. Partial-field updates. Bytes over JSON for transport; varint variable-width fields; pre-compute exact size and allocate once. Lexical ordering for deterministic tie-breaking.
  • 200-line vertical slices. One responsibility per file. Complete input→process→output. Zero-config defaults correct for 90%. Universal runtime: browser, Node, mobile, Bare.
  • Async boundary explicit. Sequential awaitable primitives. No implicit callback ordering. Unified error channel. Tests await real ops; mock-free.
  • Naming by scale. <50 lines: single-letter algebraic. 50–200: short descriptors. >200: full names. Iterators short; public APIs explicit.
  • Fail fast, loud, deterministic. Halt on precondition violation with exact state. Assert on emitted semantics, not return values. Sentinel words + checksum headers verified on every access.
  • Binary transport, append-only persistence. Lexical cursors for sparse reads. Append-only sequence for replay. Chunked by lexical range; modify only the touched chunk.
  • Single focused task per session. No drive-by refactors. Pre-compute and inline; code growth < cognitive overhead.

Token discipline. English describing intent = liability when code can encode it. Comments = liability when names + structure encode the same. Duplication that must sync = liability. Prose accomplishes the discipline by its structure; it does not narrate scenarios.

38. Discussion

38.1 Relationship to Formal Methods

The approach shares surface similarities with model checking (named state variables, explicit transitions) and design-by-contract (pre/post-conditions on advancement). It operates at a coarser abstraction level: the “states” are task phases, not program states, and the “contracts” concern what the agent knows rather than program correctness. The governance layer (§13) adds route families that function like typing domains: they partition the space of valid operations rather than verifying program properties.

38.2 Relationship to Test-Driven Development

There is a loose analogy to TDD: both emphasize verification before completion. The difference is that verifications in this system are not persisted as a test suite during EXECUTE, they exist as witnessed execution. The single integration test (§26) is the COMPLETE-phase residual of this witnessing. The 200-line budget trades regression depth for context-window economy.

38.3 External vs. Self-Regulation

A recurring design choice is external enforcement over self-regulation. Marker-file gates enforce phase boundaries because the same model that reasons through a problem also reasons through the justification for shortcuts. The agent cannot bypass the stop-checkpoint gate by deciding it has already verified; the dispatcher reads .gm/prd.yml and .gm/mutables.yml on disk, independent of the agent’s judgment. The hard rules in §§21–37 are externally enforced in the same sense: they live in the wasm orchestrator’s instruction output and in the marker-file gates, not in the agent’s in-context reasoning. The earlier hook-event architecture made this enforcement contingent on a working callback pump; the current architecture reduces it to a file read.

38.4 Overhead vs. Benefit

Every backward transition introduces overhead: discovering a new unknown during EMIT discards partial work and requires re-planning and re-executing. In production, this cost is offset by reduced late-stage rework. The snake-back mechanism surfaces problems early when their scope is still bounded; problems discovered at COMPLETE have compounded across all subsequent work.

38.5 Limitations

  1. Finite unknown assumption: Termination requires discoverable unknowns to be finite. In adversarial or chaotic environments this may not hold. Specifically, the two-pass reclassification allowance (§9.2) does not prevent an unknown from persisting across both passes if the underlying blocker is structural rather than transient. If an unknown persists after the second pass, the prescribed behavior is exec:pause with the blocker recorded in the PRD header, not reclassification to a third pass. Without this termination condition, persistent blockers could produce unbounded reclassification cycles.
  2. Witnessed execution cost: Real execution against expensive external APIs may be prohibitive for some resolution paths.
  3. Memory staleness window: Time between memory creation and verification creates a window for stale-memory-induced errors.
  4. Browser witness scope: The managed Chromium runtime is single-instance; browser tasks serialize and cannot exploit wave parallelism.
  5. Test budget ceiling: The 200-line cap is a forcing function for compact coverage, but genuinely large subsystems may not be adequately witnessed within it.

39. Conclusion

We have described the complete gm discipline: a phase-gated lifecycle model, a mutable-naming and witnessed-execution framework, an in-phase quality discipline, a governance layer with seven route families and four state planes, second-generation rules for autonomy and scope, operational hard rules that close gaps the earlier model left open, and a third generation of dispatch-boundary rules that move the discipline out of the agent’s context window and onto disk into marker-file gates.

The discipline is not a set of aspirational guidelines. Every rule in this document is enforced either by the spool-dispatch gate running below the agent’s decision layer, by file-based state the gate reads independently of the agent, or by hard-rule encoding in the orchestrator’s instruction response that the agent cannot reason its way around. The agent’s job is to execute within the envelope; the envelope’s job is to prevent the agent from collapsing the discipline under the pressure of local optimality.

Our experience operating a single-skill harness, gm-skill, one npm package, one wasm artifact, across every Claude-style host that can load a skill folder suggests that agent reliability depends substantially on lifecycle control, not just model capability. The marker-file architecture in particular suggests that reliability also depends on moving discipline-state out of the agent’s memory and out of the host’s event pump into externally-readable artifacts: a spool file, a mutables row, a stop-checkpoint marker, a CI run. The agent is a participant in the discipline, not the keeper of it; the host is its delivery surface, not its enforcer. We offer this body of work as a practical engineering artifact rather than a theoretical claim.
