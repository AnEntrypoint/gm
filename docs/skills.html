One skill, every verb

one skill body, one wasm orchestrator, you dispatch every phase on demand
cushion-certified · 2026
tl;dr
There is exactly one shipped skill: gm-skill. Its SKILL.md is roughly twelve lines and contains no phase prose. The state machine, PLAN → EXECUTE → EMIT → VERIFY → COMPLETE, lives inside the wasm orchestrator (rs-plugkit) and is served to the agent on demand through the instruction verb, with the current FSM state baked into every response. The agent dispatches verbs by writing files into .gm/exec-spool/in/<verb>/<N>.txt; the wasm watcher reads them, the orchestrator routes them, the output lands in .gm/exec-spool/out/. No chain of skill files, no subagent invocations.

The state machine, in one diagram

stateDiagram-v2
  direction TB
  [*] --> PLAN: user message
  PLAN --> EXECUTE: zero new unknowns last pass
  EXECUTE --> EMIT: every mutable witnessed
  EMIT --> VERIFY: post-emit matches pre-emit
  VERIFY --> COMPLETE: prd empty AND git clean AND CI green
  COMPLETE --> [*]

  EXECUTE --> PLAN: new unknown
  EMIT --> PLAN: new unknown
  VERIFY --> PLAN: new unknown
  EMIT --> EXECUTE: logic wrong
  VERIFY --> EMIT: broken output
  VERIFY --> EXECUTE: logic wrong

The FSM is owned by rs-plugkit/src/orchestrator/{mod,state,transitions,mutables,memorize}.rs. The agent never advances the state machine directly; it dispatches the transition verb with the requested target, and the orchestrator either accepts (returns the next-phase instruction body) or refuses with a reason naming the unmet gate condition.

How the one skill works

The gm-skill SKILL.md body says, in essence: on every turn, dispatch the instruction verb through the spool; obey what it returns. It contains no phase prose, no checklists, no transition diagrams, those would drift the moment the orchestrator changed. Every instruction surface the agent reads at runtime is generated by the orchestrator with the current FSM state, the current PRD, the current mutables, and the current residual-scan state baked in.

This collapses what used to be a chain of fourteen skill files (gm, planning, gm-execute, gm-emit, gm-complete, update-docs, governance, code-search, browser, ssh, pages, create-lang-plugin, memorize, textprocessing) into a single dispatch surface. The behaviors those files used to elicit are now properties of the orchestrator’s response to instruction and the verbs the agent dispatches in reply.

The orchestrator verbs

These are the verbs the agent uses to drive the state machine. Each is dispatched the same way: write a request body to .gm/exec-spool/in/<verb>/<N>.txt, read the response from .gm/exec-spool/out/<verb>-<N>.json.

instruction

phase prose, generated per callevery turn
roleReturn the current phase’s imperative body, with the live FSM state, PRD summary, mutables status, residual-scan state, and recall pack already woven in. Replaces the bodies of the former gm / planning / gm-execute / gm-emit / gm-complete / update-docs / governance / code-search skill files.
forbidsTreating the response as static. The body changes every call as state advances.

transition

phase advance / regressphase gate
roleRequest a forward transition (ladder) or backward transition (snake). Orchestrator validates against the four state planes (epistemic, operational, relational, temporal). Returns the new phase’s instruction body on success, or a refusal naming the unmet gate.

phase-status

read current FSM stateany
roleOne-shot probe of where the orchestrator thinks the agent is. Used by the harness to render breadcrumbs and by the residual-scan gate to decide whether to fire.

prd-add / prd-resolve / prd-list

work-item ledgerPLAN, EXECUTE, COMPLETE
roleAppend PRD items, mark them resolved with witness evidence, enumerate the live state. Backed by .gm/prd.yml; the orchestrator owns the serializer so concurrent dispatch from parallel exec_js calls cannot corrupt it.

mutable-add / mutable-resolve / mutable-list

named-unknown ledgerPLAN, EXECUTE
roleEvery unknown surfaced during PLAN lands here with status: unknown; mutable-resolve flips a row to witnessed only when witness_evidence (file:line, codesearch hit, exec output snippet) is filled. Write/Edit/git operations are hard-blocked while any row is still unknown.

memorize-fire

background fact writeEXECUTE, EMIT
roleClassifies the supplied fact and persists it to AGENTS.md (top-level rules) or rs-learn (everything else). Replaces the prior Agent(subagent_type='gm:memorize') spawn; this is a direct verb dispatch.
forbidsInline edits to AGENTS.md or CLAUDE.md. Both are managed exclusively through this verb.

residual-scan

stop-gate challengeCOMPLETE
roleFires once per stop window when the PRD is empty/missing AND no open browser sessions AND no running tasks. Agent either expands the PRD with in-spirit residuals and re-enters EXECUTE, or explicitly states “residual scan: none reachable in-spirit.” Marker .gm/residual-check-fired keeps the challenge one-shot per window.

auto-recall

prompt-submit recall injectionevery prompt
roleThe orchestrator derives a 2–6 word recall query from the user prompt, calls rs-learn, and emits a “## Recall for this prompt” block into the next instruction body. Together with the session-start codeinsight overview, this ensures every turn begins with prior memory loaded.

The host verbs

Side-effecting capabilities the agent uses inside a phase. All dispatched through the same spool surface; all serviced by the wasm host.

exec_js

synchronous code execution
roleRun a snippet in nodejs, python, bash, typescript, deno, go, rust, c, cpp, or java. Synchronous under wasm: the host runs the subprocess via child_process.spawnSync and blocks the watcher for the duration. Every call requires a positive timeoutMs; missing or zero is a hard error. Partial output preserved on timeout.

codesearch

project-local search
roleBM25 + embedding fusion over the project tree, plus tree-sitter symbol layer. Two-words-then-iterate query protocol; minimum four attempts before declaring absent. Emits [index fully synced: …] so freshness is visible.
forbidsfind, grep, rg, Glob, Explore, the Bash gate denies these in favour of codesearch.

recall / memorize

read/write memory
roleBacked by .gm/rs-learn.db (KV + vec). Per-discipline isolation via @<name> sigil in the request; without a sigil, recall fans broadly across enabled disciplines and memorize writes narrowly to default.

browser

live-page witness
roleThe wasm host’s managed browser session. Boot the surface, navigate, page.evaluate the invariant, capture witnessed values. Required for any edit to code that runs in a browser. The only sanctioned way to drive a live page; parallel browser tooling fragments witness state.
forbidsscreenshot without page.evaluate. Treating “node test.js passes” as substitute for a live witness.

fs_read / fs_write / fs_stat / fs_readdir

file-system access
roleWasm-host filesystem verbs. Used by the orchestrator and by user-side dispatch for any read/write that does not go through Edit/Write.

kv_get / kv_put / kv_query

project-scoped KV store
roleBacked by rs-learn. Used by the orchestrator for residual-scan markers, gm-fired sentinels, mutables/PRD serialization, and any project-state the agent needs to survive context compaction.

fetch

HTTPS GET/POST
roleDirect HTTPS egress. Embedding calls are serviced in-wasm via baked Q4 weights; LLM-tier work is offloaded to the host agent via callback protocol.

health

one-shot system probe
roleReturns plugkit version + pin-match, watcher liveness, runner state, rs-learn status, cache dirs, inbox/outbox counts, recent dispatch fires, recent errors. Use before assuming any component is broken.
