Lawful Downgrade, Bounded Autonomy,
and the Closing of the Learning Loop

AnEntrypoint
April 2026
Abstract
We describe four further advances to the gm orchestrator, each addressing a failure mode that survived the governance discipline of [3]. First, we promote lawful downgrade from a principle into a hard rule, and we strengthen its constructive half: refusing a task whose full scope exceeds the agent’s reach is a forced closure, structurally identical to bluffing strength — both bypass witnessed execution. The required move is not to pick a bounded subset, which silently re-introduces the same triage failure (the parts not picked are still refused), but to construct a covering family of bounded subsets — a group of witnessable subsets whose union covers as much of the requested scope as the session can reach — and execute every member. Second, we tighten autonomy: once a PRD exists, asking the user mid-execution is forbidden except under a narrow last-resort clause, and even then routed through a structured pause channel rather than free-text questions. Third, we close the learning loop by making recall a hard precondition for naming any unknown, paired with the same-turn memorize handoff of [3]; together they form a read-write loop on the same fact store, so a fact resolved in session N is automatically available in session N+1 without re-discovery. Fourth, we harden the execution budget: every exec:<lang> call now carries a mandatory timeout enforced by the runner, and every fetch-style operation carries a stall watchdog independent of the socket-idle timeout. The first three are policy advances enforced by skill-level instructions; the fourth is a runtime invariant enforced by the Rust runner. Together they close the remaining categories of unwitnessed work that survived [1], [2], and [3].

1. Introduction

Paper I [1] introduced the phase-gated state machine; paper II [2] addressed in-phase quality; paper III [3] introduced the governance layer separating route discovery, weak-prior handoff, and emission legitimacy. Operational experience over the subsequent month surfaced four residual failure modes that none of the prior disciplines closed:

  1. Refusal-as-rigor, and its quieter sibling, single-subset triage. An agent presented with a task whose full scope exceeds what is witnessable from the current session sometimes refuses on grounds of intellectual honesty. Paper III treated bluffing strength as the failure to refuse; the inverse failure — bluffing weakness, refusing the witnessable subset — was unaddressed. The inverse failure has a subtler form: the agent attempts a single bounded subset, ships it, and treats the remainder as “follow-up.” That is still refusal, distributed: every other witnessable subset of the request was abandoned without being attempted. The fix is not a subset but a cover.
  2. Permission-seeking inflation. An agent with a written PRD sometimes interrupts execution to ask the user for permission to continue, to confirm scope, to choose between options, or to summarize partial work. Each interruption shifts cognitive load to the user and re-opens scope that the PRD had closed. Paper III did not specify autonomy boundaries.
  3. Forgetting between sessions. Paper III closed the same-turn forgetting window via the memorize handoff. It did not address the read side: a fact memorized in session N is useful only if session N+1 looks for it. An agent that resolves an unknown by fresh execution when memory already holds the answer is duplicating work and silently disagreeing with its own past witness.
  4. Unbounded exec costs. The exec:<lang> dispatch makes process spawn cheap. Hooks that invoke the runner without a budget can leak children under failure: every retry spawns again, every retry inherits no kill deadline, and the system thrashes. The architecture made the failure cheap to encounter; only an explicit budget makes it cheap to bound.

This paper describes the discipline advances addressing each. §2 promotes lawful downgrade. §3 specifies the autonomy contract. §4 closes the learning loop. §5 hardens the execution budget. §6–8 discuss related work, limitations, and conclusions.

2. Lawful Downgrade As Hard Rule

2.1 The Symmetry of Bluffing

Paper III §2.5 named the principle: lawful downgrade — writing a weaker, true statement in place of a stronger, unearned one — is always available; forced closure never is. The principle was framed against bluffing-strength: claiming a conclusion that has not earned route_fit, authorization, repair_legality, and stress-suite passage. Subsequent operation revealed the symmetric failure.

An agent presented with a multi-part task — “document the distribution strategy on the website AND write the next paper” — sometimes locates a part it cannot fully witness from inside the session (e.g., a third-party site to verify, a long-running CI cascade to wait on, a deployment gate to query) and concludes that the entire task is out of reach. The conclusion takes one of these shapes:

  • “Honest stop — I cannot do this from inside this conversation.”
  • “Stopping for a hard, honest call.”
  • “Pretending I can would be the most expensive kind of lie.”
  • “X is unavailable in this environment” (when X is in fact available).

Each refusal frames itself as intellectual honesty. Each is, in fact, a forced closure: the agent has selected a single failure mode (full-scope completion) and let it dominate every other available action. The user receives no work, no partial result, no PRD, and no explicit gap statement they can redirect. The conversation degrades worse than if the agent had attempted the bounded subset and named the residual scope as a follow-up.

2.2 The Symmetric Failure

The two failures are structurally identical:

failurebluff strengthbluff weakness
authorization claimstronger than earnedweaker than earned
user receivesunearned conclusionno work product
witnessed executionskipped by overclaimingskipped by underclaiming
recovery movedowngrade conclusionupgrade attempt

Both move in the wrong direction along the same axis. Both cost a session. Both look, from the outside, like a careful agent acting in good faith.

2.3 The Required Move: A Covering Family

When the requested scope exceeds the agent’s reach, the discipline is to construct a covering family rather than triage to a single subset. Let R be the requested scope and let W be the set of operations witnessable from this session. The agent must:

  1. Decompose R into a family of bounded subsets {S1, S2, …, Sn} such that each Si ⊆ W and the union ⋃Si is maximal — i.e., it covers as much of R as W permits, with no Si ⊆ W deliberately omitted.
  2. Write the family into the PRD as separate items, with their dependency graph made explicit so EXECUTE can parallelize the independent ones.
  3. Execute every member to COMPLETE. Partial coverage of R by an incomplete family is still distributed refusal.
  4. At end-of-turn, name the residual R \ ⋃Si as a follow-up item — explicitly, with the reason each piece was excluded from W (irreversible side-effect with no PRD coverage, third-party system the session cannot reach, etc.). Residual scope must be the complement of a maximal cover, never the convenient remainder of a single triaged pick.

The single-subset move is a degenerate case of the cover (n=1) that is legitimate only when no further Si ⊆ W exists. Choosing n=1 by triage when n>1 is feasible is structurally the same failure as refusal: parts of R that could have been witnessed were not attempted. The user can redirect a delivered cover plus a named complement; the user cannot redirect either a refusal or a silently-narrowed pick.

Lawful Downgrade (hard form). Refusal is forced closure. Refusing the task because part of it is hard is the inverse of bluffing — both bypass witnessed execution. The required move is to construct a maximal covering family of witnessable subsets, PRD-write the family, execute every member, and name the residual complement as follow-up. A single subset suffices only when no other witnessable subset exists; otherwise, single-subset delivery is distributed refusal.

2.4 Detecting The Failure Mode

The discipline is enforced by what the agent does, not by which words appear in its output. The test is constructive: at the moment the agent commits its next action, does that action deliver work toward the request, or does it withhold? Two classes of withholding are recognized:

  • Total refusal. The next action is a closing statement, an apology, or a request for confirmation, and no PRD covering any subset of R exists or is being written. This is the failure paper III named.
  • Distributed refusal. The next action is a PRD covering a single subset S1 of R, but additional witnessable subsets S2, …, Sn exist and were not enumerated. The PRD looks like work; the omission is the refusal. This is the failure §2.3 names.

Both are detected the same way: the agent, before closing the turn, checks whether the union of its committed work plus its named follow-up complement equals the witnessable closure of R. If something witnessable falls in neither set, the discipline is broken and the planning skill must be re-entered to expand the cover. Skill instructions do not enumerate phrases to forbid; the implicit principle — deliver the maximal cover, name the exact complement — is what the agent must internalize. Phrase-policing trains the agent to evade the lexicon while preserving the failure; principle-policing trains it to expand the cover.

3. Bounded Autonomy

3.1 The PRD As Authorization

A PRD is not a draft. It is the authorization that subsequent phases consume. Once .gm/prd.yml exists with non-empty contents, EXECUTE owns the work and the user is no longer in the loop until COMPLETE. Pre-paper-IV practice allowed the agent to revisit scope mid-execution by asking the user; this practice is now forbidden.

3.2 The Asking-Cost Principle

Any question the agent could resolve with its own tools — by reading code, querying memory, or running an exec probe — is a question the agent owes itself, not the user. The cost of asking is rarely the user’s annoyance; it is the loss of the witnessed-execution discipline. An interruption mid-execution shifts the cost of resolving a doubt from EXECUTE (where it belongs) to the user (where it doesn’t), and replaces a witnessed answer with a verbal one. The agent’s options at the moment of doubt are: invoke the next skill in the chain, regress to PLAN if a new unknown surfaced, or use witnessed execution to resolve the doubt. Asking the user is not on the list. We deliberately do not enumerate forbidden phrasings here: the test is constructive (did the next action deliver work or withhold it), and a phrase-list trains evasion of the lexicon while preserving the failure.

3.3 The Last-Resort Clause

Asking is permitted only as last resort:

  • The decision is destructive-irreversible AND has no PRD coverage.
  • OR the user’s intent is genuinely ambiguous AND cannot be inferred from PRD, memory, or code.

Even then the channel is structured: the agent runs exec:pause, which renames .gm/prd.yml → .gm/prd.paused.yml and writes the question into the file’s header. In-conversation asking is last-resort beneath last-resort. The structured pause makes the question (a) durable across compaction, (b) revisitable in a later session if the user does not respond immediately, and (c) inspectable by the user as part of the same artifact that documents what work is paused.

3.4 Why Not Ask More

The cost of asking is rarely the user’s annoyance; it is the loss of the witnessed-execution discipline. An agent that asks “should I do X?” is offering the user a choice that the agent should have resolved by reading the code, querying memory, or running an exec probe. The user, lacking the agent’s tools, answers from intuition. The agent then proceeds with weak-prior authorization that the user supplied verbally rather than witnessed evidence the agent gathered. The whole architecture of paper III’s governance layer is replaced by “the user said so.”

The autonomy hard rule restores the architecture. The PRD is the user’s authorization. The agent’s job is to honor it, not to renegotiate it.

4. Closing The Learning Loop

4.1 Recall As Hard Precondition

Paper III tightened the write side of fact persistence: same-turn memorize handoff at the moment an unknown resolves. The read side was unspecified. Operational evidence showed agents resolving the same unknown across multiple sessions because no one had told them to look first.

The discipline introduced here is: before resolving any unknown by fresh execution, run exec:recall <2-6 word query> against the rs-learn store. The query cost is small (~200 tokens, ~5 ms when the embedding service is hot); the savings on a hit can be enormous. Hits are treated as weak prior per paper III §2.3, not as authorization — the agent still witnesses the recalled fact via EXECUTE before adopting it — but having a starting point dramatically reduces the search space.

4.2 Trigger Contract

The recall trigger fires when any of the following holds:

  • The unknown matches a topic the agent suspects has been seen.
  • The current task is a sub-task of a known project.
  • The agent is about to ask the user something likely already discussed.
  • The agent is about to make a design decision where a prior decision likely exists.
  • A quirk feels familiar.

Recall is skipped only on brand-new projects, trivially-bounded edits, or surgical user instructions. The default is to recall.

4.3 The Loop

Recall + memorize together form a closed loop on a single fact store:

UNKNOWN
  — recall —> HIT (weak_prior)
                 — witness —> KNOWN
                 — (no memorize: already stored)
  — recall —> MISS
                 — witness —> KNOWN
                 — memorize (background, parallel) —> stored

session N+1: same UNKNOWN
  — recall —> HIT —> (continues from line 2)

The loop guarantees monotone progress: every unknown resolved in any session is available to every subsequent session, and every recall pass is itself a witnessed probe of memory state. The store is shared across machines because .gm/rs-learn.db is committed, not gitignored. The store is shared across agents because rs-learn is a library, not a daemon.

4.4 Memorize Triggers, Re-stated

Paper III enumerated the memorize triggers; we restate them here for completeness, with the read-side recall trigger paired:

  • An exec: output answers a prior “let me check” → memorize.
  • A code read confirms or refutes an assumption → memorize.
  • A CI log reveals a root cause → memorize.
  • The user states a preference, constraint, deadline, or decision → memorize.
  • A fix works for a non-obvious reason → memorize.
  • A tool or environment quirk bites once → memorize.

Each memorize call is a separate Agent(subagent_type='gm:memorize', model='haiku', run_in_background=true) invocation, parallelized when multiple events resolve in the same turn, never serialized.

Learning Loop Invariant. Skipping recall breaks the loop on the read side; skipping memorize breaks it on the write side. An agent that does only one is doing half the work and growing a memory store it never consults.

5. Hardening The Execution Budget

5.1 The Unbudgeted Exec Class

Paper I introduced exec:<lang> as the only legal route to local code execution. The contract was correctness (real code, real output) but not bounded cost. A series of production incidents showed the cost-axis failure: hooks that invoke the runner without a deadline can leak children indefinitely, particularly under retry storms. One incident produced “hundreds of node processes” that thrashed the host before manual intervention.

The architectural cause is simple: the runner’s execute RPC accepted a code blob and a runtime, with no timeout parameter. A naive caller that retried on transient failure would re-spawn the same long-lived child each time, with no kill deadline on any of them.

5.2 The Three-Layer Budget

The remediation places budgets at three layers:

  1. CLI: rs-exec exec --timeout <ms> and rs-exec bash --timeout <ms> are mandatory. clap rejects missing --timeout with exit 2 and zero with “--timeout must be > 0 ms”. There is no default at the CLI layer.
  2. RPC: The execute RPC accepts an optional timeoutMs. A missing or zero value applies a 300-second default and emits a stderr deprecation warning naming the missing field. The default is a transitional safety net, not a license for unbounded waits; downstream callers (rs-plugkit, rs-codeinsight, rs-search) are expected to pass per-call-site budgets and silence the warning.
  3. Runner: The runner enforces the budget via tokio::time::timeout wrapping wait_for_completion. On elapse, the runner removes the task from the active store, calls kill_tree(pid) to terminate the child and any descendants, and fails the task with the error string “execution timed out after <N> ms”. The CLI passes timeout + 5000ms as the RPC read deadline so the client does not time out before the runner can respond with the kill notice.

The combination is: the user’s budget is the source of truth, the runner enforces it as a kill deadline, and the protocol survives partial failures (slow runner, slow client, late stdout) without losing the kill guarantee.

5.3 The Stall Watchdog

A separate failure shape appeared in the bootstrap fetch path. https.request’s timeout option fires only on socket-level inactivity — the connection going completely silent. A response stream that opens, transmits a few bytes, and then stalls for thirty minutes never trips the socket-idle timer because the TCP layer is technically alive. The user observation was “the fetch is at 0% network activity but the timeout never fires.”

The remediation is a stall watchdog that runs alongside the socket-idle timeout. The watchdog records the timestamp of every received chunk; a 2-second timer checks whether 20 seconds have elapsed since the last chunk; if so, it destroys the request with a stalled: no bytes for 20000ms error. Combined with the socket-idle timeout (60 seconds) and the maximum attempt count (5 with backoff 2s/5s/15s/30s), the worst-case time to declare a fetch dead is approximately 2 minutes rather than 30.

5.4 Kill-Tree

The runner uses a per-platform kill-tree implementation rather than killing the immediate child. On Windows this matters because cmd.exe spawns child shells that survive the parent. On Unix it matters because bash -c with a pipeline spawns a process group whose leader does not necessarily own all the children. The kill-tree implementation walks the process tree from pid downward via sysinfo and sends the platform-appropriate termination signal to each. A timeout that fails to propagate to the leaves is no timeout at all.

Execution Budget Invariant. Every exec:<lang> call carries a budget. The runner enforces the budget by killing the task tree on elapse. No call path leaks children past its budget. Fetch-style operations carry stall watchdogs independent of socket-idle timeouts, so a stalled stream cannot hide behind a live TCP connection.

6. Relationship To Earlier Work

Lawful downgrade as hard rule (§2) extends paper III §2.5 from principle to enforced contract by specifying the constructive move — the maximal covering family — and by recognizing that the failure mode has both a total form (refusal) and a distributed form (single-subset triage that silently abandons the rest). The principle was correct; what was missing was the symmetric failure mode and the constructive obligation that the cover be maximal rather than convenient. Operational experience showed that, without that obligation, agents would attempt one subset, ship it, and treat the remainder as “follow-up” — technically not refusing, in practice refusing everything outside the chosen slice. Earlier drafts of this paper enumerated forbidden phrases as the enforcement surface; we now treat the constructive obligation itself as the enforcement surface, since phrase-lists train evasion of the lexicon while preserving the failure.

The autonomy hard rule (§3) operates orthogonally to paper III’s governance layer. Governance gates emission; autonomy gates pre-emission interruption. Both protect the witnessed-execution discipline; governance from over-claiming, autonomy from outsourcing the claim to the user.

The learning loop (§4) extends paper III §5 from a write-only handoff to a read-write loop. The earlier paper closed the same-turn write window; this paper closes the cross-session read window by making recall a precondition. The combination produces a memory store that is consulted at every unknown and updated at every resolution, eliminating both same-turn forgetting and cross-session re-discovery.

The execution budget (§5) addresses a class of failure neither paper I nor paper III named: cost-axis failures of the witnessed-execution mechanism itself. Paper I established the witnessing contract; this paper establishes the kill deadline that makes the contract operationally safe at scale. The Rust runner’s tokio::time::timeout wrapper plus kill-tree is the mechanism; the mandatory CLI --timeout and the RPC default-with-warning are the policy.

7. Limitations

  1. Lawful downgrade is a discipline, not a gate. The constructive obligation — deliver a maximal covering family, name the exact complement — catches both total refusal and single-subset triage when the agent honestly attempts to enumerate W. An agent that under-enumerates the witnessable closure (mistakenly believing some Si is out of reach when an exec probe would settle it) still produces a smaller-than-maximal cover that looks compliant. The discipline reduces but does not eliminate the failure surface; recall (§4) is the structural mitigation, since prior sessions are most likely to know which probes resolve which uncertainty about W.
  2. Autonomy boundary is fuzzy at the destructive-irreversible line. An agent reasoning about whether a proposed action is destructive-irreversible can err in either direction: asking too often (autonomy violation) or assuming reversibility too readily (destructive action without consent). The structured pause channel (exec:pause) lets the agent be conservative without re-opening the in-conversation question, but the boundary classification itself remains a judgment call.
  3. Learning loop is bounded by query quality. Recall is keyword-similarity over the rs-learn store; an unknown phrased in vocabulary disjoint from the stored fact can miss. The protocol mitigates by recommending two-to-six-word queries and accepting that a miss is a permitted outcome that flows through to fresh execution. Embedding drift across model upgrades is an unaddressed concern.
  4. Execution budget assumes correct caller behavior. The 300-second RPC default is transitional. A caller that ignores the deprecation warning indefinitely receives 300-second tasks instead of correctness. The default is a safety net, not a substitute for per-call-site budgets, and a future revision is expected to remove the default.
  5. Stall watchdog is fetch-specific. The 20-second-no-bytes pattern protects HTTP fetches; a long-running compute that produces no output for an extended period is indistinguishable from a stall by this mechanism alone. Compute tasks are bounded by the execution-budget timeout instead. The two mechanisms are complementary; neither covers the other’s shape.

8. Related Work

The lawful-downgrade hard rule is operationally adjacent to graceful degradation in distributed systems [9]: a service that cannot satisfy the full request returns a reduced but truthful answer rather than failing closed. The agent-coding context adds the requirement that the reduced answer be witnessed, not merely plausible — degradation without witnessing is just bluffing weakness.

The autonomy contract has analogues in contract-based design [10]: a precondition-postcondition pair that, once agreed, both parties honor without renegotiation. The PRD-as-authorization stance treats the PRD as the precondition the user agreed to and the COMPLETE state as the postcondition the agent owes; mid-execution renegotiation breaks the contract.

The learning loop’s pairing of recall and memorize echoes experience replay in reinforcement learning [11], where prior episodes are sampled into the current decision process. The differences are that gm’s store is structured rather than tensor-encoded, the recall is keyword-driven rather than statistically sampled, and the witness step in the middle prevents stale memories from autonomously authorizing new emission.

The execution-budget invariant is a specific instance of resource-bounded computation [12]: every operation carries a deadline; deadlines propagate through composition; the system enforces them at the lowest layer. The tokio::time::timeout wrapper plus kill-tree is the standard implementation; the contribution here is the three-layer (CLI, RPC, runner) policy that ensures the budget is present at every entry point.

9. Conclusion

We have described four advances. Lawful downgrade is now a hard rule rejecting both the total and distributed forms of the symmetric failure to bluffing-strength: refusal of a task whose part is hard, and single-subset triage that silently abandons the witnessable rest, are structurally identical to over-claiming. The constructive obligation is a maximal covering family of bounded subsets, executed in full, with the residual complement named explicitly. Autonomy is bounded: a written PRD is the user’s authorization, and asking mid-execution is forbidden except as last resort routed through a structured pause. The learning loop is closed: recall before naming any unknown, memorize at the moment of resolution, both reading and writing the same rs-learn store across sessions. The execution budget is hardened: every exec:<lang> carries a kill deadline enforced by the runner, every fetch carries a stall watchdog independent of socket-idle timeouts.

The four advances share a structure. Each names a failure mode that survived the prior disciplines; each maps the mode to an existing primitive (PRD, recall, memorize, exec timeout); each upgrades the primitive’s contract from suggested to required. None introduces a new mechanism. The cumulative effect is that more of the agent’s behavior is mechanically gated rather than narratively endorsed: the agent that wants to refuse, ask, skip recall, or invoke the runner without a budget finds the path closed and is forced into the witnessed-execution channel.

Our experience continues to suggest that agent reliability improves monotonically with the elimination of unwitnessed work. Paper I removed unwitnessed local execution. Paper II removed unwitnessed in-phase quality. Paper III removed unwitnessed emission and unwitnessed remote actions. This paper removes unwitnessed refusal, unwitnessed user-asks, unwitnessed memory consultation, and unbounded execution cost. The witnessed-execution principle has not changed since paper I; what continues to change is the perimeter at which it is enforced.
