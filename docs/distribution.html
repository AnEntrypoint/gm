The SQLite Model:
One Skill, One Wasm, Open All The Way Down

AnEntrypoint
May 2026
Thesis
A coding agent should install in one command and never need a build toolchain on the user’s machine. It should also have no proprietary core: every artifact in the distribution must be built, in public, by CI, from source repositories anyone can fork. SQLite achieved this combination twenty-five years ago for an embedded database. gm applies the same combination to coding-agent infrastructure: the user gets one npm package — gm-skill — whose body is roughly twelve lines and whose engine is a single wasm module loaded under any Node-capable harness. The artifact’s entire genealogy — every Rust crate, every workflow, every published wasm — is open source under one organization, one cascade, one push at a time.

1. Why Distribution Is The Hard Part

Most coding agents fail at the install step. They ask for a Python virtualenv, a Node version, a Rust toolchain, an environment variable, a config file, an API key, and a directory layout the user has to memorize. By the time the agent runs, the user has spent more time on it than they intended to spend in a week. Worst, every component the agent needs is a separate vendor decision the user has to ratify implicitly.

SQLite’s design choice was: ship one C file. No toolchain, no daemon, no driver, no version skew. The user’s code links it; that’s the install. Two decades later, SQLite is in every phone, every browser, every aircraft. The lesson is not that databases should be embedded — the lesson is that the install experience is the product, and that radical simplicity at install is compatible with radical openness underneath.

For an agent, the equivalent is: one published skill package, with a single hook into the host harness, with all the heavy machinery (search, indexing, memory, code execution, browser automation, phase orchestration) compiled into one wasm module that the agent dispatches on demand through a file-spool surface. The agent is the state machine; the wasm is the durable scratchpad it drives.

2. The One-Command Install

gm-skill is one npm package. Any harness that can resolve a Claude-style skill folder installs it the same way:

bun x skills add AnEntrypoint/gm-skill -y -g

Then add this line to your agent’s global memory / system prompt:

always use the gm-skill skill for everything, always fan out subagents

That’s the whole thing. No cargo, no Python, no PATH edits, no plugin marketplace, no per-platform installer. The skill body is a ~12-line SKILL.md that tells the agent to dispatch verbs through the file-spool. The first time the agent dispatches anything, gm-plugkit (a companion npm package, auto-resolved as a dependency) ensures the wasm artifact and the plugkit-wasm-wrapper.js loader are on disk under node_modules/plugkit-wasm/. After that, every dispatch is a local wasm.instantiate plus a dispatch_verb call. The wasm artifact is under 200 KB; the agent never sees a download progress bar after the first run.

2.1 What “Turnkey” Excludes

  • No host toolchain. The user does not need cargo, rustup, cmake, gcc, or platform SDKs.
  • No native build step at install. The single wasm artifact runs everywhere a Node 20+ WASI host runs — one binary for every OS and every CPU.
  • No daemon to install or supervise. The watcher and host run inside the same wasm instance as the orchestrator. There is no port, no rpc daemon, no kill -9 story.
  • No mandatory secrets at install. Embeddings run in-wasm via baked Q4 weights; LLM-tier work is offloaded to the host agent via callback. Offline use is the default.

3. Open All The Way Down

The SQLite analogy holds on the openness axis only because every link in the chain is in public. There is no “and then a private build server takes over” step. The chain has six repositories under one GitHub organization (AnEntrypoint), and every one of them is MIT-licensed and forkable in its current state.

rs-execWasm host helpers. Compiled to wasm32-wasip1. Provides exec_js, browser session management, file-system glue, and the spool-watcher loop that the orchestrator imports. No daemon, no port; runs inside the wasm host process.
rs-codeinsightCode structure analysis via tree-sitter. Compiled to wasm. Powers the codesearch verb’s symbol layer and session-start architectural overviews.
rs-searchFile-content search. BM25 plus embedding fusion, scan-embed-sweep pipeline. Compiled to wasm. The codebase index lives under .gm/code-search/ (tracked in git so the index travels).
rs-plugkitThe orchestrator. Compiled to wasm. Owns the PLAN→EXECUTE→EMIT→VERIFY→COMPLETE state machine in src/orchestrator/, exposes the dispatch_verb wasm export, services every orchestrator verb (instruction, transition, phase-status, prd-add, mutable-resolve, memorize-fire, residual-scan, auto-recall) and routes host verbs to the imported helpers. The Cargo.toml here is the version source of truth.
rs-learnMemory layer. KV plus vec store backing recall and memorize. Compiled to wasm. Per-project store under .gm/rs-learn.db, tracked in git so memory follows the project across machines.
gmThis repository. Builds gm-skill via node cli.js gm-starter ./build. Publishes the npm package on every cascade tick. Holds the canonical SKILL.md, the bootstrap shim, and the version pins in gm-starter/gm.json.

4. The Cascade

Six repositories cooperate but no human steps between them. A push to any of the sibling Rust repos triggers a deterministic cascade that ends with a fresh gm-skill publish on npm:

rs-exec / rs-codeinsight / rs-search / rs-learn — push to main —> .github/workflows/cascade.yml fires rs-plugkit release.yml via gh workflow run + PUBLISHER_TOKEN (siblings publish their own @anentrypoint/<name>-wasm artifacts independently; the cascade does NOT recompile their source into rs-plugkit) rs-plugkit — release.yml: build plugkit.wasm (wasm32-wasip1, ~<200 KB) — publish plugkit-bin Releases + npm plugkit-wasm — bump gm-starter/gm.json::plugkitVersion and gm-starter/bin/plugkit.wasm.sha256 in AnEntrypoint/gm gm — that bump commit triggers publish.yml — node cli.js gm-starter ./build (regenerates skill body) — npm publish build/gm-skill (one package, one tag) users — bun add -g gm-skill (or harness-local install) — on first dispatch, gm-plugkit ensures plugkit.wasm is present — every subsequent verb is a local wasm call

No human edits a version string. No human runs cargo build. The cascade is the publishing pipeline. The phrase “to update anything, push to the relevant repo” is literal: a one-line fix in rs-search propagates to a new gm-skill on npm within one CI cycle.

5. One Skill Replaces Fifteen Plugin Repos

An earlier generation of this project fanned out fifteen downstream plugin repos (gm-cc, gm-gc, gm-oc, gm-kilo, gm-codex, gm-qwen, gm-copilot-cli, gm-hermes, gm-thebird, gm-vscode, gm-cursor, gm-zed, gm-jetbrains, gm-antigravity, gm-windsurf), one per host editor or CLI, each holding its own bootstrap shim and per-platform native binary. Those repos are now archived; they hold only a redirect README pointing at gm-skill.

The fan-out existed because every host had a different conception of what a “plugin” was. Once every host gained the ability to load a Claude-style skill directory, the fan-out became dead weight: fifteen pieces of orchestration around a payload that wanted to be one package. Skills are the universal harness; gm-skill is the universal payload.

6. The Bootstrap, In One File

On first dispatch, gm-plugkit/lib/skill-bootstrap.js runs once per session:

  1. Resolve plugkit.wasm via the bundled plugkit-wasm npm dependency. If absent (rare; npm install was offline), fetch from github.com/AnEntrypoint/plugkit-bin/releases/download/v<version>/plugkit.wasm, verify against the bin/plugkit.wasm.sha256 pinned in the skill.
  2. Atomically write the artifact under .gm/plugkit.wasm. Atomic-write is sufficient under wasm; there is no running daemon to kill.
  3. Instantiate the wasm module with the Node WASI imports plus the host-fn imports for exec_js, fetch, kv_*, fs_*, browser, etc.
  4. Record bootstrap status under .gm/exec-spool/.bootstrap-status.json for spool awareness; record any error under .bootstrap-error.json.
  5. Return; the spool watcher inside the wasm instance services every subsequent dispatch.

Bootstrap failure is non-fatal. A failed wasm fetch leaves a documented error in .bootstrap-error.json; the skill body degrades to surfacing the error string to the agent rather than to a stack trace.

7. The Verb Surface

Distribution is not just “getting bits onto the user’s disk” — it is also the verb surface that the agent talks to once the bits are there. Every capability the harness exposes is a verb dispatched through the file-spool at .gm/exec-spool/in/<verb>/<N>.<ext>, with output landing at out/<verb>-<N>.json (nested) or out/<N>.json (root). Three families:

familyexamplesserviced by
orchestrator verbsinstruction, transition, phase-status, prd-add, prd-resolve, prd-list, mutable-add, mutable-resolve, mutable-list, memorize-fire, residual-scan, auto-recallrs-plugkit orchestrator (src/orchestrator/)
host verbsfs_read, fs_write, fs_stat, fs_readdir, kv_get, kv_put, kv_query, fetch, exec_js, env_get, recall, codesearch, memorize, health, browserwasm host imports (rs-exec, rs-learn, rs-search, rs-codeinsight)
language verbsnodejs, python, bash, typescript, deno, go, rust, c, cpp, javaexec_js with the matching runtime probe

Adding a verb is a single Rust function plus a row in the dispatcher; the surface to the agent is identical for every verb. No per-platform binary, no per-host adapter, no shape-shifting payload.

The differences are that gm’s artifact is a wasm module (because the user is not compiling against it; the host is loading it), and updates arrive via npm tags rather than a swap of one C file.

9. Trust And Verifiability

A turnkey wasm distribution buys convenience by trusting whoever built the wasm. We make this trust auditable, not implicit:

  • Public build pipeline. Every wasm artifact is built by a GitHub Actions workflow whose YAML is in the same repository as the source. There is no offline build step. plugkit-bin Releases and plugkit-wasm on npm are produced by rs-plugkit CI; the run logs are public.
  • SHA-256 manifest. The skill ships bin/plugkit.wasm.sha256 pinned next to gm-starter/gm.json::plugkitVersion. The bootstrap verifies the wasm against this hash before instantiation; mismatch is fatal-soft (recorded to .bootstrap-error.json rather than thrown into the agent).
  • One target. wasm32-wasip1 is reproducible from Cargo across Linux and macOS runners; mismatch between runs is a CI red, not silent drift.
  • Forkable. Any user who does not trust the published wasm can fork all seven repositories and point their skill at their own fork’s npm package. Nothing in the distribution requires AnEntrypoint specifically.

10. Why Not Just Publish To cargo / pip?

Two reasons:

  1. The host is Node. Every harness that loads a skill loads it inside a Node-capable runtime. Publishing to crates.io or PyPI would force every host to bridge Rust or Python at install time, which is exactly the toolchain dependency the SQLite model exists to avoid. npm is the lowest-common-denominator registry for skill-loading hosts.
  2. The wasm is the binary. A cargo install would require rustup; a pip install would require Python. The npm tarball ships the precompiled wasm and the JS wrapper together — no transpilation, no native build, no postinstall script.

11. Operational Lessons

Three production failures shaped the current pipeline:

  • Native-binary fan-out doubled CI duration linearly with platform count. Building six platform binaries on every push, then signing them, then uploading them, then bumping fifteen downstream repos, took 18–25 minutes per cascade tick. The wasm-only model collapses this to one build, one publish, one npm tag — under four minutes.
  • Stale binaries pinned to dead PIDs. The previous daemonized rs-exec wrote a port file then sometimes died, leaving clients pointing at a port nothing listened on. The wasm model has no port and no daemon — exec_js is a synchronous wasm host call via child_process.spawnSync, alive only for the duration of the call.
  • Concurrent CI publishes raced on downstream branches. Resolved by collapsing fifteen downstream branches into one npm publish.

12. The Invariant

Distribution Invariant. A user installs gm-skill with one npm command and never sees a build step. Every artifact they execute was built by a public CI run from a public Rust source repository under a single GitHub organization, anchored by SHA-256, fetched over HTTPS or npm at first use, cached locally for subsequent use, and replaceable at any time by forking six repositories. There is no proprietary core, no private server, no daemon, no locked-in vendor surface, and no toolchain on the user’s machine.

This is what we mean when we call gm “the SQLite of coding-agent infrastructure.” Not that it is a database; that it has SQLite’s ratio of install-experience-to-openness. One npm install in. Seven repositories out (gm, rs-plugkit, rs-exec, rs-codeinsight, rs-search, rs-learn, plugkit-bin). One wasm in between.
