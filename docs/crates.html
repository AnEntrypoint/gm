tl;dr
gm ships one wasm artifact, plugkit.wasm, assembled from five Rust crates that each target wasm32-wasip1 as a cdylib. Push to any of them and a new gm-skill publishes to npm within minutes. You do not run cargo build. You do not bump versions. You push, the cascade does the rest.

Five crates contribute. One wasm deploys. One npm package updates.

rs-plugkit

the orchestrator AnEntrypoint/rs-plugkit ↑

Job: own the PLAN→EXECUTE→EMIT→VERIFY→COMPLETE state machine, dispatch every verb the agent fires, and serve phase prose on demand via the instruction verb. This is the wasm module that loads into the Node host; the other four crates are its imports.

what it actually does

  • compiles to a single plugkit.wasm targeting wasm32-wasip1; bundled in the gm-plugkit npm package as plugkit-wasm
  • exposes the dispatch_verb wasm export, one entry point, every verb
  • owns the FSM in src/orchestrator/{mod,state,transitions,mutables,memorize}.rs; the agent never advances state directly, it dispatches transition and the orchestrator validates against the four state planes
  • services orchestrator verbs natively: instruction, transition, phase-status, prd-add, prd-resolve, prd-list, mutable-add, mutable-resolve, mutable-list, memorize-fire, residual-scan, auto-recall
  • routes host verbs (fs_*, kv_*, fetch, exec_js, recall, codesearch, memorize, browser, health) to the imported host functions
  • watches .gm/exec-spool/in/<verb>/<N>.txt, writes responses to out/<verb>-<N>.json (nested) or out/<N>.json (root)

commandments it enforces

  • blocked direct Bash for node|npm|npx|bun, use exec_js via the spool
  • redirect find, grep, rg, Glob, Explore → codesearch
  • stop-block while .gm/prd.yml has items, while .gm/mutables.yml has unwitnessed rows, while git is uncommitted or unpushed
  • residual-scan one-shot challenge at COMPLETE before allowing stop

rs-exec

wasm host helpers AnEntrypoint/rs-exec ↑

Job: provide the host-side machinery that plugkit’s wasm imports use to actually touch the world. Code execution, browser sessions, file-system glue, idle cleanup. Formerly a native exec daemon with its own port and rpc; now a wasm-only utility crate published as @anentrypoint/rs-exec-wasm.

what it actually does

  • implements exec_js as a synchronous child_process.spawnSync call from the wasm host, the watcher blocks for the duration; long subprocesses are not supported under wasm
  • spawns the user’s code in nodejs, python, bash, typescript, deno, go, rust, c, cpp, java
  • enforces an explicit timeoutMs per call; missing or zero is a hard error
  • preserves partial streamed output on timeout; emits timedOut: true on the response
  • manages project-scoped browser sessions for the browser verb; idle-cleans dead sessions on a 60-second tick

commandments it enforces

  • mandatory timeoutMs on every exec_js dispatch, the host treats missing or zero as a hard error
  • explicit timedOut field on the response; clients do not string-match the error
  • streamed output preserved through timeout-kill; the output array drains either way

rs-codeinsight

the AST x-ray AnEntrypoint/rs-codeinsight ↑

Job: tree-sitter-based code structure analysis. Powers the symbol layer of codesearch and the session-start architectural overview that lands in the agent’s opening context. Published as @anentrypoint/rs-codeinsight-wasm.

what it actually does

  • walks the project tree, parses every supported language via tree-sitter, builds a structural index
  • computes per-file metrics: lines, functions, classes, average complexity, parameter counts, long functions, large files
  • derives architectural layers (L0 pure exports → L3 pure imports), identifies hub modules and circular dependencies
  • flags issues: complex functions, large files, duplicated groups, dead exports, orphaned files
  • caches results in .codeinsight with a digest of (mtime sum + git HEAD + dirty-tree marker); default invocation always runs fresh, --read-cache permitted only on exact digest match

rs-search

the file finder AnEntrypoint/rs-search ↑

Job: answer codesearch <words>. BM25 plus embedding fusion over file content, scan-embed-sweep pipeline, with snippet windows and tunable context. Published as @anentrypoint/rs-search-wasm.

what it actually does

  • scans the project (respecting .gitignore), tokenizes file content, builds a sparse term-frequency index plus a dense embedding index
  • fuses BM25 with embedding cosine; reranks with a sweep pass; returns top matches with surrounding line context
  • indexes PDFs as code (extracts text via pdf-extract, caches per-page under .gm/code-search/pdf-cache/)
  • emits an [index fully synced: …] line before the first result so freshness is visible
  • tracks the index under .gm/code-search/, committed to git so the index travels with the project

commandments it enforces

  • blocked find, grep, rg, Glob, Explore, the Bash gate denies these in favour of codesearch
  • two words minimum query length to start; iterate by changing or adding one word per pass
  • four attempts before declaring something absent

rs-learn

the memory store AnEntrypoint/rs-learn ↑

Job: hold every fact the agent ever resolved, across every session, on every machine. The recall verb reads from it. The memorize-fire orchestrator verb writes to it. The store is committed to the repo as .gm/rs-learn.db, so memory shares across machines, sessions, CI, and contributors. Published as @anentrypoint/rs-learn-wasm.

what it actually does

  • KV plus vec store backing recall, memorize, and the kv_* host verbs that the orchestrator uses for its own state
  • each episode has a source tag, content, embedding, classification
  • community detection (label propagation) groups related episodes; LLM summarization synthesizes per-community context
  • graph search via Searcher with configurable result count plus relevance threshold
  • fast-path ingest avoids re-embedding when content is identical
  • called directly from the orchestrator’s wasm imports, no rpc, no daemon, no bun x rs-learn shim

commandments it enforces

  • tracked .gm/rs-learn.db is committed, never gitignored, memory must travel
  • classified facts route to AGENTS.md (top-level rules) or rs-learn (everything else); the classifier rejects changelog-shaped facts from AGENTS.md ingestion
  • isolated per-discipline stores at .gm/disciplines/<name>/rs-learn.db; the @<name> sigil in a request scopes reads and writes

How to update any of them

Push to the relevant repo. That’s it. No cargo update, no local cargo build, no version bumps. The sibling repos (rs-exec, rs-codeinsight, rs-search, rs-learn) publish their own @anentrypoint/<name>-wasm artifacts independently, then cascade.yml uses gh workflow run to fire rs-plugkit’s release.yml. rs-plugkit rebuilds its own wasm, publishes it, and bumps gm-starter/gm.json::plugkitVersion plus bin/plugkit.wasm.sha256 in this repo. The bump commit triggers publish.yml, which runs node cli.js gm-starter ./build and npm publish build/gm-skill. The next bun add -g gm-skill picks up the change.

The PUBLISHER_TOKEN secret in rs-exec, rs-codeinsight, rs-search, and rs-learn authorizes their cascade.yml to trigger rs-plugkit’s release.yml via workflow_dispatch. If the cascade ever stops working, that token has rotated. Set it with gh secret set PUBLISHER_TOKEN --repo AnEntrypoint/<repo>.

notes on the architecture

"Five crates is two more than necessary and three fewer than tempting. The temptation is to fold rs-search and rs-codeinsight into one, they both walk the tree. The reason not to: codeinsight runs once per session, search runs hundreds of times. Different latency budget, different cache shape, different release cadence. Sit down. Don’t merge them.", from the gm field guide
