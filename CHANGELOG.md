## 2026-07-12 - fix exec_js default path: top-level `return` was a SyntaxError; wrap in async IIFE + capture result

Fix the `host_exec_js` default (non-mem, non-profile) path in `gm-plugkit/plugkit-wasm-wrapper.js` so `exec_js {code:"return 2+2;"}` runs. Root cause (live-witnessed): the default path did `spawnSync(process.execPath, ['-e', code])`, injecting the user's code verbatim into `node/bun -e` where a top-level `return` -- the documented gm exec_js contract ("return the result") -- is a SyntaxError (`Return statements are only valid inside functions`), so every `return`-bearing dispatch failed with exit 1 (or, on an older stale watcher, a spurious SIGTERM read as `timed_out`). The `mem`/`profile`/`browser` paths already wrapped user code in an async IIFE and captured the value; the default path was the sole outlier that did not. Fix: wrap the default-path code in `(async () => { try { const __r = await (async () => {<code>})(); console.log('__GM_RESULT__'+JSON.stringify(__r)); } catch (__e) { console.error(__e.stack); process.exitCode=1; } })()` -- so top-level `return` and top-level `await` both work, the returned value is surfaced as a `result` field via a `__GM_RESULT__` sentinel (stripped from the caller's stdout, matching the browser verb's mechanism), and a throw still lands its stack on stderr with exit 1 (documented error channel preserved). Two hardenings rode along, shared across all three branches: (1) `timed_out` now requires `signal==='SIGTERM' && duration >= 90% of timeoutMs`, so a spurious sub-deadline kill (the old SIGTERM-at-28ms) is no longer mislabeled a timeout; (2) `spawnSync`'s `result.error` (spawn race / ETIMEDOUT / EBADF / ENOENT) is surfaced as a `spawn_error` field so a `status:-1` is diagnosable instead of a silent empty failure. Witnessed live through the webix `.gm/exec-spool`: `return 2+2` -> `{ok:true,exit_code:0,timed_out:false,result:4}`; `console.log("side output"); return {sum:42}` -> `{ok:true,stdout:"side output",result:{sum:42}}`; `throw new Error("kaboom")` -> `{ok:false,exit_code:1,stderr:"Error: kaboom...",timed_out:false}`; top-level `await` -> `result:42`. Also verified standalone on both node v24.15.0 and bun 1.3.8 across return/log+return/await/throw/undefined/bignum. Source is what ships; the installed copy at `~/.gm-tools/plugkit-wasm-wrapper.js` was resynced and the watcher rebooted onto it (sha-confirmed) for the live witness.

Diagnosing the above surfaced a second, separate defect that made the spool nearly unusable on a large repo: an unbounded watcher restart loop. Root cause -- the boot warmup at watcher start runs a synchronous `codesearch` dispatch that, on an absent/stale codeinsight digest, triggers a full in-wasm embed of the whole project. That embed is a blocking wasm call, so the 5s heartbeat `setInterval` cannot fire while it runs and the heartbeat freezes. The warmup only reserved a 360s `busy_until` window; on a large repo the cold embed exceeds it, so `busy_until` expires while the loop is still blocked, the supervisor's 30s stale-heartbeat check kills the watcher mid-embed, the new digest never persists, and every respawn re-embeds from scratch -- spawning duplicate watchers that contend on the shared libsql index and never converge. Fix: raise the boot-warmup `busy_until` window from 360000 to 1200000 ms so it comfortably exceeds a worst-case cold embed. The supervisor already returns early from its health check while `busy_until > now`, so the longer window is sufficient to let one embed complete; it is a one-time cost because the digest persists after a single completion (subsequent boots skip the reindex, so their warmup is near-instant). Both fixes ship in the same `plugkit-wasm-wrapper.js` change.
