CONSOLIDATE-phase CI-churn root cause: rs-plugkit's handle_instruction() wrote .gm/next-step.md unconditionally on EVERY `instruction` dispatch (fresh Updated: timestamp stamped even when phase+instruction text were unchanged), so ANY subsequent git_finalize in the same session saw a dirty file and produced a real commit+push, retriggering CI in repos without a .gm/** path-ignore -- a session could burn N commit/push/CI-wait cycles purely from repeated instruction dispatches between the real code push and CONSOLIDATE's own push. Fix: compare existing on-disk next-step.md's instruction body (split on the doc's own separator) and Phase: line against the freshly-computed phase+instruction before writing; skip the write when both match, so only a genuine phase transition or instruction-text change touches disk. Chose this over seeding a .gitattributes/CI-paths-ignore template into driven repos: gitattributes does not affect GitHub Actions triggering at all (needs workflow-level paths-ignore, unsafe to auto-edit into arbitrary driven-repo CI configs), whereas eliminating the spurious write is correct unconditionally regardless of any given repo's CI setup and is strictly the smaller surface. Verified via a throwaway cargo-run scratch binary (not a test file) exercising the exact should-write predicate against adversarial cases: empty-disk, overflow-size unchanged content, malformed/degenerate existing content (fails open to write, never panics/gets stuck), concurrent-reader agreement, phase-name-prefix boundary (EXECUTE vs EXECUTE_EXTRA), and instruction-text containing the separator substring itself -- all as expected, then deleted the scratch binary.
