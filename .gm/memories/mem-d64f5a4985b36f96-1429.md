embed.rs architecture (rs-plugkit): gemm::set_wasm_simd128(true) is force-enabled unconditionally in init_ctx's wasm-model-load path because gemm's wasm32 SIMD dispatch (gemm_common::get_wasm_simd128()) is gated by a runtime AtomicBool that defaults FALSE regardless of whether the binary was compiled with -C target-feature=+simd128 -- the compile flag only makes simd128 microkernels buildable, it does not select them at runtime. Without this call every matmul in candle's BertModel::forward silently falls back to gemm's scalar path, live-witnessed as a ~10-14 second single-chunk embed floor (consistent with scalar BERT-small matmul FLOP cost at realistic wasm32 throughput). Safe to force-enable unconditionally since every real host loading this wasm module (gm-plugkit's JS wrapper via bun/Node WebAssembly.Instance, or agentplug-runner via wasmtime) runs on a wasm engine with simd128 support, a baseline feature of both V8 and wasmtime since ~2021. A live diagnostic (gemm_simd128_flag_check event, reading back gemm::get_wasm_simd128()) was added because setting the flag did NOT produce the expected embed-speed improvement in one session (still ~10-27s/chunk post-fix) -- the read-back check rules out a silently-no-op call before investigating further upstream (whether gemm's own simd128 microkernel path is reachable at all on that exact wasm engine). This remains an open investigation thread, not a closed fix.
