iter46 (2026-05-26, 0.1.582 live): FIXED the recurring watcher-fresh-boot-death (the top operational fragility that hit every version transition across iter25-45). ROOT CAUSE: plugkit-wasm-wrapper.js is a NODE-targeted watcher (wasm host via child_process.spawnSync, Node fs/event-loop semantics), but the daemon was spawned under process.execPath which is BUN.exe when invoked via bun x, and supervisor.js defaulted PLUGKIT_RUNTIME to bun. Running the wrapper under bun makes it report running (writes the daemon-ok status) then DIE on the first heartbeat (status.json stays empty, pid gone) - which is exactly why only the manual direct-node `node ~/.gm-tools/plugkit-wasm-wrapper.js spool` recovery survived every fire. FIX (gm-plugkit): bootstrap.js resolveNodeRuntime() (candidates: node-named execPath, GM_NODE_PATH, where/which node; validates each via --version status 0; falls back to execPath) used for BOTH the daemon runtime AND the supervisor spawn (+ passes PLUGKIT_RUNTIME=node); supervisor.js resolveNode() ignores bun and resolves a real node for the wrapper. So the long-lived watcher always runs under Node regardless of `bun x` being the bootstrap invoker. node --check passes both files. node --check passed; ships via publish.yml on next gm version bump. This should END the manual-direct-node-boot workaround that every prior version transition required. LESSON: bun x is the right fast bootstrap INVOKER but a Node-API-dependent long-lived process must be spawned under node, not process.execPath (which is bun under bun x) - resolve the runtime by capability, not by inheriting the launcher.
