orchestrator/instructions/mod.rs architecture (rs-plugkit): compiled_default_for_prose_key is used ONLY as prose::resolve's fallback when neither a .gm/ override file NOR (for a custom phase) any prose_key entry applies -- a phase name the active graph doesn't recognize at all still falls back to entry::TEXT. get_instruction looks up the current phase's prose_key in the ACTIVE graph (built-in default, or a project's .gm/instructions/fsm/graph.json override) rather than a fixed match -- a custom phase name (e.g. a project-defined REVIEW between EMIT and VERIFY) resolves through its own graph-declared prose_key, served from .gm/instructions/<prose_key>.md the same way the six built-ins always have. ENTRY/ORCHESTRATOR/BROWSER stay as direct pseudo-phase pass-throughs (not real FSM states a project's graph declares). next_phase_hint's terminal-state suppression: a terminal state's default edge (if the graph declares a self-loop, matching the built-in COMPLETE->COMPLETE) is not a genuine "next" hint -- suppressed the same way the old hardcoded COMPLETE=>None arm did.
