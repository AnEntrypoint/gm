wasm_dispatch/verbs.rs history (rs-plugkit): db_path_from/agentplug-libsql statelessness -- agentplug-libsql is now stateless and process-wide shared across every concurrently active project; open/close are accepted-but-inert no-ops plugin-side, and every exec/query/serialize/deserialize call is keyed PURELY off the JSON body's path field (defaulting to :memory:, silently throwaway, if absent) -- the plugin never looks at db. There is no remembered connection carrying a path forward from an earlier sql_open call to a later sql_exec/sql_query; every sql_* verb requires or defaults an explicit path on EVERY call, not just sql_open. Falls back to project_db_path(None) when the caller's body omits path entirely, so an old caller expecting later calls to inherit an earlier sql_open path is instead routed to the default project db rather than a throwaway :memory: -- a real behavior change forced by the plugin's statelessness, but strictly safer than the silent :memory: default alternative.

sql_smoke(): the probe originally passed {"db": n} with no path on every call after the first under stateless semantics, so each step silently defaulted to :memory: and got a FRESH EMPTY database each time -- which is why create_table reported success and the very next statement returned "no such table: memos". Fixed by threading a real absolute file path through every step. Path MUST be absolute: libsql's only WASI preopen is the drive root mapped to "/" (no "." preopen, WASI has no cwd), so a relative path like ".gm/x.db" resolves to "C:\.gm\x.db" which does not exist, failing sqlite3_open_v2 rc=14 "unable to open database file" while the identical statement on an absolute path succeeds -- resolved via crate::libsql_wasm::absolute_db_path, same as code_index.rs. sql_smoke's "ok" field was previously hardcoded true, so the designated health probe for the whole sql subsystem could never fail -- it reported ok:true while its own steps returned "no such table: memos", reading as "libsql is broken" and costing a full investigation to disprove against a database that was in fact healthy. A probe that always passes is worse than no probe -- now computed from whether any step actually failed.
