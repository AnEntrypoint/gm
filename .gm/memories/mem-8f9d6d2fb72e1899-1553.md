agentplug-host exec_js.rs/task.rs time-sliced background handoff: an exec_js call that hits its declared timeoutMs still running is NOT killed (the old behavior, which discarded a job that might have been one dispatch away from finishing) -- it is handed off ALIVE into task.rs's background process registry (adopt_running, the same registry host_task_proc's spawn() populates) and the call returns immediately with {in_progress:true, task_id, decision_required: <explanation>}. The calling agent explicitly decides next via task-output {id} (keep it running, poll later, the worker queue is NOT blocked on it) or task-stop {id} (kill it) -- never silently continues unattended forever, and never silently discards a near-complete job by hard-killing on the very call that names the decision point. Live-hit motivating this: a 150s+ cargo check inside exec_js previously left the agent with zero visibility and zero choice until it finally returned. adopt_running's started_ms is computed by subtracting the child's already-elapsed wall time from now (not simply 'now') so timeout bookkeeping in poll_entry still measures from the child's TRUE spawn instant, not from the moment of hand-off into the registry. Separately, resolve_bash_cmd() explicitly prefers Git Bash's real bash.exe (C:\\Program Files\\Git\\bin\\bash.exe, falling back to \\usr\\bin\\bash.exe) over a bare 'which bash' lookup on Windows, because System32\\bash.exe is the WSL launcher stub, not a real POSIX shell -- it either hangs waiting on a WSL distro or behaves unlike Git Bash.
