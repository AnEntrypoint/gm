agentplug-host browser.rs OS-orphan reaping + wedged-renderer detection: reap_os_orphans closes a real gap where a crash/SIGKILL/TerminateProcess/OOM-kill leaves a live Chrome child running with NO chance for close_all_sessions/kill_session to run first -- the in-memory SESSIONS registry that tracked it dies with the owning process, so a restarted daemon starts with an empty registry and no way to know chrome.exe is still alive holding the CDP port and profile-dir lock. Mechanism: every launched Chrome writes its pid to <profile_dir>/chrome.pid at launch, removed on any clean kill_session; on every run() call, scan this cwd's .gm/browser-chrome-profile-* dirs, and for any dir NOT claimed by a live SESSIONS entry in this process, read the sidecar pid and check real OS liveness (tasklist/kill -0) -- a live unclaimed pid is a genuine orphan, killed directly. A 15s grace period keyed off the sidecar file's own mtime (not a wall-clock payload) protects against false-flagging a Chrome that's mid-launch in ANOTHER process (e.g. a self-update handoff briefly overlapping old+new daemon both scanning .gm/) as an orphan before it's had a chance to get claimed. Separately: an eval timeout used to kill ONLY the transient node cdp-eval helper process, leaving the actual Chrome registered and reused on the next dispatch even when the real cause was a wedged renderer -- a hung tab still answers the CDP /json/version HTTP probe (that's Chrome's always-alive browser process, not the wedged renderer), so a single re-poll can't distinguish 'helper was slow' from 'renderer wedged'. Fixed: a timed-out eval triggers ONE bounded re-check (session_liveness_recheck) against the SAME page context via a trivial `1+1` Runtime.evaluate with a short deadline -- if that also fails to answer, the session is genuinely wedged and gets killed+evicted so the next dispatch launches fresh Chrome instead of reusing a corpse that will time out on every future eval too; a responsive re-check means the timeout was just that one script (e.g. an intentional infinite-pending promise) and the Chrome process is left alone.
