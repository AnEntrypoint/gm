agentplug-host imports.rs host_git subprocess bound + relative-cwd bug + host_kv_query key-shape bug: (1) GIT_SUBPROCESS_TIMEOUT_MS=15s bounds every host_git call via wait_timeout+kill-on-expiry -- an unbounded git subprocess (pack-refs lock contention, AV interference on Windows holding a file handle, a pathological repo) previously could block the ENTIRE dispatch forever inside a wasm host-import call while that plugin's pool slot is held, wedging every other request for the same project (and, for a size-1 shared plugin, every OTHER project) behind it; code_index.rs's own INDEX_WALL_BUDGET_MS only checks BETWEEN files in its indexing loop so it never gets a chance to fire while a single host_git call is stuck. Git commands here are always read-only/local (log/status/diff/rev-parse) so a killed-and-abandoned child leaves no repo-state cleanup needed. (2) A relative cwd_arg (e.g. a bare submodule name like 'agentplug', the shape gm's own git verb docs use) previously resolved against the DAEMON PROCESS's own cwd rather than the calling project's root, since PathBuf::from() never joins a relative path against anything -- produced OS error 267 (invalid directory) on Windows whenever the daemon's own cwd differed from the calling project (always true for the shared daemon). Fixed: join relative cwd_arg against caller.data().cwd() when not already absolute. (3) host_kv_query previously returned the bare content string per matched file instead of a {key,value} object -- every guest-side reader (code_index::load_manifests and every other kv_query consumer) indexes each row by key/value fields, so a fully-populated namespace silently read back as empty (114 valid manifests loaded as 0 chunks, dropping codesearch to mode:fallback_kv with no hits) until fixed to emit {key: filename_stem, value: content}.
