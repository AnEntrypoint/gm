orchestrator/instructions/mod.rs architecture (rs-plugkit): expire_stale_marker exists because staleness markers describe a condition true when some watcher wrote them, and nothing rewrites/clears them once that condition resolves -- a marker left by a since-retired watcher reports its warning on every dispatch forever. Live-hit case: a .gm-plugkit-stale.json written by the old JS watcher kept every instruction response returning gm_plugkit_stale:true naming old versions, long after the native runtime took over and npm had moved on. Drops any marker older than a 6-hour window; a genuinely current condition is re-reported by whatever watcher is actually running. iso8601_to_ms is a minimal ISO8601-to-epoch-ms parser via the standard days-from-civil algorithm, returns None on anything unparseable so an unrecognized timestamp leaves its marker untouched rather than silently expiring it. write_turn_summary exists because .turn-summary.json is what SKILL.md's documented boot probe reads at session start (phase/prd_pending/last_instruction_age_ms) -- it was written ONLY by the retired JS wrapper's runSpoolWatcher, so once the native runtime took over nothing rewrote it and it froze, live-hit as a file reporting stale phase/prd_pending from a retired watcher while a real dispatch returned different values. Every session was starting from fabricated state, only recovering because the first instruction dispatch overrode it. Now written directly from the same authoritative values the response itself carries, so it cannot drift.
