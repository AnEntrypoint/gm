orchestrator/transitions.rs architecture (rs-plugkit): gate_rejection/gate_residuals missing-edge bypass bug -- a no-declared-edge from->to is NOT gate-free, it is the strictest possible rejection (no legal path exists at all). An earlier version used `?` on graph.edge_between, which returned None ("no rejection" == allow) on a missing edge, so `transition {to:"COMPLETE"}` from ANY phase -- including straight out of EXECUTE with 8 PRD rows still pending -- sailed through ungated because no EXECUTE->COMPLETE edge exists to attach gates to in the first place. Live-witnessed this exact bypass (phase flipped to COMPLETE with prd_pending_count=8, transition-1784612100001.json). Fixed: missing edge now returns the maximal-residual rejection/message instead of None/empty-list (an empty residuals list reads as "nothing blocking" to gates.rs's caller, which then falls through to ALLOW). known_predicates() is kept as the single source fsm-vendor's reference file generates from, so the vendored doc can never silently drift out of sync with what predicate_result actually recognizes. Every predicate fails CLOSED on an unrecognized name -- a typo'd or stale predicate name in a hand-edited graph must never silently skip a real check. hook_result: a jit-hook script's final expression value is coerced to bool -- anything that isn't exactly JSON true (thrown error, non-boolean return, missing/unreadable file) is treated as FALSE (gate denies), matching predicate_result's fail-closed default for the same reason.
