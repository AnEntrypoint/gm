SUPERVISOR DUPLICATE-SPAWN CHURN ROOT-CAUSED + FIXED (2026-06-02 ~11:38). The watcher churn (4-8 plugkit procs, all fresh 0-1min, pids rotating every fire) I half-dismissed last fire as self-inflicted is a REAL bug: gm-plugkit/supervisor.js (the running variant) writes only .supervisor.json, never .supervisor.pid, and has NO single-instance guard. findSupervisorPid (lib/skill-bootstrap.js:577) reads .supervisor.pid -> missing -> null -> every gm-skill bootstrap (every audit fire + every skill load) spawns a DUPLICATE supervisor parented to claude.exe, each spawning watchers that lock-fight. bin/plugkit-supervisor.js already had SUPERVISOR_PID_PATH + acquireSingleInstance/releaseSingleInstance; the gm-plugkit variant did not - a copy-drift between two supervisor implementations. FIX (committed): ported the single-instance guard into gm-plugkit/supervisor.js (write .supervisor.pid on startup, refuse-if-live-peer-holds-it, release on SIGTERM+exit). node --check OK, ships via gm-plugkit publish. CORRECTS my prior-fire premise: last fire I concluded no-defect/self-inflicted because I only read the EXIT handling (which IS correct - lock-rejected exits cleanly) but did NOT check the STARTUP duplicate-guard, which was the actual hole. LESSON: when churn keeps recurring across fires despite a clean exit path, the bug is in ENTRY/spawn-gating not exit; and two divergent implementations of the same component (bin vs gm-plugkit supervisor) is a copy-drift smell - the feature in one must be in both. The spool stayed functional throughout (one watcher always won the lock and served). Once the published fix deploys via bootstrap, findSupervisorPid finds the live supervisor and duplicate spawns stop.
