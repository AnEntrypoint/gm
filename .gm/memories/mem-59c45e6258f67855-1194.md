orchestrator/fsm.rs architecture (rs-plugkit): StateNode.key is the canonical uppercase phase identifier matching Phase::as_str(); prose_key is what instructions::get_instruction passes to prose::resolve (compiled-in phase prose files keep existing lowercase keys plan/execute/emit/verify/consolidate/update_docs/browser for backward compat, so a project's custom phase names its own prose_key freely). Edge.gates names zero or more GateDef.name entries that must ALL pass before the edge may be taken, in list order -- first failure's message is what the caller sees, matching the pre-dynamic-phase hardcoded gates.rs sequencing (residual-scan-fired before prd-open etc). GateDef.hook is an optional path to a jit-executor script (relative to .gm/instructions/hooks/) the orchestrator runs via exec_js instead of (hook-only) or in addition to (both) the built-in predicate -- letting a project "concrete" its own custom condition without a Rust rebuild. A hook script's return value (explicit return required -- exec_js wraps every script in an async function body so a bare trailing statement is discarded) is coerced to bool; non-boolean/missing-return/throw fails CLOSED (deny), never open.
