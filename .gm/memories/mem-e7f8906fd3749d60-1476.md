agentplug-host registry.rs libsql host_fs_root WASI quirk + fairness guard: libsql is instantiated via HostState::new_with_fs_root (not the plain ::new every other shared plugin uses) because libsql-ffi's wasm32-wasi-vfs feature makes sqlite3_open_v2 issue REAL WASI path_open syscalls -- unlike bert/treesitter/gm (which only touch files through host_fs_* imports that consult HostState.cwd fresh per call), a single project-cwd WASI preopen fixed at first instantiation would silently CANTOPEN every other project's absolute db path. Fix: preopen the WHOLE DRIVE (Windows: the drive letter of current_dir, e.g. C:\\; Unix: /) as WASI guest path '/' once, covering every caller's absolute path regardless of which project instantiated the Store first -- since all projects this host drives live under one drive/root in practice. Separately, GmFairnessGuard is a per-project RAII concurrency cap read fresh from <project>/.gm/daemon-project-config.json's gm_concurrency_limit on every dispatch (deliberately distinct from the machine-wide ~/.agentplug/daemon-config.json gm_concurrency, which sets the actual total pool size) -- this file can only ever LOWER one project's share of the shared pool (no 'total slots' field exists here), and the slot-release runs in Drop specifically so a panic unwinding through catch_unwind in dispatch_project still decrements the counter (Drop runs during unwind), never permanently wedging a project at its own cap from an unreleased slot.
