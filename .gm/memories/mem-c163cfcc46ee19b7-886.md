orchestrator/fsm.rs architecture (rs-plugkit): graph()'s active-graph resolution -- a project's .gm/instructions/fsm/graph.json REPLACES the built-in wholesale if present, NOT a per-field merge, since edges and gates are interdependent (a partial override risks referencing a gate name or state that doesn't exist elsewhere in a half-overridden graph); the fsm-vendor scaffold verb's own default output is the safe starting point for customization, not a diff against the compiled-in one. Falls back to default_graph() when absent, unreadable, or fails to parse -- a malformed override graph is treated as no-override rather than a hard error, since gm's own FSM dispatch loop must keep functioning even if a project's customization attempt has a typo, but the malformed-graph condition itself is worth surfacing so a parse failure emits an fsm_graph_override_malformed deviation event.
