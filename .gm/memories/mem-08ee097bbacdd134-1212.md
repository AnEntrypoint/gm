orchestrator/fsm.rs architecture (rs-plugkit): Policy struct holds project-level policy knobs previously hardcoded Rust consts in gates.rs (TOPLEVEL_DOC_ALLOWLIST, AWAIT_ALLOWED_VERBS, LONGGAP_EXEMPT, GATE_REPEAT_ESCALATE_THRESHOLD, the 300000ms long-gap threshold) -- each is a project-policy decision, not a code invariant, so it belongs in the vendored/overridable graph, not a rebuild-to-change const. Every field has #[serde(default="...")] so an OLDER vendored graph.json (written before a field existed) still deserializes fine, falling back to the values gates.rs used to hardcode. initial_phase/terminal_phase fields: the fresh-prompt-reset-to-initial-phase logic in instructions/mod.rs (handle_instruction) previously compared live phase against the literal strings "PLAN"/"COMPLETE" directly, bypassing the graph entirely for a project naming its initial/terminal phases something else via a custom graph.json override -- the exact "hardcoded path silently pre-empts the graph" bug class a prior CONSOLIDATE/COMPLETE gate-bypass incident already hit once for a different check. Made graph-policy-overridable additively (old overrides without these fields default to the pre-existing literal behavior).
