libsql_wasm.rs architecture (rs-plugkit): agentplug-libsql is stateless and shared as ONE process-wide instance across every concurrently active project -- every call opens the db fresh, does its one operation, and closes it before returning. open/close/begin/commit/rollback/finalize are accepted-but-inert no-ops on the plugin side now; there is no persistent connection to remember, so a bare name (formerly a lookup key into a name->connection map) means nothing anymore. Every function takes a real absolute path and forwards it in the JSON body's path field -- the plugin defaults to :memory: (silently throwaway) when path is absent. absolute_db_path resolves <host_cwd>/.gm/<filename> fresh on every call -- host_cwd is asked of the host per-dispatch (never cached wasm-side), since the same wasm instance may be serving a different project's dispatch on the very next call; returns filename unchanged if host_cwd is unavailable rather than fabricating a bad path. begin/commit/rollback are kept as no-op-forwarding calls rather than removed outright so existing call sites (code_index::index()'s batch loop) don't need every transaction-boundary call site ripped out in the same change -- each row in a batch now commits independently regardless of whether begin/commit wrap it. prepare_execute replaces the old two-step prepare()->PreparedStmt::execute_bound() API since a prepared statement handle is inherently incompatible with per-call open-operate-close statelessness -- the plugin collapsed it into one prepare_execute verb that prepares/binds/steps/finalizes atomically in one call; callers doing a prepare-once/execute-many bulk-insert loop now call this once per row instead of once per loop, paying one full cycle per row rather than amortizing prepare across the loop -- a real deliberate cost accepted in exchange for zero persistent state.
