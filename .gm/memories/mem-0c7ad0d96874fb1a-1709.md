BROWSER STARTUP EXTRA-WINDOW FIXED (2026-06-03 ~21:57). USER REPORT: an extra browser window pops up at startup and is never used - it opens an extra one to get started, then real work happens in another. ROOT CAUSE confirmed via live CDP witness: startManagedBrowser (plugkit-wasm-wrapper.js ~1031) launched playwriter chromium with --remote-debugging-port but WITHOUT --no-startup-window and no initial URL, so non-headless chromium opens its DEFAULT chrome://newtab/ window on launch (CDP /json/list showed the only page target = chrome://newtab/ plus a one-google-bar iframe + service_worker). That newtab window is the visible extra unused window; playwriter session new --direct attaches to CDP and creates its own transient page per eval, leaving the startup newtab orphaned/visible. FIX: in startManagedBrowser, when NOT headless, push --no-startup-window (headless keeps --headless=new which has no window anyway). VALIDATED LIVE THE PLAYWRITER WAY: launched the playwriter chromium binary (ms-playwright/chromium-1223 chrome.exe) with --no-startup-window on a test profile -> CDP /json/version responds, /json/list shows 0 page targets (NO startup newtab), browser process stays alive (7 procs); playwriter session new --direct succeeds (Session created), playwriter -s <id> -e "1+1" returns 2 creating a transient page that is cleaned up (0 lingering pages after). So with --no-startup-window: zero startup window, playwriter creates exactly the one page it uses per eval, no extra unused window ever. node --check OK. Goes live after publish (supervisor reverts local .gm-tools edits, must cascade-publish). Distinct from and complementary to the earlier duplicate-on-resume reaper fix (9d18fcc6).
