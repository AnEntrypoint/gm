orchestrator/fsm.rs architecture (rs-plugkit): Graph::default_edge_from returns the single outbound edge for a phase under the linear-chain semantics (next_phase/next_skill both assume exactly one forward edge per phase, no branching) -- a project's custom graph CAN define multiple outbound edges from one state (branching), but a bare `transition` (no explicit to) always takes the FIRST edge listed for the current phase, matching next_phase's old deterministic single-successor behavior; an explicit `transition {to:"X"}` bypasses this and is validated against edge_between instead. default_graph() is hand-transcribed from the pre-existing hardcoded transitions.rs next_phase/next_skill and gates.rs's CONSOLIDATE/COMPLETE gate checks, so a project with no override behaves byte-identically to the pre-dynamic-phase behavior; update_docs is COMPLETE's prose key (pre-existing, not-a-typo naming). The EXECUTE/EMIT/VERIFY -> PLAN re-plan edges are gate-free by design -- prose already documented "transition to=PLAN always legal from EXECUTE" for a reshaping discovery (scope/approach/dependency-shape change to an existing PRD row), but the linear chain had no backward edges at all until this fix, so that documented behavior silently relied on a gate_rejection/gate_residuals missing-edge bug (treating "no edge" as "no gates" == allow) -- once that bug closed, a real to=PLAN dispatch from EXECUTE/EMIT/VERIFY correctly denied with "no edge in the active FSM graph" (live-witnessed in a session). These edges make the documented behavior real instead of an accidental side effect of a bug. COMPLETE has no default forward edge, matching next_phase's old Phase::Complete => Phase::Complete self-loop (terminal, bare transition with no target is a no-op there).
