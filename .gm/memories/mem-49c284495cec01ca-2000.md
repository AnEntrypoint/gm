agentplug-runner download.rs plugin asset resolution: 'gm' is special-cased in plugin_asset_spec rather than following the generic {name}.wasm convention every other plugin (bert/libsql/treesitter) uses, because gm is not yet a real agentplug-native plugin release -- there is no AnEntrypoint/agentplug-gm-bin repo; the actual gm wasm is built by rs-plugkit's own long-standing cascade, asset name plugkit.wasm (not gm.wasm), shipping from AnEntrypoint/plugkit-bin. Routing 'gm' through the generic convention 404s permanently -- live-witnessed as 'plugin gm not loaded' on every dispatch once the daemon tried to auto-serve gm's own spool. Separately: gm_asset_basename() returns 'plugkit-slim' not 'plugkit' -- agentplug-runner always loads bert as one of its 4 default plugins, and host_vec_embed genuinely routes to that shared bert instance, so gm.wasm's own embed.rs::init_ctx()/probe_host_embed() always succeeds under agentplug, meaning gm.wasm's baked-in bert weights (embed.rs's include_bytes! of a 133MB bge-small-en-v1.5.safetensors) are provably dead data -- never deserialized into candle tensors, but still copied into the wasm instance's linear memory as a static data segment at instantiate time. plugkit-slim.wasm (same release, ~3MB, no baked-in weights) is the fix, mirroring gm's own JS wrapper's existing hasNativeEmbedRunner slim-when-a-real-embed-answerer-exists logic, which agentplug-runner needs its own equivalent of since it never routes through that JS bootstrap. A 404 on the slim sha256 sidecar (for a pinned version tag predating plugkit-slim.wasm shipping alongside plugkit.wasm, which started at v0.1.915) falls back to the fat asset_basename rather than hard-failing the whole plugin install. The runner EXECUTABLE's own self-update (stage_runner_self_update) never overwrites the running exe directly -- Windows refuses to write to its own currently-mapped executable file (a hard OS-level lock, not a permissions issue) -- so it always stages to <exe>.new first.
