gm-plugkit self-stale auto-update reliability fix (2026-05-29, wrapper edit shipping via gm-plugkit npm): the watcher's probeGmPlugkitSelfStale IS wired (setTimeout 5s + setInterval 300s) and DOES compare installed gm-plugkit.version vs registry.npmjs.org/gm-plugkit/latest, self-respawning via bun x gm-plugkit@<exact-latest> spool on drift. The bug was that the self-stale respawn exited the old watcher blindly (setTimeout(process.exit,2000)) WITHOUT waiting for the replacement's fresh heartbeat - unlike the version-drift path which polls STATUS_PATH for a fresh-ts-from-different-pid up to 90s. So a slow gm-plugkit download let the supervisor relaunch the SAME stale version before the new one landed, and wrapper-only fixes sat published-but-unloaded (witnessed: installed 2.0.1499 vs npm 2.0.1502, uptime 10.9M). FIX: the self-stale respawn now mirrors the version-drift wait-for-heartbeat (poll .status.json for fresh<15s + different pid, 90s deadline, fallback exit). LESSON: when two code paths do the same thing (respawn-and-exit), they must share the same wait-for-replacement discipline; an asymmetry where one waits and the other exits blindly is a latent reliability bug. Also confirmed: a wrapper-only fix's deploy is gated on the self-stale probe firing AND the respawn actually advancing the on-disk version; a fresh manual bun x gm-plugkit@latest spool boot always deploys the latest wrapper immediately (the reliable manual path).
