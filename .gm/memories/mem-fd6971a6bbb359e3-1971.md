## gm-method: constraints.md hot-swap propagates via two surfaces

Replacing the project-level `.gm/constraints.md` standing arbiter requires updating BOTH the live per-project file (read every phase) AND the bundled `gm-plugkit/constraints-default.md` seed template (only consulted when `.gm/constraints.md` is absent, per the no-clobber seed-if-absent contract test.js's checkConstraintsMdSeedAndIdempotency guards). Missing the second means fresh installs/repos still seed the old policy even after the live file is updated everywhere else.

When a new constraint spec targets multiple languages/paradigms (e.g. a formal-logic spec with clauses aimed at dependent-typed/affine-state languages, applied across a mixed Rust+JS codebase), apply clauses by SPIRIT not literal mechanical checking where the target language's type system doesn't map 1:1 -- e.g. a JS codebase can't be literally dependent-typed, but the underlying intent (no `any`-shaped implicit state, no hallucinated/untyped data flow) still translates into concrete review criteria (explicit checked state, avoid untyped catch-all objects). A rigid audit subagent given a formal spec with unfamiliar notation should be told explicitly to interpret aspirationally where mechanical translation doesn't exist, otherwise it either stalls or produces noise trying to force-fit.

Fan-out constraint-audit-plus-cleanup across N sibling repos after a fresh policy replacement is a legitimate immediate re-audit even hours after a prior full audit-and-fix pass on the same repos: a NEW constraint lens (e.g. explicit STATE/ERR/PERF clauses on panics, unsafe soundness, dead code) surfaces different findings than a general destructive audit did, because the audit questions asked are different. Re-verification of a just-landed critical fix (e.g. a memory-leak fix) under a stricter new lens is not redundant busywork -- it is exactly the kind of adversarial re-check a severity-graded constraint spec calls for.
