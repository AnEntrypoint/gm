FIX (this fire): atomic fs.openSync(path, wx) O_EXCL exclusive-create in BOTH gm-plugkit/supervisor.js + bin/plugkit-supervisor.js - exactly one of N concurrent spawns wins, rest EEXIST->refuse/take-over. node --check both OK. METHODOLOGY LESSON (the meta-lesson of 3 mis-diagnoses): a single-instance/lock guard MUST be atomic (O_EXCL / atomic rename / real lock), never check-then-act - a check-then-write guard silently fails under concurrency and looks like it works under sequential testing. When a guard is added and the duplicate STILL happens, suspect the guard is non-atomic before suspecting it is absent. Also: the bootstrap spawns a BURST of supervisors per skill-load (4 at once), which only an atomic guard neutralizes.
