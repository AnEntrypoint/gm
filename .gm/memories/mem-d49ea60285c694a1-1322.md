BROWSER START-URL affordance (gm-method, plugkit-wasm-wrapper.js host_browser_exec): the browser session launches on about:blank, which confused agents into thinking the tool was broken before they page.goto'd. Added two body shapes so a dispatch lands on the target page off the bat: a bare 'https://...' URL body navigates there and returns {url,title}; and a 'url=<url>\n<expression>' prefix (mirrors the timeout=/capture= prefix parses) prepends `await page.goto(url,{waitUntil:'load',timeout:navTimeout})` to the script so the global/DOM under assertion is already loaded. Parse order is timeout= -> url= -> capture (url= before capture so the goto rides inside the capture IIFE and its navigation console/network is captured). Default path (no url=/bare-URL) is byte-unchanged = non-breaking. Wrapper-only JS, no rs-exec/rs-plugkit Rust change (the goto is just JS in the runner's -e evalBody); ships via cascade wrapper-refresh, auto-wrap live post-cascade. Witnessed the produced transform live: the goto-prefixed evalBody returned {url:'https://example.com/',title:'Example Domain'} in one dispatch. LESSON: an opt-in body prefix (like timeout=) is the safe way to add a browser-verb affordance -- default unchanged bounds blast radius; mirror the existing prefix-parse chain rather than touching the launch path.
