When a git checkout leaves multiple commits behind (a real fix commit plus a legitimate CI auto-bump that landed on top of it), an ancestry-safe merge target is not automatically the content-complete one. git merge-base --is-ancestor proves a candidate is safe to merge, never that it is the right one when a chain of dangling commits exists. Merging the older parent instead of its child produces a clean fast-forward and a real-looking push confirmation while silently dropping the actual change. After any such recovery, verify by grepping the real pushed content on origin for the expected change (git show origin/branch:path | grep new-symbol), never by trusting git_push's own summary line alone -- Everything up-to-date and a real ref-update message look identical whether or not the intended content actually landed.
