iter107 (2026-05-27): FIXED a real distributability bug that blocked the 0.1.605 upgrade. The 0.1.605 wasm download FAILED across 2 boot attempts - bootstrap-error showed abs-deadline 120000ms ... after 7634259 bytes: gm-plugkit/bootstrap.js httpGetBuffer set a HARD absTimer = totalDeadlineMs (timeoutMs*2 = 120s for the gh-release fetch via downloadFromGithubReleases(url,60000)) that destroyed the download regardless of progress. The wasm is ~149MB; 120s is not enough on a normal/slow link (7.6MB in 120s ~= 63KB/s), so the timer killed a download that was actively receiving bytes, the new watcher failed ensure-ready, and the stale 0.1.604 kept running with update_available perpetually flagging 0.1.605. The req idle-timeout (socket inactivity) already catches true stalls. FIXED: made absTimer PROGRESS-AWARE - extracted armAbsTimer() that clears+re-arms the timer, called once at start AND on every res data chunk, so the deadline is since-last-progress not a total cap; a download making progress is never killed, only true inactivity aborts. Ships via publish.yml (gm-plugkit JS). LESSON: a fixed total-deadline on a large-artifact download is a distributability landmine - it silently fails upgrades on slow links and wedges the version. Large-file fetches need a since-progress deadline (or size-scaled), never a flat total cap; the socket idle-timeout is the correct stall catcher. ALSO: the iter106 prune-miss fix (0.1.605) is blocked from local validation until this download-fix ships and 0.1.605 can actually download. cores 135/135, router count 40 quality 941.
