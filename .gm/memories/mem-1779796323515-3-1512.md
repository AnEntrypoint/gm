iter41 (2026-05-26): FIXED a user-diagnosed real defect - the browser-session gate was GLOBAL, not session-scoped. residual.rs browser_sessions_open() returned true whenever browser-sessions.json was non-empty, counting ALL claudeSessionId keys; task.rs open_browser_sessions() iterated ALL (sid,sessions). CONSEQUENCE: (1) a session was wedged at residual-scan/COMPLETE by browser sessions OTHER projects/sessions opened (tv8 sessions 49/50/52/53) - browsers it never opened and must not close (closing another active projects browser is outward-facing, hard-to-reverse). (2) The same global enumeration leaked other sessions browsers into the agents orientation - this is how the tv8 agent saw other sessions browsers and got disrupted. FIX (rs-plugkit 09148c2): both fns read current session_id from state::read_state() and consider ONLY that sids entry; foreign sessions ignored. Fail-safe: no session_id or unparseable json -> conservative global/any-open (never under-gate). browser-sessions.json + browser-ports.json are already keyed by claudeSessionId so the data was session-scoped; only the gate readers were global. A session now gates-on + closes only its OWN browsers; foreign sessions re-open their own (user directive). LESSON: shared-surface gates (browser, and any other cross-project resource registry) must scope to the current session id, not the global registry - a global gate on a per-session resource wedges every session on every other sessions state and leaks cross-session visibility.
