OPERATIONAL FIX (2026-06-14 audit fire, cron 31e7e1ca): the "watcher dead after a release" incident class is the supervisor permanently giving up during a cascade version bump. Mechanism: checkWatcherHealth (gm-plugkit/supervisor.js) detects version_drifted and UNLINKS the installed ~/.gm-tools/plugkit.wasm to force a re-download, then kills the watcher; the respawn finds no wasm -> wrapper selfHeal('wasm not installed') tries to download the just-bumped build, which during the cascade window is not yet propagated to plugkit-bin gh-release/npm -> selfHeal fails -> wrapper exit(1). Six such exits in 60s tripped MAX_RESTART_BURST=5 and the old code did process.exit(2) = PERMANENT supervisor death, leaving the project watcher dead until a manual `bun x gm-plugkit@latest spool`. Witnessed this run: ~thebird + ~corpus supervisors emitted supervisor.giving-up restart-burst-exceeded; my own gm watcher hit the same self-heal crash-loop and I recovered it manually. FIX (gm 5696a5fd): burst-exceeded now resets restartTimestamps, writes status 'backoff', and setTimeout(spawnWatcher, 60s)+return instead of exit(2) -- the supervisor self-recovers once the build propagates (each respawn's selfHeal re-downloads). Lesson: a permanent-death failure mode on a transient upgrade-window error violates 'optimize the worst case, never a silent catastrophic mode'; supervisors must back off and keep retrying, never exit permanently.
