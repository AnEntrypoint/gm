agentplug daemon.rs self-update handoff protocol + panic isolation: attempt_self_update_handoff spawns the newly-staged binary as `takeover <version>`, which itself builds a real engine and pre-compiles the default plugin set BEFORE writing its readiness marker (daemon-takeover-ready.json) -- so 'ready' genuinely means 'can serve', not merely 'process started'. The OLD daemon polls for that marker (bounded, 40 x 250ms = up to a few seconds by default) then calls release_ownership_for_handoff (deletes its own owner-lock file only if it's still the recorded owner) and exits; a broken/hung new build that never confirms just leaves the old daemon running unchanged, staged .new retried next poll interval. The new (takeover) process separately waits up to 480 x 250ms (2 minutes, was originally 10s) for the OLD daemon's owner file to actually disappear before claiming it -- live-witnessed the old daemon's own release can be delayed well past a few seconds when its main loop is blocked mid one long synchronous wasm call already in flight at poll time, the identical 'stuck in one long call, misses its own heartbeat tick' condition the heartbeat-ticker fix addresses; a 10s giveup under that same load would abandon a handoff that was genuinely still coming, costing a full extra poll-interval wait. Staging (download+verify+sha256 to <exe>.new) is deliberately NOT gated on daemon idleness (never touches the running process) while the actual handoff IS idle-gated (`!any_work`) since it must not interrupt in-flight dispatch -- splitting these two halves fixed a real starvation where a busy daemon (any_work true on effectively every tick under sustained multi-project traffic) never even checked for a new version for months. Separately: run_gm_dispatch_to_file and the generic plugin-dispatch path both wrap their dispatch call in catch_unwind -- an epoch-interrupt trap from wasmtime can surface as a Rust panic from deep inside a host-import callback (e.g. write_guest_bytes's plugkit_alloc, invoked after the guest's own deadline already elapsed) rather than a clean Err from the top-level Result, and wasmtime's trap unwinds through whatever Rust frame happens to be on the stack when the epoch check fires -- not always the outermost dispatch_fn.call. Without catch_unwind this silently killed the worker thread inside thread::scope (a panicked JoinHandle's Err is discarded by `let _ = h.join()`), producing no response at all and hanging the caller until its own 30s poll timeout, even though the daemon process itself survived.
