ROBUSTNESS GAP witnessed 2026-05-28: rs-plugkit prd.yml can lose all rows under write contention. During a session with a 750KB+ prd store, a heavy prd-list read concurrent with prd-resolve writes produced write-failed then prd_total_count:0 (store emptied). Symptom: subsequent prd-resolve returns known_ids:[] for ids that were definitely added+confirmed earlier. The prd.yml read-modify-write in prd.rs is not atomic/locked against concurrent verbs. Candidate fix: atomic write (temp+rename) + a lock, or serialize prd verbs. For now: avoid prd-list on huge stores mid-resolve; the work itself was witnessed via git pushes regardless of store state.
