gm-plugkit/cli.js spool command lacked an already-serving fast-path: every invocation unconditionally ran ensureReady() (network bootstrap check) + startSpoolDaemon() (spawn attempt) + waitForWatcherHeartbeat(30s deadline) even when a fresh watcher was already alive and serving -- this is what made calling agents perceive the spool command as blocking/holding them up instead of daemonizing instantly. Fix: read .status.json first, and if statusServing(status,12000) is true (fresh ts or future busy_until), exit(0) immediately with already_serving:true, skipping bootstrap/spawn/wait entirely. Witnessed: cold path unchanged (falls through), hot path dropped from multi-second/up-to-30s to ~100ms on an already-alive watcher.
