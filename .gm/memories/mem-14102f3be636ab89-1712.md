agentplug-host browser.rs drives real Chrome directly via CDP (Chrome DevTools Protocol over HTTP+WS) instead of shelling out to the playwriter CLI, because playwriter's relay process crashes with a real, live-reproducible Windows libuv assertion on every eval dispatch: 'Assertion failed: !(handle->flags & UV_HANDLE_CLOSING), file src\\win\\async.c, line 76' -- re-confirmed live TWICE on playwriter 0.4.0, not a stale/misdiagnosed finding, specific to Windows async-handle-teardown, not evidence against playwriter broadly. Working flow: launch Chrome headless with --remote-debugging-port, poll /json/version HTTP endpoint for the DevTools websocket URL, run scripts via Runtime.evaluate over that websocket using a bundled node helper (cdp_eval.js, node has a native WebSocket, already a required runtime). No playwriter, no relay, no crash. Session persistence evolution: a fresh Chrome used to launch AND get unconditionally killed at the end of EVERY dispatch (only the on-disk --user-data-dir profile persisted, not the live process/window -- 'open the browser so we can see it' flashed open and immediately closed). Fixed via a process-wide SESSIONS registry (keyed by (cwd,session_id)) keeping a launched Chrome child + CDP port alive across dispatches, only dying via explicit session close/reset, the idle-timeout reaper, or daemon exit. Chrome's --user-data-dir itself was ALSO being deleted every dispatch from a temp-dir location outside the project (zero cookie/localStorage/login persistence) -- fixed to live permanently under <project>/.gm/browser-chrome-profile-<session_id>/, never deleted (only the process is killed on session close, profile survives for reuse under the same session id).
