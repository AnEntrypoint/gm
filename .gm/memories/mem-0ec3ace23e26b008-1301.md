code_index.rs history (rs-plugkit): likely_orphaned_symbols is a cheap text-occurrence heuristic ported from the mcp-thorns comparison (codesearch-vs-mcp-thorns-parity) -- a named function/method whose name never appears in any OTHER indexed chunk body anywhere in the corpus is a dead-code CANDIDATE, not real cross-file static analysis (no import/call-graph resolution, no scope awareness, same heuristic class mcp-thorns itself uses). False positives exist (common-word names, re-export aliases, dynamic dispatch) -- it is a fast signal for a human/agent to verify via real codesearch before deleting anything, never an automated deletion trigger. FusionCorpus::overview_for_key sources from FileManifest's cached commit_overview (computed at index time, never a live git call on the codesearch read path); symbol_for_key exposes structured path/kind/name/line_start/line_end as real JSON fields (the structured-symbol-awareness surface mcp-thorns exposes natively) instead of forcing callers to regex an unparseable "path:ls:le name\nbody" text blob. key_for_path_line maps a (path,line_start) pair (the shape search()'s vector rows come back in) to the ci-hash-hash-idx fusion key -- without it the vector half cannot contribute to fusion since the two stores identify the same chunk differently.
