Jank sweep rule at PLAN: every prompt-concerned surface gets every immaturity/unfinished-edge/half-wired-path enumerated across gui/ux/ui/client-state/server-state/boundary + reached surfaces, each its own PRD row plus a per-surface profiling row and security row. jank means rough/unpolished/almost-done, not just bugs -- scoped to the prompt's reachable closure, exhaustive within it. Every found issue spawns its own debug-repair plan same turn; every quick win gets spooled too. Architecture is pliable: an architectural change that clearly reduces maintenance burden becomes a spooled plan, but swapping bespoke code for a native/well-maintained library is only justified when it nets a SMALLER maintained surface overall -- pulling in a heavy dependency to delete a few lines is a guarded failure mode, not a win. Fan-out for this sweep is spool-native (parallel prd-add/codesearch/exec_js dispatches, plugkit task-spawn), never the platform's Task/Explore subagent. One AI-tell design element (boilerplate flourish, over-hedged comment, generic scaffold name) spawns a full-codebase sweep: scan, per-cluster findings, fix-and-verify rows, exhaustive, never a one-off local fix.
