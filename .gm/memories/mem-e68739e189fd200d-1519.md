wasm_dispatch/verbs.rs history (rs-plugkit) git_push: `git rev-parse --abbrev-ref HEAD` literally returns the string "HEAD" when the checkout is in detached-HEAD state -- the normal default state for every git submodule (this repo's own agentplug/rs-plugkit/rs-codeinsight/rs-search/agentplug-bert/agentplug-libsql/agentplug-treesitter submodules all live detached-at-pin). Falling back to that literal "HEAD" string as the push target (rather than resolving it to the real intended branch) previously produced a real git error "not a full refname" on every no-explicit-branch git_push/git_finalize dispatch against a detached submodule, live-reproduced in a session. The wrong-branch-denial guard already correctly exempted "HEAD" (detached is not a mistake), it just never resolved the fallback -- fixed by defaulting to "main" when current_branch=="HEAD", matching this repo's own direct-push-to-main-always discipline. exec_git_push_in returns (combined stdout+stderr text, whether push actually succeeded per exit_code) -- a prior version discarded exit_code entirely and let push_rejected()'s substring match on combined text be the ONLY success signal; any git push failure whose wording didn't match one of a few known-rejection phrases (network blip, auth timeout, differently-worded rejection across git versions) was silently reported as success with the commit never having reached origin. exit_code is now the actual unambiguous signal; string matching only decides whether a rejection is rebase-retryable.
