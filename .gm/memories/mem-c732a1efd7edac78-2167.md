agentplug-runner main.rs boot sequencing fixes: (1) A process-wide std::panic hook calls agentplug_host::close_all_sessions() before the default hook runs -- a panic unwinds and exits WITHOUT running the daemon loop's normal exit paths (the heartbeat-authority-lost branch), so a crashed daemon previously left every open Chrome session orphaned; live-hit as the Instant-subtraction-underflow panic on every boot right after a machine reboot meant no daemon instance survived long enough to clean up, and open sessions piled up across repeated crash-boot cycles until the machine itself needed a reboot. (2) `spool` subcommand deliberately does NOT eagerly download gm.wasm before registering -- that used to block the entire invocation for minutes on a cold cache (gm.wasm ~137MB), violating gm skill's documented fire-and-forget spool contract, live-witnessed as a 20s+ hang on a command that should return near-instantly; the daemon's own loop already downloads gm lazily on first real dispatch need, same pattern already proven for libsql/bert/treesitter. (3) `spool`'s ensure_daemon_running() bounded wait (~6s) can time out while the real winning daemon is still mid-PluginModules::new() (engine build + 4-plugin compile, measured taking well over 6s under load) and hasn't written its first heartbeat yet -- previously every timed-out invocation fell straight to a standalone long-lived watcher that NEVER calls claim_ownership()/run_daemon() (not a race the atomic guard was positioned to prevent, since it's a separate process that never contests the lock), so multiple standalone watchers coexisted indefinitely, each serving its own project -- live-witnessed as the 'multiple agentplug-runner processes' symptom. Fixed: attempt to become the shared daemon itself (call run_daemon()) before ever falling back to a private one-shot instance; if it loses the race, run_daemon() returns immediately with zero plugin state touched, so retry ensure_daemon_running() once more before finally falling back to a standalone watcher -- narrows (does not eliminate) the standalone-watcher-spawn window to two consecutive missed compile-plus-wait cycles instead of one.
