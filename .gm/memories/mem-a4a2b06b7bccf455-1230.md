agentplug-host registry.rs architecture: dispatch() must evict a pool slot's Store/Instance on ANY dispatch error (wasmtime Trap or otherwise), never reuse it for the next caller. A wasmtime Trap leaves the Store/Instance in an unspecified state per wasmtime's own documented contract -- reusing a post-Trap Store for the next dispatch is the actual root cause of a real, live-hit bug class: a poisoned shared "bert"/"gm" pool slot kept failing every subsequent dispatch with "plugin X not loaded"-shaped errors downstream, until idle-eviction or a full daemon restart happened to clear it by chance. This exact symptom (sustained "plugin gm not loaded" errors across many consecutive retries, self-resolving only after an unrelated watcher reboot) was independently observed and worked around via retry-then-reboot in a concurrent gm session before this fix landed -- the workaround was masking this root cause, not fixing it. Fix: setting the guard slot to None whenever dispatch_on returns Err forces the NEXT dispatch to instantiate_plugin() a fresh Store instead of reusing the poisoned one. Applies at both call sites sharing the identical pool/guard/dispatch_on shape: ProjectPlugins::dispatch and DispatchHandle::dispatch.
