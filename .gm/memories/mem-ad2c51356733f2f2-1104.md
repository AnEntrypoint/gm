## spooler self-recovery mechanics (apparent tooling failure)

Apparent tooling failure is the agent's own mechanical self-recovery, NEVER a user question and never an a/b-test/blind-restart. 'The spooler is not working' / a missing spool response / a stale watcher: honor a future busy_until (wait), else boot the watcher (bun x gm-plugkit@latest spool) and re-dispatch. On a transient boot hiccup (FailedToOpenSocket) retry @latest, never the non-@latest cache (lands a stale watcher). The spooler is sound by construction -- .status.json is written atomically (temp+rename, atomicWriteJson) and every long verb advertises busy_until -- so a transient unreadable/stale read is a respawn/idle-teardown window to boot through, not a broken tool; asking the user to do what a verb can do is a paper-spirit violation. Debug the live page via window.* globals + the browser verb page.evaluate as process-of-elimination, never variant-after-variant a/b testing. This is the core gm method applied to gm's own tooling: record each candidate cause as a mutable, eliminate by witness, discover more, keep going.
