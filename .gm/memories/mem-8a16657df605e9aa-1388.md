agentplug-host lib.rs wasmtime epoch-interruption three-piece mechanism: config.epoch_interruption(true) alone does nothing without ALL THREE pieces present together: (1) a process-wide background ticker thread (start_epoch_ticker, started exactly once via OnceLock) calling Engine::increment_epoch() on a fixed interval (EPOCH_TICK_INTERVAL_MS) -- this is what actually advances the epoch counter, since epoch_interruption(true) alone just enables the CHECK, nothing ever moves the counter forward without this thread running; (2) Store::set_epoch_deadline at instantiation (registry.rs) arming a real deadline; (3) config.wasm_backtrace_details(Enable) so a trapped guest call carries a real stack frame (function name + wasm offset) instead of a bare 'unreachable' with zero diagnostic context -- wasmtime's backtrace-details default is environment-dependent (often off in release builds), which is why a real remote-user trap this session produced no actionable info at all before this was set. This whole mechanism closes a gap SharedPluginPool::acquire's own timeout does NOT cover: acquire only bounds the WAIT to get a pool slot; once a caller has the slot and is inside dispatch_fn.call, nothing could previously interrupt that call itself, so a single stuck call on a process-wide pool-size-1 shared plugin (bert/libsql) wedged every project's dispatch into that plugin forever.
