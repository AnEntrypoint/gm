agentplug daemon.rs IN_FLIGHT/background-convert race fixes: the registry (root,verb,task)->InFlightHandle{detach:AtomicBool} must be populated SYNCHRONOUSLY before a gm-verb dispatch thread is even spawned, and every background-convert request in the same claimed batch must be answered only AFTER that registration -- live-witnessed repeatedly: exec_js and a background-convert targeting it, written under 150ms apart landing in the same claimed batch, background-convert answered 'already_completed' after only ~100ms even though exec_js went on to run its full real duration, because the IN_FLIGHT entry did not exist yet when background-convert checked. Second bug: the original per-file inline bounded-poll loop (claim file -> poll to completion -> move to next verb_entry/file_entry) let a background-convert sitting in a LATER verb dir (fs::read_dir order is unspecified) go unclaimed for the entire duration of an earlier file's poll loop -- measured: exec_js (20s sleep) plus a background-convert targeting it, written ~300ms apart, both resolved in the same ~50ms window ~80s later, proving background-convert was never claimed until the NEXT dispatch_project sweep. Fix: claim-and-collect every request file across every verb dir in one fast pass first (no blocking work during collection), THEN spawn every gm-verb dispatch onto its own thread with IN_FLIGHT registered first, THEN answer background-converts from that batch, THEN round-robin poll (is_finished(), ~50ms cadence) -- and the poll loop RE-SCANS in_dir on every tick (not just the initial snapshot) so any request (including background-convert) arriving mid-batch is claimed and answered within one poll tick instead of waiting a full dispatch_project-call-cycle.
