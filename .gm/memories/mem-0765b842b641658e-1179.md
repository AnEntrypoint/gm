STANDING AUDIT STEP (every 30-min audit fire MUST run this): compare rs-plugkit source version vs plugkit-bin published release to catch a silent cascade outage. Step: (1) read rs-plugkit Cargo.toml version (grep ^version), (2) fetch https://api.github.com/repos/AnEntrypoint/plugkit-bin/releases/latest tag_name, (3) if source patch > published patch by >1 (e.g. source 0.1.624 vs published 0.1.616), the cascade is DARK - fetch https://api.github.com/repos/AnEntrypoint/rs-plugkit/actions/runs?per_page=4 and check the latest Release/Build conclusion; if failure, fetch the build job check-run annotations for the ::error:: lines (CI job logs are admin-403, but the Build-WASM step echoes ::error:: annotations - rs-plugkit commit 7c843db). This is the detector that would have caught the 5-day outage on day 1. update_available=null is CORRECT when no newer release exists, so it does NOT signal source-ahead-of-published drift - this explicit version-compare is the missing signal. Implemented as a standing audit procedure (zero new code, respecting the no-speculative-tooling back-pressure) rather than a bespoke gm probe; the audit fire already does the plugkit-bin fetch.
