agentplug-host imports.rs/host_state.rs cross-plugin call safety: host_plugin_call and host_vec_embed must drive the TARGET sibling plugin's call through the SIBLING's own Store, never the calling plugin's Caller -- wasmtime's Instance::get_typed_func/call require a StoreContextMut matching the store the Instance was instantiated in, and reusing the wrong Caller previously panicked at runtime with 'object used with the wrong store' (wasmtime-46 store/data.rs:213) the first time gm.wasm's recall path called into bert via host_plugin_call. SiblingHandle (Boxed inside HostState -- an unboxed Store<HostState> field would make HostState infinitely-sized) owns its own Store+Instance pair for exactly this reason. HostState.siblings is a Mutex around the Arc ITSELF (not just the inner HashMap) because a SHARED plugin instance (is_stateless_shared_plugin) reuses this same HostState across every project's dispatch -- the map it points at must be swappable per-call via set_siblings (called fresh by registry::dispatch_on on every dispatch, mirroring set_cwd), not fixed at whichever project's load_plugin happened to instantiate the Store first (a bug that was previously correct only by accident, since every default-config project happens to load the same 4 plugins in the same order). write_guest_bytes must re-arm the epoch deadline (not just rely on dispatch_on's own entry arm) when it catches a Trap::Interrupt from its own plugkit_alloc call -- this function runs INSIDE an already-in-flight guest call (e.g. marshaling a slow host_exec_js result back after the call itself blocked past the Store's epoch deadline), so the deadline can have legitimately already elapsed by the time control reaches here; a raw .expect() here previously panicked with a wasmtime trap string instead of degrading like every other deadline-exceeded path -- fixed to return 0 (the function's own documented empty/none sentinel per docs/ABI.md) and re-arm so the NEXT call on this reused Store starts from a fresh budget.
