BRANCH_STATUS STALE-REF bug (rs-plugkit, fixed 2026-05-27): branch_status and the COMPLETE-gate remote-pushed witness (orchestrator/transitions.rs:68) computed rev-list --left-right origin/<branch>...HEAD against the LOCAL remote-tracking ref, which git_push/git_finalize do NOT refresh. So right after a successful push, ahead:N stays >0 (stale) and the four-observation COMPLETE gate rejects with remote-pushed=false even though HEAD is on origin. This was the recurring ahead:1 across iter116-118 that I kept mis-attributing to transient watcher churn - it was the stale ref every time. FIX: git fetch <remote> <branch> before the rev-list at both sites (branch_status verb + transitions.rs gate), tolerate offline. LESSON: any ahead/behind computation must fetch first or it lies after a push; never trust the local origin/<branch> ref freshness. To unblock a COMPLETE gate stuck on false ahead:N on the OLD wasm, manually git fetch first.
