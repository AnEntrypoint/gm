iter115 (2026-05-27): ROOT-FIXED the git_finalize VERB ABORT (seen twice iter114). Root cause WITNESSED: supervisor.js checkWatcherHealth reaps the watcher on status ts age>STATUS_STALE_MS(30000) via SIGTERM+taskkill, IGNORING busy_until - while git_finalizes ~30s network push blocks the 5s heartbeat, the supervisor thought the watcher hung and killed it mid-verb. The wrapper already had _writeStatusBusy(ms) writing busy_until to .status.json but ONLY the browser runner called it. FIX (gm-plugkit, ships via publish.yml): (1) supervisor checkWatcherHealth returns early when status.busy_until>now, honoring it exactly as the agent boot probe does; (2) wrapper processFile stamps _writeStatusBusy(90000) before dispatching git_finalize/git_push/git_fetch. This is the deeper fix behind iter114s response-shape patch (77542ee, now in 0.1.607). LESSON: any long synchronous wasm verb (network push, chromium spawn) MUST stamp busy_until before blocking, AND every liveness reaper (supervisor + boot probe) MUST honor it - a reaper that checks ts-staleness without busy_until kills busy watchers. STATUS_STALE_MS=30000 was right at the push duration = intermittent.
