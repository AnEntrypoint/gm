When multiple gm-driven subagents run concurrently against the same repo's .gm/prd.yml and .gm/mutables.yml, prd-resolve/mutable-resolve responses can read back as still-pending in a LATER instruction/phase-status snapshot due to a different concurrent writer's read-modify-write racing the same file -- this is not a failed resolve (the resolve response itself always confirms {resolved:<id>} truthfully), it's a stale read from a losing writer. Re-dispatching prd-resolve/mutable-resolve with the same id and witness a second time is safe (idempotent, per f.f=f) and resolves the apparent flap. Don't treat a row reappearing as pending after a confirmed resolve as a bug to chase -- re-confirm once, move on.
