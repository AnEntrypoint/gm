agentplug daemon.rs single-instance ownership (claim_ownership/holds_heartbeat_authority): daemon-owner.lock is claimed via OpenOptions::create_new (O_EXCL), the only non-racy primitive available -- filesystem arbitrates exactly one winner among simultaneous racers. A stale owner is replaced via tmp-write-then-atomic-rename, never remove-then-create (that reopens the same TOCTOU gap). Staleness is judged by a SEPARATE heartbeat file's ts (daemon-status.json), never by the lock file's own presence, AND the recorded pid must be independently verified alive via pid_is_alive (tasklist on Windows / kill -0 on POSIX) -- timestamp freshness alone is insufficient because a process that just died (Stop-Process -Force/taskkill /F/SIGKILL, no cleanup chance) leaves a heartbeat reading fresh for up to DAEMON_STALE_MS with nothing alive behind it; live-hit directly (killed daemon, immediately re-ran spool, got 'registered with shared daemon' back with zero real processes). DAEMON_STALE_MS is 2x the daemon's own heartbeat_interval (20s, was 60s) plus slack -- the looser 60s left a wide false-fresh window. The heartbeat/ownership check runs on an INDEPENDENT ticker thread (spawn_heartbeat_ticker), never inline in the main dispatch loop -- a per-project worker occupying a thread::scope slot for up to the dispatch deadline (40s) used to delay the heartbeat write by that same duration (live-witnessed: daemon stuck mid-call 12+s without heartbeating while burning a full core, until a competing daemon claimed authority and took over). The ticker only ever raises a one-way AtomicBool latch (HEARTBEAT_AUTHORITY_LOST, Relaxed ordering sufficient) on losing authority; only the main thread performs actual session-owning shutdown (close_all_sessions), since the ticker must never touch state the main loop owns.
