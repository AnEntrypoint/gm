DEAD-WATCHER diagnostic invariant (a manual-history-review pattern; deviations_30m does NOT flag a silently-dead watcher, so the manual pass is what catches it): a watcher silent-abort (the boot-time watcher.silent-abort event: prior status ts stale, shutdown_reason_present:false, exit_code 4294967295/-1, no panic in .watcher.log) occurring SIMULTANEOUSLY across multiple watchers (e.g. gm + spoint at the same minute) -- with the per-cwd supervisor also gone -- is an EXTERNAL mass-kill (OS sleep/resume or a mass process kill), NOT a gm code defect, and it is recovered by the designed boot-probe path (the next session's boot probe sees the stale ts and boots a fresh watcher; the silent-abort/lock-pid-dead-takeover/unplanned-restart instrumentation fires on the fresh boot). Distinguish from a gm crash: a real gm crash logs a panic/fatal in .watcher.log (e.g. 'fatal: import function env:host_vec_embed must be callable' = a wrapper/wasm host-import version-skew at instantiation -- check the wrapper actually provides every extern host fn). The diagnostic surface is gmsniff --grep 'supervisor|watcher\.|silent-abort|exit_code' + the .watcher.log tail. No fix for the external-kill case; the boot-probe recovery is by-design.
