vecns.rs architecture (rs-plugkit): VecTableSpec identifies one vector table's physical location (db instance, table name, libsql_vector_idx index name) -- every call site (rssearch_vectors, git_commit_vectors, code_index's code_chunks and memories tables) constructs one and delegates shared mechanics (dim-mismatch drop, index create-or-rebuild, shadow-row-recovery insert) to it instead of reimplementing per call site with slightly different constants. db_name is always a real absolute path by the time it reaches libsql_exec -- every real construction site resolves it via code_index::project_db_path before building the spec; agentplug-libsql is stateless/process-wide-shared, reads ONLY the JSON body's path field (defaulting to :memory:, silently throwaway, if absent), never db.

exec_with_shadow_row_recovery (generalized from an rssearch_vectors::write fix): a "vector index(insert): failed to insert shadow row" error was live-observed on brand-new keys that had never been written before, ruling out an ON-CONFLICT-DO-UPDATE race against an existing row. The shadow table backing libsql_vector_idx is a real on-disk structure that a watcher crash storm (abrupt process kills mid-write) can leave corrupted -- corruption persists across restarts since the shadow table lives on disk, not memory, and nothing ever drops+recreates the index once CREATE INDEX IF NOT EXISTS has run once. Recovery: drop and recreate the vector index, retry the insert exactly once; if the retry still fails, surface the (now index-rebuilt) error rather than looping. delete_then_insert_with_recovery is the delete-then-insert shape all 3 call sites use to sidestep libsql_vector_idx's unreliable ON CONFLICT DO UPDATE support -- always a fresh insert, never an update. recency_score implements: recency = floor + (1-floor) * exp(-age_ms/half_life_ms); score = cos * recency -- factored out of rssearch_vectors::search_with_recency/search_memory_hits so both share one implementation; recency_floor is the asymptotic minimum multiplier a row can never decay below (a very old but highly-relevant hit is still findable, never zeroed out).
